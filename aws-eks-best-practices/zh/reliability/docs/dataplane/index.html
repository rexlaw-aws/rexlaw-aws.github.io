    EKS 数据平面¶

    为了运行高可用和有弹性的应用程序,您需要一个高可用和有弹性的数据平面。弹性数据平面确保 Kubernetes 可以自动扩展和修复您的应用程序。弹性数据平面由两个或更多工作节点组成,可以根据工作负载进行扩展和收缩,并自动从故障中恢复。

    您可以使用 EKS 上的两种类型的工作节点: EC2 实例和 Fargate。如果您选择 EC2 实例,您可以自己管理工作节点,或使用 EKS 托管节点组。您可以拥有一个由托管、自管理工作节点和 Fargate 混合组成的集群。

    Fargate 上的 EKS 提供了最简单的弹性数据平面路径。Fargate 在隔离的计算环境中运行每个 Pod。在 Fargate 上运行的每个 Pod 都有自己的工作节点。Fargate 会根据 Kubernetes 扩展 Pod 的方式自动扩展数据平面。您可以使用水平 Pod 自动缩放器来扩展数据平面和您的工作负载。

    使用 Kubernetes Cluster Autoscaler、EC2 Auto Scaling 组或社区项目如 Atlassian's Escalator 是扩展 EC2 工作节点的首选方式。

    建议¶

    使用 EC2 Auto Scaling 组创建工作节点¶

    使用 EC2 Auto Scaling 组创建工作节点是最佳实践,而不是创建单个 EC2 实例并将它们加入集群。Auto Scaling 组将自动替换任何终止或失败的节点,确保集群始终有足够的容量来运行您的工作负载。

    使用 Kubernetes Cluster Autoscaler 扩展节点¶

    Cluster Autoscaler 会在集群没有足够资源运行 Pod 时调整数据平面的大小,并添加另一个工作节点。虽然 Cluster Autoscaler 是一个被动的过程,它会等待 Pod 处于 Pending 状态,因为集群容量不足。当发生这种情况时,它会向集群添加 EC2 实例。每当集群耗尽容量时,新的副本或新的 Pod 将处于 Pending 状态,直到添加了工作节点。如果数据平面无法足够快地扩展以满足工作负载的需求,这种延迟可能会影响应用程序的可靠性。如果工作节点一直被低利用率,并且它的所有 Pod 都可以在其他工作节点上调度,Cluster Autoscaler 会终止它。

    配置 Cluster Autoscaler 的过度配置¶

    Cluster Autoscaler 在集群中的 Pod 已经处于 Pending 状态时触发数据平面的扩容。因此,从应用程序需要更多副本到实际获得更多副本之间可能会有延迟。一种解决这种可能延迟的方法是通过添加超过所需的副本数,增加应用程序的副本数。

    Cluster Autoscaler 推荐的另一种模式使用 pause Pod 和优先级抢占特性。pause Pod 运行一个 pause 容器,顾名思义,它什么也不做,只是作为集群中可以被其他 Pod 使用的计算容量的占位符。因为它以 非常低的优先级 运行,当另一个 Pod 需要被创建而集群没有可用容量时,pause Pod 会被驱逐出节点。Kubernetes 调度器注意到 pause Pod 被驱逐,并尝试重新调度它。但由于集群正在满负荷运行,pause Pod 保持 Pending 状态,Cluster Autoscaler 会对此做出反应,添加节点。

    可以使用 Helm 图表安装 cluster overprovisioner。

    将 Cluster Autoscaler 与多个 Auto Scaling 组一起使用¶

    运行 Cluster Autoscaler 时,启用 --node-group-auto-discovery 标志。这样做将允许 Cluster Autoscaler 找到包含特定定义标签的所有自动缩放组,并防止需要在清单中定义和维护每个自动缩放组。

    将 Cluster Autoscaler 与本地存储一起使用¶

    默认情况下,Cluster Autoscaler 不会缩减具有附加本地存储的节点。将 --skip-nodes-with-local-storage 标志设置为 false,以允许 Cluster Autoscaler 缩减这些节点。

    在多个 AZ 中分散工作节点和工作负载¶

    您可以通过在多个 AZ 中运行工作节点和 Pod 来保护您的工作负载免受单个 AZ 故障的影响。您可以使用创建节点的子网来控制工作节点创建的 AZ。

    如果您使用 Kubernetes 1.18 及更高版本,推荐的跨 AZ 分散 Pod 的方法是使用 Pod 拓扑传播约束。

    下面的部署会尽可能地在 AZ 之间分散 Pod,如果不可能,也会让这些 Pod 运行:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-server
  template:
    metadata:
      labels:
        app: web-server
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          whenUnsatisfiable: ScheduleAnyway
          topologyKey: topology.kubernetes.io/zone
          labelSelector:
            matchLabels:
              app: web-server
      containers:
      - name: web-app
        image: nginx
        resources:
          requests:
            cpu: 1

    注意

    kube-scheduler 只能通过带有这些标签的节点感知拓扑域。如果上述部署被部署到只有单个区域节点的集群中,所有 Pod 都将调度到这些节点上,因为 kube-scheduler 不知道其他区域。这个问题将在 Kubernetes 1.24 中通过添加 MinDomainsInPodToplogySpread 特性门控得到解决,该特性允许指定 minDomains 属性来告知调度器合格域的数量。

    警告

    将 whenUnsatisfiable 设置为 DoNotSchedule 将导致 Pod 在无法满足拓扑传播约束时无法调度。只有在不运行 Pod 比违反拓扑传播约束更好时,才应该这样设置。

    在较旧版本的 Kubernetes 中,您可以使用 Pod 反亲和性规则在多个 AZ 中调度 Pod。下面的清单告知 Kubernetes 调度器首选在不同的 AZ 中调度 Pod。

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
  labels:
    app: web-server
spec:
  replicas: 4
  selector:
    matchLabels:
      app: web-server
  template:
    metadata:
      labels:
        app: web-server
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - web-server
              topologyKey: failure-domain.beta.kubernetes.io/zone
            weight: 100
      containers:
      - name: web-app
        image: nginx

    警告

    不要要求 Pod 在不同的 AZ 中调度,否则部署中的 Pod 数量永远不会超过 AZ 的数量。

    在使用 EBS 卷时确保每个 AZ 都有足够的容量¶

    如果您使用Amazon EBS 提供持久卷,那么您需要确保 Pod 和相关的 EBS 卷位于同一个 AZ 中。目前,EBS 卷只在单个 AZ 中可用。Pod 无法访问位于不同 AZ 的 EBS 支持的持久卷。Kubernetes 调度器知道工作节点所在的 AZ。Kubernetes 将始终在与卷所在相同的 AZ 中调度需要 EBS 卷的 Pod。但是,如果在卷所在的 AZ 中没有可用的工作节点,则无法调度 Pod。

    为每个 AZ 创建一个 Auto Scaling 组,并确保其中有足够的容量,以确保集群始终有能力在与 EBS 卷相同的 AZ 中调度 Pod。此外,您应该在 Cluster Autoscaler 中启用 --balance-similar-node-groups 功能。

    如果您运行的应用程序使用 EBS 卷,但没有高可用性要求,那么您可以将应用程序的部署限制在单个 AZ 中。在 EKS 中,工作节点会自动添加 failure-domain.beta.kubernetes.io/zone 标签,其中包含 AZ 的名称。您可以通过运行 kubectl get nodes --show-labels 查看附加到节点的标签。有关内置节点标签的更多信息,请参阅此处。您可以使用节点选择器在特定的 AZ 中调度 Pod。

    在下面的示例中,Pod 只会在 us-west-2c AZ 中调度:

    apiVersion: v1
kind: Pod
metadata:
  name: single-az-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: failure-domain.beta.kubernetes.io/zone
            operator: In
            values:
            - us-west-2c
  containers:
  - name: single-az-container
    image: kubernetes/pause

    持久卷(由 EBS 支持)也会自动标记 AZ 名称;您可以通过运行 kubectl get pv -L topology.ebs.csi.aws.com/zone 查看持久卷所属的 AZ。当创建 Pod 并声明一个卷时,Kubernetes 将在与该卷相同的 AZ 中调度 Pod。

    考虑这种情况;您有一个具有一个节点组的 EKS 集群。这个节点组有三个工作节点,分布在三个 AZ 中。您有一个使用 EBS 支持的持久卷的应用程序。当您创建这个应用程序和相应的卷时,它的 Pod 会在第一个 AZ 中创建。然后,运行这个 Pod 的工作节点变得不健康并最终不可用。Cluster Autoscaler 将用一个新的工作节点替换不健康的节点;但是,由于自动缩放组跨越三个 AZ,新的工作节点可能会在第二个或第三个 AZ 中启动,而不是在第一个 AZ 中启动,这是当前情况所需要的。由于 AZ 受限的 EBS 卷只存在于第一个 AZ 中,但该 AZ 没有可用的工作节点,因此无法调度 Pod。因此,您应该在每个 AZ 中创建一个节点组,以确保始终有足够的容量来运行无法在其他 AZ 中调度的 Pod。

    另外,EFS可以简化集群自动缩放,当运行需要持久存储的应用程序时。客户端可以从该区域的所有 AZ 并发访问 EFS 文件系统。即使使用 EFS 支持的持久卷的 Pod 被终止并在不同的 AZ 中重新调度,它也能够挂载该卷。

    运行 node-problem-detector¶

    工作节点的故障可能会影响应用程序的可用性。node-problem-detector是一个 Kubernetes 插件,您可以在集群中安装它来检测工作节点问题。您可以使用 npd 的补救系统自动排空和终止节点。

    为系统和 Kubernetes 守护进程保留资源¶

    您可以通过为操作系统和 Kubernetes 守护进程保留计算容量来提高工作节点的稳定性。 Pod - 特别是那些没有声明 limits 的 Pod - 可能会占用系统资源,使节点陷入操作系统进程和 Kubernetes 守护进程(kubelet、容器运行时等)争夺系统资源的困境。您可以使用 kubelet 标志 --system-reserved 和 --kube-reserved 分别为系统进程(udev、sshd 等)和 Kubernetes 守护进程保留资源。

    如果您使用 EKS 优化的 Linux AMI,CPU、内存和存储默认情况下会为系统和 Kubernetes 守护进程保留。当基于此 AMI 启动工作节点时,EC2 用户数据会触发 bootstrap.sh 脚本。该脚本根据 EC2 实例上可用的 CPU 内核和总内存计算 CPU 和内存保留。计算出的值被写入位于 /etc/kubernetes/kubelet/kubelet-config.json 的 KubeletConfiguration 文件。

    如果您在节点上运行自定义守护进程,并且默认保留的 CPU 和内存量不足,您可能需要增加系统资源保留。

    eksctl 提供了最简单的方式来自定义系统和 Kubernetes 守护进程的资源保留。

    实施 QoS¶

    对于关键应用程序,请考虑在 Pod 中为容器定义 requests=limits。这将确保即使另一个 Pod 请求资源,该容器也不会被终止。

    为所有容器实现 CPU 和内存限制是最佳实践,因为它可以防止容器意外消耗系统资源,从而影响其他共同位置进程的可用性。

    为所有工作负载配置和调整资源请求/限制¶

    可以应用一些一般性指导来调整工作负载的资源请求和限制:

    • 不要在 CPU 上指定资源限制。在没有限制的情况下,请求充当容器获取相对 CPU 时间的权重。这允许您的工作负载使用全部 CPU,而不受人工限制或饥饿的影响。

    • 对于非 CPU 资源,配置 requests=limits 提供了最可预测的行为。如果 requests!=limits,容器的 QOS也会从 Guaranteed 降低到 Burstable,这使其在节点压力事件中更容易被驱逐。

    • 对于非 CPU 资源,不要指定一个远大于请求的限制。limits 相对于 requests 配置得越大,节点过度承诺的可能性就越大,从而导致工作负载中断的可能性更高。

    • 正确调整请求对于使用节点自动缩放解决方案(如 Karpenter 或 Cluster AutoScaler)非常重要。这些工具根据您的工作负载请求来确定要配置的节点数量和大小。如果您的请求太小而限制较大,当工作负载被紧密打包在节点上时,您可能会发现它们被驱逐或 OOM 杀死。

    确定资源请求可能很困难,但垂直 Pod 自动缩放器等工具可以帮助您'调整'请求,通过观察运行时容器资源使用情况来实现。其他可能有用的确定请求大小的工具包括:

    • Goldilocks
    • Parca
    • Prodfiler
    • rsg

    为命名空间配置资源配额¶

    命名空间旨在用于拥有许多用户分布在多个团队或项目的环境中。它们提供了名称的作用域,是在多个团队、项目和工作负载之间划分集群资源的一种方式。您可以限制命名空间中的总体资源消耗。ResourceQuota对象可以限制可在命名空间中创建的对象的数量,以及该项目中的资源(计算资源)的总量。您可以限制命名空间中可请求的存储和/或计算(CPU 和内存)资源的总和。

    如果为命名空间启用了资源配额,该命名空间中的用户必须为每个容器指定请求或限制。

    考虑为每个命名空间配置配额。考虑使用 LimitRanges 自动将预配置的限制应用于命名空间内的容器。

    限制命名空间内容器资源使用¶

    资源配额有助于限制命名空间使用的资源量。LimitRange 对象可以帮助您实现容器可以请求的最小和最大资源。使用 LimitRange 您可以为容器设置默认请求和限制,这在您的组织中设置计算资源限制不是标准做法时很有帮助。顾名思义, LimitRange 可以在命名空间中强制执行每个 Pod 或容器的最小和最大计算资源使用量。同时,还可以在命名空间中强制执行每个 PersistentVolumeClaim 的最小和最大存储请求。

    考虑将 LimitRange 与 ResourceQuota 一起使用,以在容器和命名空间级别强制执行限制。设置这些限制将确保容器或命名空间不会侵占集群中其他租户使用的资源。

    CoreDNS¶

    CoreDNS 在 Kubernetes 中履行名称解析和服务发现功能。它默认安装在 EKS 集群上。为了互操作性,Kubernetes 的 CoreDNS 服务仍然命名为 kube-dns。CoreDNS Pod 作为 Deployment 的一部分在 kube-system 命名空间中运行,在 EKS 中,默认情况下它运行两个副本,并声明了请求和限制。DNS 查询被发送到 kube-system 命名空间中运行的 kube-dns 服务。

    建议¶

    监控 CoreDNS 指标¶

    CoreDNS 内置了对 Prometheus 的支持。您应该特别关注监控 CoreDNS 延迟(coredns_dns_request_duration_seconds_sum,在 1.7.0 版本之前,该指标被称为 core_dns_response_rcode_count_total)、错误(coredns_dns_responses_total,NXDOMAIN、SERVFAIL、FormErr)和 CoreDNS Pod 的内存消耗。

    出于故障排查目的,您可以使用 kubectl 查看 CoreDNS 日志:

    for p in $(kubectl get pods -n kube-system -l k8s-app=kube-dns -o jsonpath='{.items[*].metadata.name}'); do kubectl logs $p -n kube-system; done

    使用 NodeLocal DNSCache¶

    您可以通过运行 NodeLocal DNSCache 来提高集群 DNS 性能。此功能在集群节点上以 DaemonSet 的形式运行 DNS 缓存代理。所有 Pod 都使用节点上运行的 DNS 缓存代理进行名称解析,而不是使用 kube-dns 服务。

    为 CoreDNS 配置 cluster-proportional-scaler¶

    提高集群 DNS 性能的另一种方法是根据节点数量和集群中的 CPU 内核自动水平扩展 CoreDNS Deployment。水平集群比例自动缩放器是一个容器,它根据可调度数据平面的大小调整 Deployment 的副本数量。

    节点和节点中的总 CPU 内核是您可以用来扩展 CoreDNS 的两个指标。您可以同时使用这两个指标。如果您使用较大的节点,CoreDNS 缩放将基于 CPU 内核数量。而如果您使用较小的节点,CoreDNS 副本的数量取决于数据平面中的 CPU 内核。比例自动缩放器配置如下所示:

    linear: '{"coresPerReplica":256,"min":1,"nodesPerReplica":16}'

    选择节点组的 AMI¶

    EKS 提供了经过优化的 EC2 AMI,这些 AMI 被客户用于创建自管理和托管节点组。这些 AMI 在每个区域和每个受支持的 Kubernetes 版本中都有发布。当发现任何 CVE 或错误时,EKS 会将这些 AMI 标记为已弃用。因此,建议在为节点组选择 AMI 时不要使用已弃用的 AMI。

    您可以使用以下命令使用 Ec2 describe-images API 过滤已弃用的 AMI:

    aws ec2 describe-images --image-id ami-0d551c4f633e7679c --no-include-deprecated

    您还可以通过验证 describe-image 输出是否包含 DeprecationTime 字段来识别已弃用的 AMI。例如:
