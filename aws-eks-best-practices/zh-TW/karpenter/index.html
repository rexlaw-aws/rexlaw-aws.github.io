    Karpenter 最佳實踐¶

    Karpenter¶

    Karpenter 是一個開源的叢集自動縮放器,可自動佈置新節點以應對無法調度的 pod。Karpenter 評估待處理 pod 的總體資源需求,並選擇運行它們的最佳實例類型。它將自動縮小或終止沒有任何非 daemonset pod 的實例,以減少浪費。它還支援一個整合功能,可主動移動 pod 並刪除或替換更便宜版本的節點,以降低叢集成本。

    使用 Karpenter 的原因

    在 Karpenter 推出之前,Kubernetes 用戶主要依賴 Amazon EC2 Auto Scaling groups 和 Kubernetes Cluster Autoscaler (CAS) 來動態調整其叢集的計算能力。使用 Karpenter,您無需創建數十個節點組來實現您使用 Karpenter 獲得的靈活性和多樣性。此外,Karpenter 與 Kubernetes 版本的耦合程度不如 CAS 那麼緊密,也不需要您在 AWS 和 Kubernetes API 之間跳來跳去。

    Karpenter 將實例編排職責集中在單一系統中,這更簡單、更穩定和更了解叢集。Karpenter 旨在克服 Cluster Autoscaler 所面臨的一些挑戰,提供以下簡化方式:

    • 根據工作負載需求配置節點。
    • 使用靈活的 NodePool 選項創建不同的節點配置。而不是管理許多特定的自定義節點組,Karpenter 可以讓您使用單一、靈活的 NodePool 管理多樣化的工作負載容量。
    • 通過快速啟動節點和調度 pod 來實現更好的 pod 調度。

    有關使用 Karpenter 的信息和文檔,請訪問 karpenter.sh 網站。

    建議¶

    最佳實踐分為 Karpenter 本身、NodePools 和 pod 調度等部分。

    Karpenter 最佳實踐¶

    以下最佳實踐涵蓋與 Karpenter 本身相關的主題。

    將 Karpenter 用於需求變化的工作負載¶

    Karpenter 將縮放管理帶到比 Autoscaling Groups (ASGs) 和 Managed Node Groups (MNGs) 更接近 Kubernetes 原生 API 的地方。ASG 和 MNG 是 AWS 原生抽象,其縮放是基於 AWS 級別指標(如 EC2 CPU 負載)觸發的。Cluster Autoscaler 將 Kubernetes 抽象轉換為 AWS 抽象,但由於此原因而失去了一些靈活性,例如為特定可用區域進行調度。

    Karpenter 刪除了一層 AWS 抽象,直接將一些靈活性引入 Kubernetes。Karpenter 最適合用於工作負載遇到高峰值需求期或具有多樣化計算需求的叢集。MNG 和 ASG 適合於運行工作負載較為靜態和一致的叢集。您可以根據需求使用動態和靜態管理的節點。

    考慮其他自動縮放項目,當...¶

    您需要 Karpenter 正在開發的功能。由於 Karpenter 是一個相對較新的項目,如果您需要 Karpenter 尚未包含的功能,請暫時考慮其他自動縮放項目。

    在 EKS Fargate 或屬於節點組的工作節點上運行 Karpenter 控制器¶

    Karpenter 使用 Helm chart 進行安裝。Helm chart 安裝 Karpenter 控制器和一個 webhook pod 作為 Deployment,需要在控制器可用於縮放叢集之前運行。我們建議至少有一個小型節點組,至少有一個工作節點。或者,您可以通過為 karpenter 命名空間創建 Fargate 配置文件,在 EKS Fargate 上運行這些 pod。這將導致部署到此命名空間的所有 pod 在 EKS Fargate 上運行。不要在由 Karpenter 管理的節點上運行 Karpenter。

    避免對 Karpenter 使用自定義啟動模板¶

    Karpenter 強烈建議不要使用自定義啟動模板。使用自定義啟動模板會阻礙多架構支持、自動升級節點的能力和 securityGroup 發現。使用啟動模板也可能造成混淆,因為某些字段在 Karpenter 的 NodePools 中重複,而其他字段被 Karpenter 忽略,例如子網和實例類型。

    您通常可以通過使用自定義用戶數據和/或直接在 AWS 節點模板中指定自定義 AMI 來避免使用啟動模板。有關如何操作的更多信息,請參見 NodeClasses。

    排除不符合您工作負載的實例類型¶

    如果某些實例類型不是您叢集中正在運行的工作負載所需要的,請考慮使用 node.kubernetes.io/instance-type 密鑰將其排除。

    以下示例展示如何避免配置大型 Graviton 實例。

    - key: node.kubernetes.io/instance-type
  operator: NotIn
  values:
  - m6g.16xlarge
  - m6gd.16xlarge
  - r6g.16xlarge
  - r6gd.16xlarge
  - c6g.16xlarge

    在使用 Spot 時啟用中斷處理¶

    Karpenter 支持 原生中斷處理,通過 --interruption-queue-name CLI 參數啟用,該參數是 SQS 隊列的名稱。中斷處理監視即將發生的非自願中斷事件,這些事件會對您的工作負載造成中斷,例如:

    • Spot 中斷警告
    • 計劃的變更健康事件(維護事件)
    • 實例終止事件
    • 實例停止事件

    當 Karpenter 檢測到這些事件將發生在您的節點上時,它會自動隔離、排空和終止節點,以便在中斷事件發生之前為工作負載清理提供最大的時間。不建議將 AWS Node Termination Handler 與 Karpenter 一起使用,如此處所述。

    需要檢查點或其他形式的優雅排空,在關機前需要 2 分鐘的 pod 應該在其叢集中啟用 Karpenter 中斷處理。

    沒有出站互聯網訪問的 Amazon EKS 私有叢集¶

    在沒有到互聯網的路由的 VPC 中配置 EKS 叢集時,您必須確保您已根據 EKS 文檔中出現的私有叢集要求配置您的環境。此外,您需要確保在 VPC 中創建了 STS VPC 區域端點。否則,您將看到類似於下面出現的錯誤。

    {"level":"FATAL","time":"2024-02-29T14:28:34.392Z","logger":"controller","message":"Checking EC2 API connectivity, WebIdentityErr: failed to retrieve credentials\ncaused by: RequestError: send request failed\ncaused by: Post \"https://sts.<region>.amazonaws.com/\": dial tcp 54.239.32.126:443: i/o timeout","commit":"596ea97"}

    這些更改在私有叢集中是必要的,因為 Karpenter 控制器使用 IAM 角色服務帳戶 (IRSA)。配置有 IRSA 的 pod 通過調用 AWS Security Token Service (AWS STS) API 獲取憑據。如果沒有出站互聯網訪問,您必須在 VPC 中創建和使用 AWS STS VPC 端點。

    私有叢集還要求您創建一個 VPC 端點用於 SSM。當 Karpenter 試圖配置一個新節點時,它會查詢啟動模板配置和一個 SSM 參數。如果您的 VPC 中沒有 SSM VPC 端點,它將導致以下錯誤:

    {"level":"ERROR","time":"2024-02-29T14:28:12.889Z","logger":"controller","message":"Unable to hydrate the AWS launch template cache, RequestCanceled: request context canceled\ncaused by: context canceled","commit":"596ea97","tag-key":"karpenter.k8s.aws/cluster","tag-value":"eks-workshop"}
...
{"level":"ERROR","time":"2024-02-29T15:08:58.869Z","logger":"controller.nodeclass","message":"discovering amis from ssm, getting ssm parameter \"/aws/service/eks/optimized-ami/1.27/amazon-linux-2/recommended/image_id\", RequestError: send request failed\ncaused by: Post \"https://ssm.<region>.amazonaws.com/\": dial tcp 67.220.228.252:443: i/o timeout","commit":"596ea97","ec2nodeclass":"default","query":"/aws/service/eks/optimized-ami/1.27/amazon-linux-2/recommended/image_id"}

    沒有 VPC 端點用於 Price List Query API。 因此,定價數據將隨時間變得過時。 Karpenter 通過在其二進制文件中包含按需定價數據來解決這個問題,但只在 Karpenter 升級時更新該數據。 定價數據的請求失敗將導致以下錯誤消息:

    {"level":"ERROR","time":"2024-02-29T15:08:58.522Z","logger":"controller.pricing","message":"retreiving on-demand pricing data, RequestError: send request failed\ncaused by: Post \"https://api.pricing.<region>.amazonaws.com/\": dial tcp 18.196.224.8:443: i/o timeout; RequestError: send request failed\ncaused by: Post \"https://api.pricing.<region>.amazonaws.com/\": dial tcp 18.185.143.117:443: i/o timeout","commit":"596ea97"}

    總之,要在完全私有的 EKS 叢集中使用 Karpenter,您需要創建以下 VPC 端點:

    com.amazonaws.<region>.ec2
com.amazonaws.<region>.ecr.api
com.amazonaws.<region>.ecr.dkr
com.amazonaws.<region>.s3 – 用於拉取容器映像
com.amazonaws.<region>.sts – 用於 IAM 角色服務帳戶
com.amazonaws.<region>.ssm - 用於解析默認 AMI
com.amazonaws.<region>.sqs - 用於訪問 SQS(如果使用中斷處理)

    Note

    Karpenter (控制器和 webhook 部署)容器映像必須位於 Amazon ECR 私有或其他 VPC 內部可訪問的私有註冊表中。原因是 Karpenter 控制器和 webhook pod 當前使用公共 ECR 映像。如果這些映像在 VPC 內部或與 VPC 對等的網絡中不可用,Kubernetes 在嘗試從 ECR 公有拉取這些映像時會出現 Image pull 錯誤。

    有關更多信息,請參見 Issue 988 和 Issue 1157。

    創建 NodePools¶

    以下最佳實踐涵蓋與創建 NodePools 相關的主題。

    當...時創建多個 NodePools¶

    當不同的團隊共享一個叢集並需要在不同的工作節點上運行其工作負載,或者有不同的操作系統或實例類型要求時,請創建多個 NodePools。例如,一個團隊可能想使用 Bottlerocket,而另一個團隊可能想使用 Amazon Linux。同樣,一個團隊可能有權訪問其他團隊不需要的昂貴的 GPU 硬件。使用多個 NodePools 可確保每個團隊都可以訪問最合適的資產。

    創建互斥或加權的 NodePools¶

    建議創建互斥或加權的 NodePools,以提供一致的調度行為。如果不是這樣,並且匹配多個 NodePools,Karpenter 將隨機選擇使用哪一個,從而導致意外結果。創建多個 NodePools 的有用示例包括以下內容:

    創建一個具有 GPU 的 NodePool,並僅允許特殊工作負載在這些(昂貴的)節點上運行:

    # NodePool for GPU Instances with Taints
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: gpu
spec:
  disruption:
    consolidateAfter: 1m0s
    consolidationPolicy: WhenEmpty
    expireAfter: Never
  template:
    metadata: {}
    spec:
      nodeClassRef:
        name: default
      requirements:
      - key: node.kubernetes.io/instance-type
        operator: In
        values:
        - p3.8xlarge
        - p3.16xlarge
      - key: kubernetes.io/os
        operator: In
        values:
        - linux
      - key: kubernetes.io/arch
        operator: In
        values:
        - amd64
      - key: karpenter.sh/capacity-type
        operator: In
        values:
        - on-demand
      taints:
      - effect: NoSchedule
        key: nvidia.com/gpu
        value: "true"

    具有容忍度的部署:

    # Deployment of GPU Workload will have tolerations defined
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inflate-gpu
spec:
  ...
    spec:
      tolerations:
      - key: "nvidia.com/gpu"
        operator: "Exists"
        effect: "NoSchedule"

    對於另一個團隊的一般部署,NodePool 規範可以包括 nodeAffinify。部署然後可以使用 nodeSelectorTerms 來匹配 billing-team。

    # NodePool for regular EC2 instances
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: generalcompute
spec:
  disruption:
    expireAfter: Never
  template:
    metadata:
      labels:
        billing-team: my-team
    spec:
      nodeClassRef:
        name: default
      requirements:
      - key: node.kubernetes.io/instance-type
        operator: In
        values:
        - m5.large
        - m5.xlarge
        - m5.2xlarge
        - c5.large
        - c5.xlarge
        - c5a.large
        - c5a.xlarge
        - r5.large
        - r5.xlarge
      - key: kubernetes.io/os
        operator: In
        values:
        - linux
      - key: kubernetes.io/arch
        operator: In
        values:
        - amd64
      - key: karpenter.sh/capacity-type
        operator: In
        values:
        - on-demand

    使用 nodeAffinity 的部署:

    # Deployment will have spec.affinity.nodeAffinity defined
kind: Deployment
metadata:
  name: workload-my-team
spec:
  replicas: 200
  ...
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                - key: "billing-team"
                  operator: "In"
                  values: ["my-team"]

    使用計時器(TTL)自動刪除叢集中的節點¶

    您可以對配置的節點使用計時器來設置何時刪除沒有工作負載 pod 或已到期的節點。節點到期可用作升級的一種方式,使節點被退役並替換為更新的版本。有關使用 spec.disruption.expireAfter 配置節點到期的信息,請參見 Karpenter 文檔中的到期。

    避免過度限制 Karpenter 可以配置的實例類型,特別是在使用 Spot 時¶

    使用 Spot 時,Karpenter 使用 Price Capacity Optimized 分配策略來配置 EC2 實例。此策略指示 EC2 從數量最多的池中配置實例,這些池的中斷風險最低。EC2 Fleet 然後從這些池中最便宜的池請求 Spot 實例。您允許 Karpenter 使用的實例類型越多,EC2 就越能優化您的 spot 實例的運行時間。默認情況下,Karpenter 將使用 EC2 在叢集部署所在區域和可用區中提供的所有實例類型。Karpenter 根據待處理的 pod 智能地從所有實例類型集中選擇,以確保您的 pod 被調度到適當大小和配備的實例上。例如,如果您的 pod 不需要 GPU,Karpenter 將不會將您的 pod 調度到支持 GPU 的 EC2 實例類型上。當您不確定使用哪些實例類型時,可以運行 Amazon ec2-instance-selector 來生成一個符合您計算需求的實例類型列表。例如,CLI 將內存 vCPU、架構和區域作為輸入參數,並為您提供滿足這些約束條件的 EC2 實例列表。

    $ ec2-instance-selector --memory 4 --vcpus 2 --cpu-architecture x86_64 -r ap-southeast-1
c5.large
c5a.large
c5ad.large
c5d.large
c6i.large
t2.medium
t3.medium
t3a.medium

    在使用 Spot 實例時,您不應該對 Karpenter 設置太多限制,因為這樣做可能會影響您應用程序的可用性。例如,某個特定類型的所有實例都被回收,並且沒有合適的替代品可用。您的 pod 將保持待處理狀態,直到為配置的實例類型補充 spot 容量。通過在不同可用區域中分散您的實例,可以減少不足容量錯誤的風險,因為 spot 池在 AZ 之間是不同的。但是,一般最佳實踐是在使用 Spot 時允許 Karpenter 使用多樣化的實例類型。

    調度 Pod¶

    以下最佳實踐與使用 Karpenter 進行節點配置時部署 pod 相關。

    遵循 EKS 高可用性最佳實踐¶

    如果您需要運行高可用性應用程序,請遵循一般 EKS 最佳實踐建議。有關如何跨節點和區域分散 pod 的詳細信息,請參見 Karpenter 文檔中的拓撲傳播。使用中斷預算設置需要維護的最小可用 pod 數,以防止嘗試驅逐或刪除 pod。

    使用分層約束來限制雲提供商提供的計算功能¶

    Karpenter 的分層約束模型允許您創建複雜的 NodePool 和 pod 部署約束集,以獲得最佳的 pod 調度匹配。 pod 規範可以請求的約束示例包括以下內容:

    • 需要在特定可用區域運行。例如,您有一個 pod 必須與另一個在特定可用區域上運行的 EC2 實例通信。如果您的目標是減少 VPC 中的跨 AZ 流量,您可能希望將 pod 與 EC2 實例位於同一 AZ 中。這種定位通常通過節點選擇器來實現。有關節點選擇器的更多信息,請參考 Kubernetes 文檔。
    • 需要特定類型的處理器或其他硬件。有關 pod 規範示例,請參見 Karpenter 文檔中的加速器部分,該示例要求 pod 在 GPU 上運行。

    創建計費警報來監控您的計算支出¶

    當您配置叢集自動縮放時,您應該創建計費警報來警告您支出是否已超過閾值,並在 Karpenter 配置中添加資源限制。使用 Karpenter 設置資源限制類似於設置 AWS 自動縮放組的最大容量,它代表 Karpenter NodePool 可以實例化的最大計算資源量。

    Note

    無法為整個叢集設置全局限制。限制適用於特定的 NodePools。

    下面的片段告訴 Karpenter 只能配置最多 1000 個 CPU 核心和 1000Gi 的內存。只有在達到或超過限制時,Karpenter 才會停止添加容量。當超過限制時,Karpenter 控制器將在控制器日誌中寫入 memory resource usage of 1001 exceeds limit of 1000 或類似的消息。如果您將容器日誌路由到 CloudWatch 日誌,您可以創建一個指標過濾器來查找日誌中的特定模式或術語,然後創建一個CloudWatch 警報來在您配置的指標閾值被違反時提醒您。

    有關在 Karpenter 中使用限制的更多信息,請參見 Karpenter 文檔中的設置資源限制。

    spec:
  limits:
    cpu: 1000
    memory: 1000Gi

    如果您不使用限制或限制 Karpenter 可以配置的實例類型,Karpenter 將繼續根據需要向您的叢集添加計算容量。雖然以這種方式配置 Karpenter 允許您的叢集自由擴展,但也可能產生重大的成本影響。正因為如此,我們建議您配置計費警報。計費警報允許您在您的帳戶中的計算估計費用超過定義的閾值時得到警報和主動通知。有關更多信息,請參見設置 Amazon CloudWatch 計費警報以主動監控估計費用。

    您還可能想啟用成本異常檢測,這是 AWS 成本管理功能之一,它使用機器學習持續監控您的成本和使用情況,以檢測異常支出。更多信息可以在 AWS 成本異常檢測入門指南中找到。如果您已經在 AWS Budgets 中創建了預算,您也可以配置一個操作來通知您何時違反了特定的閾值。使用預算操作,您可以發送電子郵件、發送消息到 SNS 主題或發送消息到 Slack 等聊天機器人。有關更多信息,請參見配置 AWS Budgets 操作。

    使用 karpenter.sh/do-not-disrupt 註解來防止 Karpenter 取消配置節點¶

    如果您在 Karpenter 配置的節點上運行關鍵應用程序,例如長時間運行的批處理作業或有狀態應用程序,並且節點的 TTL 已過期,應用程序將在實例終止時中斷。通過在 pod 上添加 karpenter.sh/karpenter.sh/do-not-disrupt 註解,您正在指示 Karpenter 保留節點,直到 Pod 被終止或刪除 karpenter.sh/do-not-disrupt 註解。有關更多信息,請參見中斷文檔。

    如果節點上剩下的唯一非 daemonset pod 是與作業相關的 pod,只要作業狀態為成功或失敗,Karpenter 就能夠定位和終止這些節點。

    在使用整合時為所有非 CPU 資源配置 requests=limits¶

    整合和調度通常通過比較 pod 資源請求與節點上可分配資源的數量來工作。不考慮資源限制。例如,具有比內存請求大的內存限制的 pod 可以突破請求。如果同一節點上的多個 pod 同時突破,這可能會導致由於內存不足(OOM)而終止某些 pod。整合可能會使這種情況更容易發生,因為它只考慮 pod 的請求就努力將 pod 打包到節點上。

    使用 LimitRanges 為資源請求和限制配置默認值¶

    由於 Kubernetes 沒有設置默認請求或限制,容器對主機、CPU 和內存資源的消耗是無限的。Kubernetes 調度程序查看 pod 的總請求(pod 容器的總請求或 pod Init 容器的總資源中的較高者)來確定將 pod 調度到哪個工作節點上。同樣,Karpenter 考慮 pod 的請求來確定配置哪種類型的實例。您可以使用限制範圍為命名空間應用合理的默認值,以防某些 pod 未指定資源請求。

    請參見為命名空間配置默認內存請求和限制
