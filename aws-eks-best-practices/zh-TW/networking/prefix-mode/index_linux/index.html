    Linux 的前綴模式¶

    Amazon VPC CNI 將網路前綴分配給 Amazon EC2 網路介面，以增加節點可用的 IP 地址數量並提高每個節點的 Pod 密度。您可以配置 Amazon VPC CNI 加載項的 1.9.0 或更高版本，將 IPv4 和 IPv6 CIDR 分配給網路介面，而不是分配單個次要 IP 地址。

    前綴模式在 IPv6 集群上默認啟用，這是唯一支持的選項。VPC CNI 將 /80 IPv6 前綴分配給 ENI 上的一個插槽。有關更多信息，請參考本指南的 IPv6 部分。

    使用前綴分配模式，每個實例類型的彈性網路介面的最大數量保持不變，但您現在可以配置 Amazon VPC CNI 分配 /28 (16 個 IP 地址) IPv4 地址前綴，而不是將單個 IPv4 地址分配給網路介面上的插槽。當 ENABLE_PREFIX_DELEGATION 設置為 true 時，VPC CNI 會從分配給 ENI 的前綴中分配 IP 地址給 Pod。請按照 EKS 用戶指南 中提到的說明啟用前綴 IP 模式。

    您可以分配給網路介面的 IP 地址的最大數量取決於實例類型。您分配給網路介面的每個前綴都算作一個 IP 地址。例如,c5.large 實例的限制是每個網路介面 10 個 IPv4 地址。每個網路介面都有一個主要 IPv4 地址。如果網路介面沒有任何次要 IPv4 地址,您可以將最多 9 個前綴分配給網路介面。對於分配給網路介面的每個額外 IPv4 地址,您可以分配一個前綴更少。請查看 AWS EC2 文檔,了解每個實例類型的每個網路介面的 IP 地址數和將前綴分配給網路介面。

    在工作節點初始化期間,VPC CNI 會將一個或多個前綴分配給主要 ENI。CNI 通過維護一個預熱池來預先分配前綴,以實現更快的 Pod 啟動。可以通過設置環境變量來控制預熱池中要保留的前綴數量。

    • WARM_PREFIX_TARGET,超出當前需求的前綴數量。
    • WARM_IP_TARGET,超出當前需求的 IP 地址數量。
    • MINIMUM_IP_TARGET,任何時候都要保持可用的最小 IP 地址數量。
    • 如果設置了 WARM_IP_TARGET 和 MINIMUM_IP_TARGET,它們將覆蓋 WARM_PREFIX_TARGET。

    隨著更多 Pod 被調度,將為現有的 ENI 請求其他前綴。首先,VPC CNI 嘗試將新前綴分配給現有的 ENI。如果 ENI 已滿,VPC CNI 將嘗試為節點分配一個新的 ENI。將附加新的 ENI,直到達到最大 ENI 限制(由實例類型定義)。當附加新的 ENI 時,ipamd 將分配一個或多個前綴,以維持 WARM_PREFIX_TARGET、WARM_IP_TARGET 和 MINIMUM_IP_TARGET 設置。

    建議¶

    在以下情況下使用前綴模式¶

    如果您在工作節點上遇到 Pod 密度問題,請使用前綴模式。為了避免 VPC CNI 錯誤,我們建議在遷移到前綴模式之前,檢查子網是否有足夠的連續地址塊用於 /28 前綴。有關子網預留的詳細信息,請參考"使用子網預留來避免子網碎片化(IPv4)"部分。

    為了向後兼容,已將 max-pods 限制設置為支持次要 IP 模式。要增加 Pod 密度,請將 max-pods 值指定給 Kubelet,並將 --use-max-pods=false 作為節點的用戶數據。您可以考使用 max-pod-calculator.sh 腳本來計算 EKS 推薦的給定實例類型的最大 Pod 數量。請參考 EKS 用戶指南中的示例用戶數據。

    ./max-pods-calculator.sh --instance-type m5.large --cni-version ``1.9``.0 --cni-prefix-delegation-enabled

    前綴分配模式對於使用 CNI 自定義網絡的用戶特別相關,在這種情況下,主 ENI 不用於 Pod。使用前綴分配,您仍然可以在幾乎所有 Nitro 實例類型上附加更多 IP,即使主 ENI 不用於 Pod。

    在以下情況下避免使用前綴模式¶

    如果您的子網非常碎片化,沒有足夠的可用 IP 地址來創建 /28 前綴,請避免使用前綴模式。如果生成前綴的子網碎片化嚴重(一個使用量很大的子網,二次 IP 地址分散),前綴附加可能會失敗。這個問題可以通過創建一個新的子網並預留一個前綴來避免。

    在前綴模式下,分配給工作節點的安全組是由 Pod 共享的。如果您有一個安全性要求,需要在共享計算資源上運行具有不同網絡安全要求的應用程序,請考慮使用 Pod 的安全組。

    在同一個節點組中使用相似的實例類型¶

    您的節點組可能包含許多類型的實例。如果一個實例的最大 Pod 數量很低,則該值將應用於節點組中的所有節點。考慮在節點組中使用相似的實例類型,以最大化節點使用率。如果您使用 Karpenter 進行自動節點伸縮,我們建議在 provisioner API 的 requirements 部分配置 node.kubernetes.io/instance-type。

    Warning

    節點組中所有節點的最大 Pod 數量由任何單一實例類型的最低最大 Pod 數量定義。

    配置 WARM_PREFIX_TARGET 以節省 IPv4 地址¶

    安裝清單中 WARM_PREFIX_TARGET 的默認值為 1。在大多數情況下,WARM_PREFIX_TARGET 的推薦值 1 將提供快速 Pod 啟動時間,同時最大限度地減少分配給實例的未使用 IP 地址。

    如果您需要進一步節省每個節點的 IPv4 地址,請使用 WARM_IP_TARGET 和 MINIMUM_IP_TARGET 設置,這些設置將覆蓋配置的 WARM_PREFIX_TARGET。通過將 WARM_IP_TARGET 設置為小於 16 的值,您可以防止 CNI 保留整個多餘的前綴。

    優先分配新前綴而不是附加新的 ENI¶

    將額外的前綴分配給現有的 ENI 是一個比創建和附加新的 ENI 到實例更快的 EC2 API 操作。使用前綴可以提高性能,同時節省 IPv4 地址分配。分配前綴通常在一秒鐘內完成,而附加新的 ENI 可能需要 10 秒鐘。對於大多數用例,當在前綴模式下運行時,CNI 每個工作節點只需要一個 ENI。如果您可以承受(在最壞的情況下)每個節點最多 15 個未使用的 IP,我們強烈建議使用更新的前綴分配網絡模式,並實現由此帶來的性能和效率收益。

    使用子網預留來避免子網碎片化(IPv4)¶

    當 EC2 將 /28 IPv4 前綴分配給 ENI 時,它必須是來自您子網的一個連續的 IP 地址塊。如果生成前綴的子網碎片化嚴重(一個高度使用的子網,二次 IP 地址分散),前綴附加可能會失敗,您將在 VPC CNI 日誌中看到以下錯誤消息:

    failed to allocate a private IP/Prefix address: InsufficientCidrBlocks: There are not enough free cidr blocks in the specified subnet to satisfy the request.

    為了避免碎片化並為創建前綴保留足夠的連續空間,您可以使用 VPC 子網 CIDR 預留在子網內預留 IP 空間供前綴專用使用。一旦創建了預留,VPC CNI 插件將調用 EC2 API 從預留的空間分配前綴。

    建議創建一個新的子網,為前綴預留空間,並為在該子網上運行的工作節點啟用 VPC CNI 的前綴分配。如果新的子網僅用於在您的 EKS 集群中運行使用 VPC CNI 前綴分配的 Pod,則可以跳過前綴預留步驟。

    避免降級 VPC CNI¶

    前綴模式適用於 VPC CNI 1.9.0 及更高版本。一旦啟用前綴模式並將前綴分配給 ENI,就必須避免降級 Amazon VPC CNI 加載項到 1.9.0 以下的版本。如果您決定降級 VPC CNI,您必須刪除並重新創建節點。

    在過渡到前綴委派期間替換所有節點¶

    我們強烈建議您創建新的節點組來增加可用 IP 地址的數量,而不是對現有的工作節點進行滾動替換。將所有現有節點隔離並排空,以安全地驅逐所有現有 Pod。為了防止服務中斷,我們建議在生產集群上為關鍵工作負載實施 Pod 中斷預算。新節點上的 Pod 將從分配給 ENI 的前綴中獲取 IP。確認 Pod 正在運行後,您可以刪除舊節點和節點組。如果您使用托管節點組,請按照此處提到的步驟安全刪除節點組。
