    事故響應和取證¶

    您能夠迅速應對事故發生,可以幫助最大限度地減少違規造成的損害。擁有可靠的警報系統,能夠警告您可疑行為,是良好事故響應計劃的第一步。當事故發生時,您必須快速決定是銷毀並替換受影響的容器,還是隔離和檢查容器。如果您選擇隔離容器作為取證調查和根本原因分析的一部分,則應遵循以下一系列活動:

    示例事故響應計劃¶

    識別有問題的 Pod 和工作節點¶

    您的首要行動應該是隔離損害。首先確定違規發生的位置,並將該 Pod 及其節點與基礎設施的其他部分隔離。

    使用工作負載名稱識別有問題的 Pod 和工作節點¶

    如果您知道有問題的 pod 的名稱和命名空間,您可以按如下方式識別運行該 pod 的工作節點:

    kubectl get pods <name> --namespace <namespace> -o=jsonpath='{.spec.nodeName}{"\n"}'

    如果 工作負載資源 如 Deployment 已被入侵,那麼屬於該工作負載資源的所有 pod 很可能也已被入侵。使用以下命令列出工作負載資源的所有 pod 及其運行的節點:

    selector=$(kubectl get deployments <name> \
 --namespace <namespace> -o json | jq -j \
'.spec.selector.matchLabels | to_entries | .[] | "\(.key)=\(.value)"')

kubectl get pods --namespace <namespace> --selector=$selector \
-o json | jq -r '.items[] | "\(.metadata.name) \(.spec.nodeName)"'

    上述命令適用於 deployments。您可以對其他工作負載資源(如 replicasets、statefulsets 等)運行相同的命令。

    使用服務帳戶名稱識別有問題的 Pod 和工作節點¶

    在某些情況下,您可能會發現服務帳戶已被入侵。使用該服務帳戶的 pod 很可能也已被入侵。您可以使用以下命令識別使用該服務帳戶的所有 pod 及其運行的節點:

    kubectl get pods -o json --namespace <namespace> | \
    jq -r '.items[] |
    select(.spec.serviceAccount == "<service account name>") |
    "\(.metadata.name) \(.spec.nodeName)"'

    識別使用有漏洞或被入侵映像的 Pod 和工作節點¶

    在某些情況下,您可能會發現集群中的 pod 使用的容器映像是惡意的或已被入侵。如果發現容器映像包含惡意軟件、是已知的不良映像或已被利用的 CVE,則應將其視為惡意或已被入侵。您應該認為使用該容器映像的所有 pod 都已被入侵。您可以使用以下命令識別使用該映像的 pod 及其運行的節點:

    IMAGE=<Name of the malicious/compromised image>

kubectl get pods -o json --all-namespaces | \
    jq -r --arg image "$IMAGE" '.items[] | 
    select(.spec.containers[] | .image == $image) | 
    "\(.metadata.name) \(.metadata.namespace) \(.spec.nodeName)"'

    通過創建拒絕所有入口和出口流量的網絡策略來隔離 Pod¶

    拒絕所有流量的規則可能有助於阻止正在進行的攻擊,方法是切斷 pod 的所有連接。以下網絡策略將應用於帶有標籤 app=web 的 pod。

    apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
spec:
  podSelector:
    matchLabels: 
      app: web
  policyTypes:
  - Ingress
  - Egress

    Attention

    如果攻擊者已經獲得了底層主機的訪問權限,網絡策略可能會無效。如果您懷疑發生了這種情況,您可以使用 AWS 安全組 將受感染的主機與其他主機隔離。更改主機的安全組時,請注意它將影響在該主機上運行的所有容器。

    如果必要,撤銷分配給 pod 或工作節點的臨時安全憑證¶

    如果工作節點被分配了一個 IAM 角色,允許 pod 訪問其他 AWS 資源,請刪除該角色,以防止攻擊造成進一步損害。同樣,如果 pod 被分配了 IAM 角色,請評估是否可以安全地從該角色中刪除 IAM 策略,而不影響其他工作負載。

    隔離工作節點¶

    通過隔離受影響的工作節點,您可以告知調度程序避免在該節點上調度 pod。這將允許您在不中斷其他工作負載的情況下,移除該節點進行取證研究。

    Info

    此指南不適用於 Fargate,在 Fargate 中每個 Fargate pod 都在自己的沙箱環境中運行。不是隔離,而是通過應用拒絕所有入口和出口流量的網絡策略來隔離受影響的 Fargate pod。

    為受影響的工作節點啟用終止保護¶

    攻擊者可能會試圖通過終止受影響的節點來抹去他們的罪行。啟用 終止保護 可以防止這種情況發生。 實例縮容保護 將保護節點免受縮容事件的影響。

    Warning

    您無法對 Spot 實例啟用終止保護。

    用標籤標記有問題的 Pod/節點,表示它們正在進行活動調查¶

    這將警告集群管理員,在調查完成之前不要篡改受影響的 Pod/節點。

    在工作節點上捕獲易失性工件¶

    • 捕獲操作系統內存。這將捕獲 Docker 守護程序(或其他容器運行時)及其每個容器的子進程。這可以使用 LiME 和 Volatility 等工具完成,或通過 Automated Forensics Orchestrator for Amazon EC2 等更高級別的工具完成,這些工具建立在它們之上。
    • 執行 netstat 樹轉儲,查看正在運行的進程和開放的端口。這將捕獲 Docker 守護程序及其每個容器的子進程。
    • 在證據被篡改之前,運行命令保存容器級別的狀態。您可以利用容器運行時的功能來捕獲有關當前運行容器的信息。例如,使用 Docker,您可以執行以下操作:
    • docker top CONTAINER 查看正在運行的進程。
    • docker logs CONTAINER 查看守護程序級別保存的日誌。

    • docker inspect CONTAINER 查看容器的各種信息。

      使用 nerdctl CLI 可以對 containerd 執行類似的操作,而不是使用 docker(例如 nerdctl inspect)。根據容器運行時的不同,可能還有一些其他的命令可用。例如,Docker 有 docker diff 查看容器文件系統的變更,或 docker checkpoint 保存包括易失性內存(RAM)在內的所有容器狀態。請參閱 此 Kubernetes 博客文章 了解有關 containerd 或 CRI-O 運行時類似功能的討論。

    • 暫停容器以進行取證捕獲。

    • 對實例的 EBS 卷進行快照。

    重新部署受損的 Pod 或工作負載資源¶

    收集完取證分析所需的數據後,您可以重新部署受損的 pod 或工作負載資源。

    首先,部署修復了導致漏洞的問題的新替換 pod。然後刪除易受攻擊的 pod。

    如果易受攻擊的 pod 由較高級別的 Kubernetes 工作負載資源(例如 Deployment 或 DaemonSet)管理,刪除它們將調度新的 pod。因此,易受攻擊的 pod 將再次啟動。在這種情況下,您應該在修復漏洞後部署一個新的替換工作負載資源,然後刪除易受攻擊的工作負載。

    建議¶

    查看 AWS 安全事故響應白皮書¶

    雖然本節簡要概述了處理疑似安全違規的一些建議,但該主題在 AWS 安全事故響應指南 白皮書中有詳盡的介紹。

    進行安全演習¶

    將您的安全從業人員分成 2 個團隊:紅隊和藍隊。紅隊將專注於探測不同系統的漏洞,而藍隊將負責防禦。如果您沒有足夠的安全從業人員來創建單獨的團隊,請考慮聘請一個外部實體,他們具有 Kubernetes 漏洞利用的知識。

    Kubesploit 是 CyberArk 開發的一個滲透測試框架,您可以使用它來進行安全演習。與其他工具(僅掃描集群中的漏洞)不同,kubesploit模擬了真實世界的攻擊。這為您的藍隊提供了一個實踐應對攻擊的機會,並評估其有效性。

    對您的集群進行滲透測試¶

    定期攻擊您自己的集群可以幫助您發現漏洞和配置錯誤。開始之前,請遵循 滲透測試指南 在對集群進行測試之前。
