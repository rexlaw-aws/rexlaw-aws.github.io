    保護基礎設施(主機)¶

    與保護容器映像同樣重要的是保護運行它們的基礎設施。本節探討了減輕直接針對主機的攻擊風險的不同方法。這些指南應與運行時安全部分概述的指南一起使用。

    建議¶

    使用針對運行容器優化的操作系統¶

    考慮使用 Flatcar Linux、Project Atomic、RancherOS 和 Bottlerocket，這是 AWS 為運行 Linux 容器而設計的專用操作系統。它包括減少攻擊面、在啟動時驗證磁盤映像以及使用 SELinux 強制執行權限邊界。

    或者,使用 EKS 優化 AMI 作為您的 Kubernetes 工作節點。EKS 優化 AMI 定期發布,包含運行容器化工作負載所需的最小操作系統包和二進制文件。

    請參考 Amazon EKS AMI RHEL Build Specification 了解可用於使用 Hashicorp Packer 構建在 Red Hat Enterprise Linux 上運行的自定義 Amazon EKS AMI 的示例配置腳本。此腳本可進一步用於構建符合 STIG 要求的自定義 EKS AMI。

    保持工作節點操作系統更新¶

    無論您使用像 Bottlerocket 這樣的容器優化主機操作系統,還是像 EKS 優化 AMI 這樣更大但仍最小化的 Amazon 機器映像,最佳做法都是保持這些主機操作系統映像與最新的安全修補程序保持同步。

    對於 EKS 優化 AMI,定期檢查 CHANGELOG 和/或 發行說明頻道,並將更新的工作節點映像自動部署到您的集群中。

    將基礎設施視為不可變,並自動替換工作節點¶

    而不是進行就地升級,請在新的修補程序或更新可用時替換您的工作節點。這可以通過以下方式實現。您可以將實例添加到使用最新 AMI 的現有自動伸縮組中,並依次隔離和排空節點,直到組中的所有節點都已替換為最新 AMI。或者,您可以將實例添加到新的節點組中,同時依次隔離和排空舊節點組中的節點,直到所有節點都已替換。EKS 托管節點組使用第一種方法,並在新 AMI 可用時在控制台中顯示消息以升級您的工作節點。eksctl 也有為使用最新 AMI 創建節點組以及優雅地隔離和排空節點組中節點的機制。如果您決定使用其他方法替換工作節點,強烈建議您自動化該過程,以最小化人工監督,因為您可能需要定期替換工作節點,因為發布了新的更新/修補程序,以及控制平面升級時。

    使用 EKS Fargate,AWS 將在更新可用時自動更新底層基礎設施。通常這可以無縫完成,但有時更新可能會導致您的 pod 被重新調度。因此,我們建議您在以 Fargate pod 運行應用程序時創建具有多個副本的部署。

    定期運行 kube-bench 以驗證是否符合 Kubernetes CIS 基準¶

    kube-bench 是 Aqua 的一個開源項目,用於根據 Kubernetes CIS 基準評估您的集群。該基準描述了保護未管理的 Kubernetes 集群的最佳做法。CIS Kubernetes 基準涵蓋控制平面和數據平面。由於 Amazon EKS 提供了完全托管的控制平面,因此 CIS Kubernetes 基準中的並非所有建議都適用。為了確保此範圍反映了 Amazon EKS 的實施方式,AWS 創建了 CIS Amazon EKS 基準。EKS 基準繼承自 CIS Kubernetes 基準,並從社區獲取有關 EKS 集群特定配置注意事項的其他輸入。

    在 EKS 集群上運行 kube-bench 時,請遵循 Aqua Security 的這些說明。有關更多信息,請參見介紹 CIS Amazon EKS 基準。

    最小化對工作節點的訪問¶

    不要啟用 SSH 訪問,而是使用 SSM Session Manager 遠程訪問主機。與可能丟失、複製或共享的 SSH 密鑰不同,Session Manager 允許您使用 IAM 控制對 EC2 實例的訪問。此外,它還提供了對在實例上運行的命令的審核跟蹤和日誌。

    從 2020 年 8 月 19 日開始,托管節點組支持自定義 AMI 和 EC2 啟動模板。這允許您將 SSM 代理嵌入 AMI 或在引導工作節點時安裝它。如果您不想修改優化 AMI 或 ASG 的啟動模板,可以使用像這個示例中那樣的 DaemonSet 安裝 SSM 代理。

    用於基於 SSM 的 SSH 訪問的最小 IAM 策略¶

    AmazonSSMManagedInstanceCore AWS 托管策略包含了一些不需要用於 SSM Session Manager / SSM RunCommand 的權限,如果您只是想避免 SSH 訪問的話。特別值得關注的是 ssm:GetParameter(s) 的 * 權限,這將允許該角色訪問參數存儲區中的所有參數(包括使用 AWS 托管 KMS 密鑰配置的 SecureStrings)。

    以下 IAM 策略包含了通過 SSM Systems Manager 啟用節點訪問所需的最小權限集。

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableAccessViaSSMSessionManager",
      "Effect": "Allow",
      "Action": [
        "ssmmessages:OpenDataChannel",
        "ssmmessages:OpenControlChannel",
        "ssmmessages:CreateDataChannel",
        "ssmmessages:CreateControlChannel",
        "ssm:UpdateInstanceInformation"
      ],
      "Resource": "*"
    },
    {
      "Sid": "EnableSSMRunCommand",
      "Effect": "Allow",
      "Action": [
        "ssm:UpdateInstanceInformation",
        "ec2messages:SendReply",
        "ec2messages:GetMessages",
        "ec2messages:GetEndpoint",
        "ec2messages:FailMessage",
        "ec2messages:DeleteMessage",
        "ec2messages:AcknowledgeMessage"
      ],
      "Resource": "*"
    }
  ]
}

    有了這個策略,並安裝了 Session Manager 插件,您就可以運行

    aws ssm start-session --target [EKS_NODE_INSTANCE_ID]

    來訪問節點。

    Note

    您可能還想考慮添加權限來啟用 Session Manager 日誌記錄。

    將工作節點部署到私有子網¶

    通過將工作節點部署到私有子網,您可以最大限度地減少它們對互聯網的暴露,互聯網通常是攻擊的來源。從 2020 年 4 月 22 日開始,對托管節點組中節點的公共 IP 地址分配將由它們部署到的子網控制。在此之前,托管節點組中的節點會自動分配公共 IP。如果您選擇將工作節點部署到公共子網,請實施限制性的 AWS 安全組規則以限制其暴露。

    運行 Amazon Inspector 評估主機的暴露、漏洞和偏離最佳做法的情況¶

    您可以使用 Amazon Inspector 檢查您的節點是否存在意外網絡訪問和底層 Amazon EC2 實例上的漏洞。

    Amazon Inspector 只有在安裝並啟用了 Amazon EC2 Systems Manager (SSM) 代理程序時,才能為您的 Amazon EC2 實例提供常見漏洞和暴露(CVE)數據。此代理程序預安裝在幾個 Amazon Machine Images (AMI) 上,包括 EKS 優化 Amazon Linux AMI。無論 SSM 代理程序的狀態如何,都會掃描所有 Amazon EC2 實例以查找網絡可達性問題。有關為 Amazon EC2 配置掃描的更多信息,請參見掃描 Amazon EC2 實例。

    Attention

    Inspector 無法在運行 Fargate pod 的基礎設施上運行。

    替代方案¶

    運行 SELinux¶

    Info

    可用於 Red Hat Enterprise Linux (RHEL)、CentOS、Bottlerocket 和 Amazon Linux 2023

    SELinux 提供了一個額外的安全層,可以將容器彼此隔離,也可以將它們與主機隔離。SELinux 允許管理員為每個用戶、應用程序、進程和文件強制執行強制訪問控制(MAC)。可以將其視為一個後備,根據一組標籤限制對特定資源的操作。在 EKS 上,SELinux 可用於防止容器訪問彼此的資源。

    容器 SELinux 策略定義在 container-selinux 包中。Docker CE 需要此包(以及其依賴項),以便 Docker(或其他容器運行時)創建的進程和文件以有限的系統訪問權限運行。容器利用 container_t 標籤,這是 svirt_lxc_net_t 的別名。這些策略有效地防止容器訪問主機的某些功能。

    當您為 Docker 配置 SELinux 時,Docker 會自動將工作負載標記為 container_t 類型,並為每個容器分配一個唯一的 MCS 級別。這將隔離容器彼此。如果您需要更寬鬆的限制,可以在 SELinux 中創建自己的配置文件,該文件授予容器對文件系統特定區域的權限。這類似於 PSP,您可以為不同的容器/pod 創建不同的配置文件。例如,您可以有一個具有一組限制性控制的一般工作負載配置文件,以及另一個需要特權訪問的配置文件。

    容器的 SELinux 有一組可以配置的選項來修改默認限制。可以根據需要啟用或禁用以下 SELinux 布爾值:

    布爾值 默認 描述
    container_connect_any off 允許容器訪問主機上的特權端口。例如,如果您有一個需要將端口映射到主機上的 443 或 80 的容器。
    container_manage_cgroup off 允許容器管理 cgroup 配置。例如,運行 systemd 的容器將需要啟用此功能。
    container_use_cephfs off 允許容器使用 ceph 文件系統。

    默認情況下,容器被允許讀取/執行 /usr 下的內容,並讀取 /etc 的大部分內容。 /var/lib/docker 和 /var/lib/containers 下的文件具有 container_var_lib_t 標籤。要查看完整的默認標籤列表,請參見 container.fc 文件。

    docker container run -it \
  -v /var/lib/docker/image/overlay2/repositories.json:/host/repositories.json \
  centos:7 cat /host/repositories.json
# cat: /host/repositories.json: Permission denied

docker container run -it \
  -v /etc/passwd:/host/etc/passwd \
  centos:7 cat /host/etc/passwd
# cat: /host/etc/passwd: Permission denied

    標記為 container_file_t 的文件是容器可寫的唯一文件。如果您想使卷掛載可寫,您需要在末尾指定 :z 或 :Z。

    • :z 將重新標記文件,使容器可以讀寫
    • :Z 將重新標記文件,使只有容器可以讀寫
    ls -Z /var/lib/misc
# -rw-r--r--. root root system_u:object_r:var_lib_t:s0   postfix.aliasesdb-stamp

docker container run -it \
  -v /var/lib/misc:/host/var/lib/misc:z \
  centos:7 echo "Relabeled!"

ls -Z /var/lib/misc
#-rw-r--r--. root root system_u:object_r:container_file_t:s0 postfix.aliasesdb-stamp

    docker container run -it \
  -v /var/log:/host/var/log:Z \
  fluentbit:latest

    在 Kubernetes 中,重新標記略有不同。不是讓 Docker 自動重新標記文件,而是指定自定義 MCS 標籤來運行 pod。支持重新標記的卷將自動重新標記,以便它們可以被訪問。具有匹配 MCS 標籤的 pod 將能夠訪問該卷。如果需要嚴格隔離,請為每個 pod 設置不同的 MCS 標籤。

    securityContext:
  seLinuxOptions:
    # 為每個容器提供一個唯一的 MCS 標籤
    # 您也可以指定 user、role 和 type
    # 基於 type 和 level (svert) 的強制執行
    level: s0:c144:c154

    在這個例子中,s0:c144:c154 對應於容器被允許訪問的文件分配的 MCS 標籤。

    在 EKS 上,您可以創建允許運行特權容器(如 FluentD)的策略,並創建一個 SELinux 策略,允許它在不需要重新標記主機目錄的情況下從 /var/log 讀取。具有相同標籤的 pod 將能夠訪問相同的主機卷。

    我們已經實現了用於 Amazon EKS 的示例 AMI,它們在 CentOS 7 和 RHEL 7 上配置了 SELinux。這些 AMI 是為了演示滿足高度管制客戶(如 STIG、CJIS 和 C2S)要求的示例實現而開發的。

    Caution

    SELinux 將忽略類型為 unconfined 的容器。
