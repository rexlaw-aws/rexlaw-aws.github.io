    租戶隔離¶

    當我們考慮多租戶時,我們通常希望將用戶或應用程式與在共享基礎設施上運行的其他用戶或應用程式隔離。

    Kubernetes 是一個_單租戶編排器_,即控制平面的單個實例在集群中的所有租戶之間共享。然而,有各種 Kubernetes 物件可用於創建多租戶的假象。例如,可以實施命名空間和基於角色的訪問控制 (RBAC) 來邏輯上隔離租戶。同樣,配額和限制範圍可用於控制每個租戶可以使用的集群資源量。儘管如此,集群是提供強大安全邊界的唯一構造。這是因為能夠訪問集群內主機的攻擊者可以檢索該主機上掛載的_所有_密鑰、ConfigMaps 和卷。他們還可以模擬 Kubelet,這將允許他們操縱節點的屬性和/或在集群內橫向移動。

    以下部分將解釋如何在減輕使用單租戶編排器 (如 Kubernetes) 的風險的同時實施租戶隔離。

    軟多租戶¶

    使用軟多租戶,您使用本機 Kubernetes 構造,例如命名空間、角色和角色綁定以及網路策略,來創建租戶之間的邏輯分離。例如,RBAC 可以防止租戶訪問或操縱彼此的資源。配額和限制範圍控制每個租戶可以使用的集群資源量,而網路策略可以幫助防止部署到不同命名空間的應用程式相互通信。

    但是,這些控制措施並不能防止來自不同租戶的 pod 共享節點。如果需要更強的隔離,您可以使用節點選擇器、反親和性規則和/或污點和容忍來強制不同租戶的 pod 被調度到單獨的節點;通常稱為_專用租戶節點_。這可能會變得相當複雜和成本高昂,尤其是在擁有許多租戶的環境中。

    Attention

    使用命名空間實現的軟多租戶不允許您為租戶提供經過篩選的命名空間列表,因為命名空間是一種全局作用域的類型。如果租戶有查看特定命名空間的能力,它就可以查看集群中的所有命名空間。

    Warning

    使用軟多租戶,租戶默認保留查詢 CoreDNS 以獲取集群內運行的所有服務的記錄的能力。攻擊者可以通過在集群中的任何 pod 上運行 dig SRV *.*.svc.cluster.local 來利用這一點。如果您需要限制對集群內運行的服務的 DNS 記錄的訪問,請考虑使用 CoreDNS 的防火牆或策略插件。有關更多信息,請參見 https://github.com/coredns/policy#kubernetes-metadata-multi-tenancy-policy。

    Kiosk 是一個開源項目,可以幫助實現軟多租戶。它實現為一系列 CRD 和控制器,提供以下功能:

    • 帳戶和帳戶用戶以分隔共享 Kubernetes 集群中的租戶
    • 自助服務命名空間供應供帳戶用戶使用
    • 帳戶限制以確保在共享集群時的服務質量和公平性
    • 命名空間模板用於安全的租戶隔離和自助服務命名空間初始化

    Loft 是 Kiosk 和 DevSpace 維護者提供的商業產品,增加了以下功能:

    • 跨集群訪問用於授予對不同集群中空間的訪問權限
    • 睡眠模式在不活動期間縮小空間中的部署
    • 單點登錄使用 GitHub 等 OIDC 身份驗證提供程序

    可以通過軟多租戶解決三個主要用例。

    企業設置¶

    第一個是在企業設置中,其中"租戶"是半受信任的,因為他們是員工、承包商或否則被組織授權。每個租戶通常都會與部門或團隊等行政部門保持一致。

    在這種設置中,集群管理員通常負責創建命名空間和管理策略。他們還可能實施委派管理模型,其中某些個人被賦予命名空間的監督權,允許他們執行與策略無關的部署、服務、pod、作業等對象的 CRUD 操作。

    容器運行時提供的隔離可能在此設置中可以接受,或者可能需要通過其他控制措施進行增強,以實現更嚴格的 pod 安全。如果需要更嚴格的隔離,還可能需要限制不同命名空間中服務之間的通信。

    Kubernetes 即服務¶

    相比之下,軟多租戶可用於提供 Kubernetes 即服務 (KaaS) 的設置。在 KaaS 中,您的應用程式托管在共享集群中,以及一組控制器和 CRD,提供一組 PaaS 服務。租戶直接與 Kubernetes API 服務器交互,並被允許對非策略對象執行 CRUD 操作。這種環境中也有一定程度的自助服務,因為租戶可能被允許創建和管理自己的命名空間。在這種環境中,租戶被假定運行不受信任的代碼。

    為了隔離這種環境中的租戶,您可能需要實施嚴格的網路策略以及_pod 沙箱化_。沙箱化是指在像 Firecracker 或用戶空間內核中的微型 VM 內運行 pod 的容器。如今,您可以使用 EKS Fargate 創建沙箱化的 pod。

    軟件即服務 (SaaS)¶

    軟多租戶的最後一個用例是在軟件即服務 (SaaS) 環境中。在這種環境中,每個租戶都與在集群中運行的特定_實例_的應用程式相關聯。每個實例通常都有自己的數據,並使用通常獨立於 Kubernetes RBAC 的單獨的訪問控制。

    與其他用例不同,SaaS 設置中的租戶不會直接與 Kubernetes API 交互。相反,SaaS 應用程序負責與 Kubernetes API 交互,以創建支持每個租戶所需的必要對象。

    Kubernetes 構造¶

    在這些實例中,使用以下構造來隔離租戶:

    命名空間¶

    命名空間是實現軟多租戶的基礎。它們允許您將集群劃分為邏輯分區。配額、網路策略、服務帳戶和實現多租戶所需的其他對象都範圍限定在命名空間中。

    網路策略¶

    默認情況下,Kubernetes 集群中的所有 pod 都被允許相互通信。這種行為可以使用網路策略進行更改。

    網路策略使用標籤或 IP 地址範圍限制 pod 之間的通信。在需要嚴格的網路隔離的多租戶環境中,我們建議從拒絕 pod 之間的通信的默認規則開始,以及允許所有 pod 查詢 DNS 服務器以進行名稱解析的另一個規則。有了這些,您就可以開始添加更寬鬆的規則,允許在命名空間內進行通信。這可以根據需要進一步細化。

    Note

    Amazon VPC CNI 現在支持 Kubernetes 網路策略來創建可隔離敏感工作負載並保護它們免受未經授權訪問的策略,在 AWS 上運行 Kubernetes。這意味著您可以在 Amazon EKS 集群中使用網路策略 API 的所有功能。這種細粒度控制使您能夠實施最小權限原則,確保只有授權的 pod 才能相互通信。

    Attention

    網路策略是必要的但不足夠的。網路策略的執行需要一個策略引擎,如 Calico 或 Cilium。

    基於角色的訪問控制 (RBAC)¶

    角色和角色綁定是 Kubernetes 用於在 Kubernetes 中實施基於角色的訪問控制 (RBAC) 的對象。角色包含可對集群中的對象執行的操作列表。角色綁定指定適用角色的個人或群組。在企業和 KaaS 設置中,RBAC 可用於允許選定群組或個人管理對象。

    配額¶

    配額用於定義托管在集群中的工作負載的限制。使用配額,您可以指定 pod 可以消耗的最大 CPU 和內存量,或者您可以限制可在集群或命名空間中分配的資源數量。限制範圍允許您聲明每個限制的最小值、最大值和默認值。

    在共享集群中過度提交資源通常是有益的,因為它允許您最大化資源利用率。但是,對集群的無限訪問可能會導致資源耗竭,從而導致性能下降和應用程式可用性損失。如果 pod 的請求設置太低,實際資源利用率超過節點的容量,節點將開始經歷 CPU 或內存壓力。發生這種情況時,pod 可能會被重新啟動和/或從節點中驅逐。

    為了防止這種情況發生,您應該計劃在多租戶環境中對命名空間施加配額,以迫使租戶在將 pod 調度到集群上時指定請求和限制。它還將緩解潛在的拒絕服務攻擊,通過限制 pod 可以消耗的資源量。

    您還可以使用配額根據租戶的支出來分配集群的資源。這在 KaaS 場景中特別有用。

    Pod 優先級和搶佔¶

    Pod 優先級和搶佔在您希望相對於其他 Pod 提供更高重要性的 Pod 時很有用。例如,使用 pod 優先級,您可以配置來自客戶 A 的 pod 的優先級高於客戶 B。當可用容量不足時,調度程序將驅逐來自客戶 B 的較低優先級 pod 以容納來自客戶 A 的較高優先級 pod。這在 SaaS 環境中特別有用,在這種環境中,願意支付溢價的客戶會獲得更高的優先級。

    Attention

    Pod 優先級可能會對其他具有較低優先級的 Pod 產生不希望的影響。例如,儘管受害 pod 被優雅地終止,但不能保證 PodDisruptionBudget,這可能會破壞依賴 Pod 法定人數的較低優先級應用程式,請參見搶佔的局限性。

    緩解控制措施¶

    作為多租戶環境的管理員,您最關心的是防止攻擊者訪問底層主機。應考慮以下控制措施來減輕此風險:

    容器的沙箱化執行環境¶

    沙箱化是一種技術,通過該技術每個容器都在自己的隔離虛擬機中運行。執行 pod 沙箱化的技術包括 Firecracker 和 Weave 的 Firekube。

    有關使 Firecracker 成為 EKS 支持的運行時的工作的更多信息,請參見 https://threadreaderapp.com/thread/1238496944684597248.html。

    Open Policy Agent (OPA) 和 Gatekeeper¶

    Gatekeeper 是一個 Kubernetes 准入控制器,它使用 OPA 創建的策略進行強制執行。使用 OPA,您可以創建一個策略,該策略在單獨的實例上運行來自租戶的 pod,或者以高於其他租戶的優先級運行。可以在 GitHub 存儲庫中找到一組常見的 OPA 策略。

    還有一個實驗性的 OPA 插件for CoreDNS,允許您使用 OPA 來過濾/控制 CoreDNS 返回的記錄。

    Kyverno¶

    Kyverno 是一個本地 Kubernetes 策略引擎,可以使用策略作為 Kubernetes 資源來驗證、變異和生成配置。Kyverno 使用 Kustomize 樣式的覆蓋進行驗證,支持 JSON Patch 和戰略合併 patch 進行變異,並可以根據靈活的觸發器跨命名空間克隆資源。

    您可以使用 Kyverno 隔離命名空間、強制實施 pod 安全和其他最佳實踐,並生成默認配置,如網路策略。本項目的 GitHub 存儲庫中包含了一些示例。Kyverno 網站的策略庫中還包含了許多其他示例。

    將租戶工作負載隔離到特定節點¶

    將租戶工作負載限制在為相應租戶配置的節點上可用於增強軟多租戶模型中的隔離。使用這種方法,租戶特定的工作負載只在為相應租戶配置的節點上運行。為實現此隔離,使用本機 Kubernetes 屬性(節點親和性和污點與容忍)來定位特定節點進行 pod 調度,並防止來自其他租戶的 pod 被調度到租戶特定節點。

    第 1 部分 - 節點親和性¶

    Kubernetes 節點親和性用於根據節點標籤定位節點進行調度。使用節點親和性規則,pod 被吸引到與選擇器條件匹配的特定節點。在下面的 pod 規格中,應用了 requiredDuringSchedulingIgnoredDuringExecution 節點親和性。結果是 pod 將定位到標有以下鍵/值的節點: node-restriction.kubernetes.io/tenant: tenants-x。

    ...
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-restriction.kubernetes.io/tenant
            operator: In
            values:
            - tenants-x
...

    使用此節點親和性,標籤在調度期間是必需的,但在執行期間不是必需的;如果底層節點的標籤發生變化,pod 不會僅因為該標籤變化而被驅逐。但是,未來的調度可能會受到影響。

    Warning

    node-restriction.kubernetes.io/ 前綴有特殊含義。NodeRestriction 在 EKS 集群中啟用,阻止 kubelet 添加/刪除/更新具有此前綴的標籤。攻擊者無法使用 kubelet 的憑據更新節點對象或修改系統設置以將這些標籤傳遞給 kubelet,因為 kubelet 不允許修改這些標籤。如果此前綴用於所有 pod 到節點調度,它可以防止攻擊者可能想通過修改節點標籤來吸引不同的工作負載到節點的情況。

    Info

    我們也可以使用節點選擇器。但是,節點親和性更具表現力,允許在 pod 調度期間考慮更多條件。有關差異和更高級調度選擇的更多信息,請參見 CNCF 博客文章高級 Kubernetes pod 到節點調度。

    第 2 部分 - 污點和容忍¶

    吸引 pod 到節點只是這種三部分方法的第一部分。為了使這種方法有效,我們必須阻止未經授權的 pod 被調度到節點。為了阻止不需要或未經授權的 pod,Kubernetes 使用節點污點。污點用於在節點上放置條件,以防止 pod 被調度。下面的污點使用鍵值對 tenant: tenants-x。

    ...
    taints:
      - key: tenant
        value: tenants-x
        effect: NoSchedule
...

    給定上述節點污點,只有容忍污點的 pod 才被允許調度到節點上。為了允許授權的 pod 被調度到節點上,相應的 pod 規格必須包含對污點的容忍,如下所示。

    ...
  tolerations:
  - effect: NoSchedule
    key: tenant
    operator: Equal
    value: tenants-x
...

    具有上述容忍的 pod 將不會被阻止調度到節點上,至少不是因為該特定污點。污點也被 Kubernetes 用於在某些條件下暫時停止 pod 調度,例如節點資源壓力。使用節點親和性和污點與容忍,我們可以有效地吸引所需的 pod 到特定節點並阻止不需要的 pod。

    Attention

    某些 Kubernetes pod 需要在所有節點上運行。這些 pod 的示例包括由容器網絡接口 (CNI) 和 kube-proxy daemonsets啟動的 pod。因此,這些 pod 的規格包含非常寬鬆的容忍。應小心不要更改這些容忍。更改這些容忍可能會導致集群操作不正確。此外,策略管理工具,如 OPA/Gatekeeper 和 Kyverno可用於編寫驗證策略,以防止未經授權的 pod 使用這些寬鬆的容忍。

    第 3 部分 - 基於策略的節點選擇管理¶

    有幾種工具可用於幫助管理 pod 規格的節點親和性和容忍,包括在 CICD 管道中強制執行規則。但是,隔離的強制執行也應在 Kubernetes 集群級別完成。為此,可以使用策略管理工具來_變異_入站 Kubernetes API 服務器請求,基於請求有效負載應用上述節點親和性規則和容忍。

    例如,目標為_tenants-x_命名空間的 pod 可以被_加上_正確的節點親和性和容忍,以允許在_tenants-x_節點上進行調度。利用使用 Kubernetes Mutating Admission Webhook配置的策略管理工具,可以使用策略來變異入站 pod 規格。這些變異添加了所需的元素,以允許所需的調度。下面是一個添加節點親和性的 OPA/Gatekeeper 策略示例。

    apiVersion: mutations.gatekeeper.sh/v1alpha1
kind: Assign
metadata:
  name: mutator-add-nodeaffinity-pod
  annotations:
    aws-eks-best-practices/description: >-
      Adds Node affinity - https://kubernetes.io/docs/concepts/scheduling-eviction/assign-pod-node/#node-affinity
spec:
  applyTo:
  - groups: [""]
    kinds: ["Pod"]
    versions: ["v1"]
  match:
    namespaces: ["tenants-x"]
  location: "spec.affinity.nodeAffinity.requiredDuringSchedulingIgnoredDuringExecution.nodeSelectorTerms"
  parameters:
    assign:
      value: 
        - matchExpressions:
          - key: "tenant"
            operator: In
            values:
            - "tenants-x"

    上述策略應用於 Kubernetes API 服務器請求,以將 pod 應用於_tenants-x_命名空間。該策略添加了 requiredDuringSchedulingIgnoredDuringExecution 節點親和性規則,以便 pod 被吸引到具有 tenant: tenants-x 標籤的節點。

    使用相同的目標命名空間和組、種類和版本的匹配標準,第二個策略如下所示,將容忍添加到相同的 pod 規格中。

    apiVersion: mutations.gatekeeper.sh/v1alpha1
kind: Assign
metadata:
  name: mutator-add-toleration-pod
  annotations:
    aws-eks-best-practices/description: >-
      Adds toleration - https://kubernetes.io/docs/concepts/scheduling-eviction/taint-and-toleration/
spec:
  applyTo:
  - groups: [""]
    kinds: ["Pod"]
    versions: ["v1"]
  match:
    namespaces: ["tenants-x"]
  location: "spec.tolerations"
  parameters:
    assign:
      value: 
      - key: "tenant"
        operator: "Equal"
        value: "tenants-x"
        effect: "NoSchedule"

    上述策略特定於 pod;這是由於策略的location元素中變異元素的路徑。可以編寫其他策略來處理創建 pod 的資源,如 Deployment 和 Job 資源。列出的策略和其他示例可以在此指南的配套 GitHub 項目中看到。

    這兩個變異的結果是 pod 被吸引到所需的節點,同時也不會被特定節點污點所排斥。為了驗證這一點,我們可以看到兩個 kubectl 調用的輸出片段,一個用於獲取標有 tenant=tenants-x 的節點,另一個用於獲取 tenants-x 命名空間中的 pod。

    kubectl get nodes -l tenant=tenants-x
NAME                                        
ip-10-0-11-255...
ip-10-0-28-81...
ip-10-0-43-107...

kubectl -n tenants-x get pods -owide
NAME                                  READY   STATUS    RESTARTS   AGE   IP            NODE
tenant-test-deploy-58b895ff87-2q7xw   1/1     Running   0          13s   10.0.42.143   ip-10-0-43-107...
tenant-test-deploy-58b895ff87-9b6hg   1/1     Running   0          13s   10.0.18.145   ip-10-0-28-81...
tenant-test-deploy-58b895ff87-nxvw5   1/1     Running   0          13s   10.0.30.117   ip-10-0-28-81...
tenant-test-deploy-58b895ff87-vw796   1/1     Running   0          13s   10.0.3.113    ip-10-0-11-255...
tenant-test-pod                       1/1     Running   0          13s   10.0.35.83    ip-10-0-43-107...

    如我們從上述輸出中看到的,所有 pod 都被調度到標有 tenant=tenants-x 的節點上。簡單地說,pod 只會在所需的節點上運行,而其他 pod (沒有所需的親和性和容忍)將不會。租戶工作負載被有效隔離。

    下面是一個變異的 pod 規格示例。

    apiVersion: v1
kind: Pod
metadata:
  name: tenant-test-pod
  namespace: tenants-x
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: tenant
            operator: In
            values:
            - tenants-x
...
  tolerations:
  - effect: NoSchedule
    key: tenant
    operator: Equal
    value: tenants-x
...

    Attention

    使用變異和驗證准入 webhook 集成到 Kubernetes API 服務器請求流的策略管理工具被設計為在指定的時間框內(通常為 3 秒或更短)對 API 服務器的請求做出響應。如果 webhook 調用未能在配置的時間內返回響應,則可能會發生或可能不會發生入站 API 服務器請求的變異和/或驗證,這取決於准入 webhook 配置是設置為Fail Open 還是 Fail Close。

    在上面的示例中,我們使用了為 OPA/Gatekeeper 編寫的策略。但是,也有其他策略管理工具可以處理我們的節點選擇用例。例如,這個 Kyverno 策略可用於處理節點親和性變異。

    Tip

    如果操作正確,變異策略將對入站 API 服務器請求有效負載進行所需的更改。但是,還應該包括驗證策略來驗證所需的更改是否發生,然後才允許更改持久化。這在將這些策略用於租戶到節點隔離時尤其重要。定期檢查集群是否存在不需要的配置的審核策略也是一個好主意。

    硬多租戶¶

    通過為每個租戶配置單獨的集群來實現硬多租戶。雖然這提供了非常強大的租戶隔離,但也有一些缺點。

    首先,當您有很多租戶時,這種方法很快就會變得昂貴。您不僅需要為每個集群的控制平面成本付費,而且無法在集群之間共享計算資源。這最終會導致碎片化,一些集群資源利用率低,而其他集群資源利用率高。

    其次,您可能需要購買或構建特殊工具來管理所有這些集群。隨著時間的推移,管理數百或數千個集群可能變得過於繁瑣。

    最後,每個租戶創建一個集群相對於創建一個命名空間要慢。儘管如此,在高度管制的行業或需要強隔離的 SaaS 環境中,硬租戶方法可能是必要的。

    未來方向¶

    Kubernetes 社區已經認識到軟多租戶的當前缺陷以及硬多租戶的挑戰。多租戶特殊興趣小組 (SIG)正試圖通過幾個孵化項目解決這些缺陷,包括階層命名空間控制器 (HNC) 和虛擬集群。

    HNC 提案 (KEP) 描述了一種創建命名空間父子關係的方式,並沿承[策略]對象,同時允許租戶管理員創建子命名空間。

    虛擬集群提案描述了在集群內為每個租戶創建控制平面服務(包括 API 服務器、控制器管理器和調度器)的機制(也稱為"Kubernetes 上的 Kubernetes")。

    多租戶基準提案提供了使用命名空間進行隔離和分段的指南,以及一個命令行工具 kubectl-mtb 來驗證是否符合這些指南。
