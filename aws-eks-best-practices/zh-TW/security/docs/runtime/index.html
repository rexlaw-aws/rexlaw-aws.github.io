    運行時安全性¶

    運行時安全性為正在運行的容器提供主動保護。其目的是檢測和/或阻止容器內部的惡意活動。這可以通過 Linux 內核或與 Kubernetes 集成的內核擴展中的多種機制來實現,例如 Linux 功能、安全計算 (seccomp)、AppArmor 或 SELinux。還有一些選擇,如 Amazon GuardDuty 和第三方工具,可以幫助建立基線並檢測異常活動,而無需過多手動配置 Linux 內核機制。

    Attention

    Kubernetes 目前不提供加載 seccomp、AppArmor 或 SELinux 配置文件到節點的本地機制。它們必須手動加載或在引導節點時安裝。這必須在您的 Pod 中引用它們之前完成,因為調度器不知道哪些節點有配置文件。請參見下文,了解 Security Profiles Operator 等工具如何幫助自動將配置文件部署到節點。

    安全上下文和內置的 Kubernetes 控制¶

    許多 Linux 運行時安全機制都與 Kubernetes 緊密集成,可以通過 Kubernetes 安全上下文進行配置。其中一個選項是 privileged 標誌,默認為 false,如果啟用,它實際上等同於主機上的 root。在生產工作負載中幾乎總是不適合啟用特權模式,但還有許多其他控制可以根據需要為容器提供更細粒度的權限。

    Linux 功能¶

    Linux 功能允許您向 Pod 或容器授予某些功能,而無需提供 root 用戶的所有功能。例如 CAP_NET_ADMIN 允許配置網絡接口或防火牆,CAP_SYS_TIME 允許操作系統時鐘。

    Seccomp¶

    使用安全計算 (seccomp),您可以阻止容器化應用程序對底層主機操作系統內核進行某些系統調用。雖然 Linux 操作系統有幾百個系統調用,但大部分對於運行容器來說並不必要。通過限制容器可以進行的系統調用,您可以有效地減少應用程序的攻擊面。

    Seccomp 通過攔截系統調用並只允許通過白名單的調用來工作。Docker 有一個默認 seccomp 配置文件,適用於大多數通用工作負載,其他容器運行時如 containerd 也提供類似的默認值。您可以通過在 Pod 規範的 securityContext 部分添加以下內容來配置容器或 Pod 使用容器運行時的默認 seccomp 配置文件:

    securityContext:
  seccompProfile:
    type: RuntimeDefault

    從 1.22 版本開始(在 alpha 版本中,在 1.27 版本中穩定),上述 RuntimeDefault 可以使用 單個 kubelet 標誌 --seccomp-default 在節點上為所有 Pod 使用。然後,securityContext 中指定的配置文件只需要用於其他配置文件。

    您也可以為需要更多權限的情況創建自己的配置文件。這手動完成起來非常繁瑣,但有工具如 Inspektor Gadget(也在網絡安全部分推薦用於生成網絡策略)和 Security Profiles Operator支持使用 eBPF 或日誌來記錄基線權限要求作為 seccomp 配置文件。Security Profiles Operator 還可以自動將記錄的配置文件部署到節點,供 Pod 和容器使用。

    AppArmor 和 SELinux¶

    AppArmor 和 SELinux 被稱為強制訪問控制或 MAC 系統。它們在概念上與 seccomp 相似,但 API 和功能不同,允許對特定文件系統路徑或網絡端口進行訪問控制。對這些工具的支持取決於 Linux 發行版,Debian/Ubuntu 支持 AppArmor,RHEL/CentOS/Bottlerocket/Amazon Linux 2023 支持 SELinux。另請參見基礎設施安全部分,了解有關 SELinux 的更多討論。

    AppArmor 和 SELinux 都與 Kubernetes 集成,但截至 Kubernetes 1.28,AppArmor 配置文件必須通過註釋指定,而 SELinux 標籤可以通過安全上下文上的 SELinuxOptions字段直接設置。

    與 seccomp 配置文件一樣,上述提到的 Security Profiles Operator 可以協助將配置文件部署到集群中的節點。(未來,該項目還計劃為 AppArmor 和 SELinux 生成配置文件,就像它為 seccomp 所做的那樣。)

    建議¶

    使用 Amazon GuardDuty 進行運行時監控和檢測 EKS 環境中的威脅¶

    如果您目前沒有解決方案持續監控 EKS 運行時並分析 EKS 審核日誌,以及掃描惡意軟件和其他可疑活動,Amazon 強烈建議 AWS 客戶使用 Amazon GuardDuty 作為一種簡單、快速、安全、可擴展和經濟高效的一鍵方式來保護其 AWS 環境。Amazon GuardDuty 是一項安全監控服務,它分析和處理基礎數據源,如 AWS CloudTrail 管理事件、AWS CloudTrail 事件日誌、VPC 流量日誌(來自 Amazon EC2 實例)、Kubernetes 審核日誌和 DNS 日誌。它還包括 EKS 運行時監控。它使用不斷更新的威脅情報源,如惡意 IP 地址和域名列表,以及機器學習來識別您 AWS 環境中意外的、可能未經授權的和惡意活動。這可能包括權限升級、使用暴露的憑據、與惡意 IP 地址或域名通信、Amazon EC2 實例和 EKS 容器工作負載上存在惡意軟件,或發現可疑的 API 活動。GuardDuty 通過生成安全發現來告知您 AWS 環境的狀態,您可以在 GuardDuty 控制台或通過 Amazon EventBridge 查看這些發現。GuardDuty 還支持將您的發現導出到 Amazon Simple Storage Service (S3) 存儲桶,並與其他服務(如 AWS Security Hub 和 Detective)集成。

    觀看這個 AWS Online Tech Talk "使用 Amazon GuardDuty 增強對 Amazon EKS 的威脅檢測 - AWS Online Tech Talks",了解如何在幾分鐘內啟用這些額外的 EKS 安全功能。

    可選:使用第三方解決方案進行運行時監控¶

    如果您不熟悉 Linux 安全性,創建和管理 seccomp 和 Apparmor 配置文件可能很困難。如果您沒有時間成為專家,請考慮使用第三方商業解決方案。許多解決方案已經超越了 Apparmor 和 seccomp 等靜態配置文件,開始使用機器學習來阻止或警告可疑活動。下面的工具部分列出了一些這樣的解決方案。您可以在 AWS Marketplace for Containers上找到更多選擇。

    在編寫 seccomp 策略之前,請考慮添加/刪除 Linux 功能¶

    功能涉及內核函數中可通過系統調用訪問的各種檢查。如果檢查失敗,系統調用通常會返回錯誤。檢查可以在特定系統調用的開頭進行,也可以在可通過多個不同系統調用訪問的內核區域(如寫入特定特權文件)更深處進行。另一方面,seccomp 是一個系統調用過濾器,在系統調用運行之前應用。進程可以設置一個過濾器,撤銷其運行某些系統調用或特定參數的權利。

    在使用 seccomp 之前,請考慮添加/刪除 Linux 功能是否能給您所需的控制。請參閱為容器設置功能了解更多信息。

    看看您是否可以通過使用 Pod 安全策略 (PSP) 來實現您的目標¶

    Pod 安全策略提供了許多不同的方式來改善您的安全態勢,而無需引入過度的複雜性。在嘗試構建 seccomp 和 Apparmor 配置文件之前,請先探索 PSP 中可用的選項。

    Warning

    從 Kubernetes 1.25 版本開始,PSP 已被刪除,並被 Pod 安全准入控制器取代。第三方替代方案包括 OPA/Gatekeeper 和 Kyverno。可以從 GitHub 上的 Gatekeeper 庫中獲取一組 Gatekeeper 約束和約束模板,用於實現通常在 PSP 中找到的策略。並且可以在 Kyverno 策略庫中找到許多 PSP 的替代方案,包括完整的 Pod 安全標準集合。
