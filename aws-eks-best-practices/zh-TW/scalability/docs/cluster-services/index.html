    叢集服務¶

    叢集服務在 EKS 叢集內運行,但它們不是使用者工作負載。如果您有一台 Linux 伺服器,通常需要運行像 NTP、syslog 和容器運行時等服務來支援您的工作負載。叢集服務類似,支援可幫助您自動化和操作叢集的服務。在 Kubernetes 中,它們通常在 kube-system 命名空間中運行,有些作為 DaemonSets 運行。

    叢集服務預計會有很高的正常運行時間,在停電和故障排除期間通常很關鍵。如果核心叢集服務不可用,您可能會失去有助於恢復或防止停電的數據(例如高磁碟利用率)。它們應該在專用的計算實例上運行,如單獨的節點組或 AWS Fargate。這將確保叢集服務不會受到共享實例上可能擴展或使用更多資源的工作負載的影響。

    擴展 CoreDNS¶

    擴展 CoreDNS 有兩種主要機制。減少對 CoreDNS 服務的調用次數和增加副本數量。

    通過降低 ndots 來減少外部查詢¶

    ndots 設置指定域名中被視為足夠避免 DNS 查詢的句點(又名"點")數。如果您的應用程序有一個 ndots 設置為 5(預設值),並且請求來自外部域(如 api.example.com,2 個點)的資源,則 CoreDNS 將被查詢以尋找在 /etc/resolv.conf 中定義的每個搜索域的更具體的域。默認情況下,在進行外部請求之前,將搜索以下域。

    api.example.<namespace>.svc.cluster.local
api.example.svc.cluster.local
api.example.cluster.local
api.example.<region>.compute.internal

    namespace 和 region 值將被替換為您的工作負載命名空間和計算區域。根據您的叢集設置,您可能還有其他搜索域。

    您可以通過降低工作負載的 ndots 選項或完全限定您的域請求(包括尾部 .)來減少對 CoreDNS 的請求次數(例如 api.example.com.)。如果您的工作負載通過 DNS 連接到外部服務,我們建議將 ndots 設置為 2,以便工作負載不會在叢集內部進行不必要的叢集 DNS 查詢。如果工作負載不需要訪問叢集內的服務,您可以設置不同的 DNS 服務器和搜索域。

    spec:
  dnsPolicy: "None"
  dnsConfig:
    options:
      - name: ndots
        value: "2"
      - name: edns0

    如果將 ndots 降低到太低的值,或者您連接的域不包含足夠的特定性(包括尾部 .)，則可能會導致 DNS 查找失敗。請確保您測試此設置如何影響您的工作負載。

    水平擴展 CoreDNS¶

    可以通過添加更多副本到部署來擴展 CoreDNS 實例。建議您使用 NodeLocal DNS 或 cluster proportional autoscaler 來擴展 CoreDNS。

    NodeLocal DNS 將需要在每個節點上運行一個實例 - 作為 DaemonSet - 這需要在叢集中使用更多計算資源,但它將避免 DNS 請求失敗並降低叢集中 DNS 查詢的響應時間。集群比例自動縮放器將根據節點數量或核心數量來擴展 CoreDNS。這與請求查詢的直接相關性不大,但可能取決於您的工作負載和叢集大小而有所不同。默認的比例縮放是為每 256 個核心或 16 個節點添加一個額外的副本 - 以先發生的為準。

    垂直擴展 Kubernetes Metrics Server¶

    Kubernetes Metrics Server 支持水平和垂直擴展。通過水平擴展 Metrics Server,它將具有高可用性,但它不會水平擴展以處理更多叢集指標。您需要根據他們的建議垂直擴展 Metrics Server,因為隨著節點和收集的指標的增加,Metrics Server 需要更多的資源。

    Metrics Server 會保留它收集、聚合和提供的數據。隨著叢集的增長,Metrics Server 存儲的數據量也會增加。在大型叢集中,Metrics Server 將需要比默認安裝中指定的內存和 CPU 預留更多的計算資源。您可以使用 Vertical Pod Autoscaler(VPA) 或 Addon Resizer 來擴展 Metrics Server。Addon Resizer 會根據工作節點的比例進行垂直擴展,而 VPA 則根據 CPU 和內存使用情況進行擴展。

    CoreDNS 延遲退出持續時間¶

    Pods 使用 kube-dns 服務進行名稱解析。Kubernetes 使用目的地 NAT (DNAT) 將 kube-dns 流量從節點重定向到 CoreDNS 後端 pods。當您擴展 CoreDNS 部署時, kube-proxy 會更新節點上的 iptables 規則和鏈,以將 DNS 流量重定向到 CoreDNS pods。在擴展時傳播新的端點,以及在縮減 CoreDNS 時刪除規則,這可能需要 1 到 10 秒,具體取決於叢集的大小。

    這種傳播延遲可能會導致在 CoreDNS pod 終止但節點的 iptables 規則尚未更新的情況下出現 DNS 查找失敗。在這種情況下,節點可能會繼續將 DNS 查詢發送到已終止的 CoreDNS Pod。

    您可以通過在 CoreDNS pods 中設置延遲退出持續時間來減少 DNS 查找失敗。在延遲退出模式下,CoreDNS 將繼續響應正在進行的請求。設置延遲退出持續時間將延遲 CoreDNS 關閉過程,從而為節點更新其 iptables 規則和鏈提供所需的時間。

    我們建議將 CoreDNS 延遲退出持續時間設置為 30 秒。

    CoreDNS 就緒探針¶

    我們建議使用 /ready 而不是 /health 作為 CoreDNS 的就緒探針。

    為了與前面建議的將延遲退出持續時間設置為 30 秒相一致,為節點的 iptables 規則更新提供充足的時間,在 CoreDNS 啟動時使用 /ready 而不是 /health 作為就緒探針,可確保 CoreDNS pod 已完全準備就緒,可以立即響應 DNS 請求。

    readinessProbe:
  httpGet:
    path: /ready
    port: 8181
    scheme: HTTP

    有關 CoreDNS Ready 插件的更多信息,請參考 https://coredns.io/plugins/ready/

    日誌和監控代理¶

    日誌和監控代理可能會給您的叢集控制平面增加大量負載,因為代理需要查詢 API 伺服器來豐富日誌和指標與工作負載元數據。節點上的代理只能訪問本地節點資源來查看容器和進程名稱等內容。查詢 API 伺服器可以添加更多詳細信息,如 Kubernetes 部署名稱和標籤。這對於故障排除非常有用,但對於擴展卻是不利的。

    由於日誌和監控有很多不同的選擇,我們無法為每個提供商顯示示例。對於 fluentbit,我們建議啟用 Use_Kubelet 從本地 kubelet 獲取元數據,而不是 Kubernetes API 伺服器,並設置 Kube_Meta_Cache_TTL 為一個可以在數據可以緩存時減少重複調用的數字(例如 60)。

    擴展監控和日誌有兩個一般選項:

    • 禁用集成
    • 採樣和過濾

    禁用集成通常不是一個選項,因為您會失去日誌元數據。這消除了 API 擴展問題,但在需要所需元數據時會引入其他問題。

    採樣和過濾減少了收集的指標和日誌數量。這將降低對 Kubernetes API 的請求量,並減少收集的指標和日誌的存儲量。減少存儲成本將降低整個系統的成本。

    配置採樣的能力取決於代理軟件,並且可以在不同的攝取點實現。重要的是要盡可能靠近代理添加採樣,因為那可能是 API 伺服器調用發生的地方。請聯繫您的提供商以了解更多關於採樣支持的信息。

    如果您正在使用 CloudWatch 和 CloudWatch Logs,您可以使用文檔中描述的模式添加代理過濾。

    為了避免丟失日誌和指標,您應該將數據發送到一個可以在接收端點發生停電時緩存數據的系統。使用 fluentbit,您可以使用 Amazon Kinesis Data Firehose 暫時保留數據,這可以減少過載最終數據存儲位置的機會。
