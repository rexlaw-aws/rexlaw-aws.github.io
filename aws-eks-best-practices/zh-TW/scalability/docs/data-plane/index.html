    Kubernetes 資料平面¶

    Kubernetes 資料平面包括 EC2 實例、負載平衡器、存儲和 Kubernetes 控制平面使用的其他 API。為了組織目的,我們將叢集服務分組到一個單獨的頁面,而負載平衡器縮放可以在工作負載部分中找到。本節將重點關注擴展計算資源。

    選擇 EC2 實例類型可能是客戶面臨的最困難決定之一,因為在具有多個工作負載的叢集中沒有一刀切的解決方案。以下是一些建議,可以幫助您避免擴展計算時的常見陷阱。

    自動節點自動縮放¶

    我們建議您使用節點自動縮放,它可以減少工作量並與 Kubernetes 深度集成。Managed Node Groups 和 Karpenter 是大規模叢集的推薦選擇。

    Managed Node Groups 將為您提供 Amazon EC2 Auto Scaling 群組的靈活性,並增加了托管升級和配置的好處。它可以使用 Kubernetes Cluster Autoscaler 進行縮放,是具有各種計算需求的叢集的常見選擇。

    Karpenter 是一個由 AWS 創建的開源、工作負載原生節點自動縮放器。它根據工作負載對資源(例如 GPU)和污點和容忍度(例如區域分散)的需求來縮放叢集中的節點,而無需管理節點組。節點直接從 EC2 創建,避免了默認節點組配額 - 每組 450 個節點 - 並提供更大的實例選擇靈活性和更少的操作開銷。我們建議客戶盡可能使用 Karpenter。

    使用多種不同的 EC2 實例類型¶

    每個 AWS 區域都有有限數量的可用實例類型。如果您創建一個只使用一種實例類型的叢集,並將節點數量擴展到超過該區域容量,您將收到沒有可用實例的錯誤。為了避免這個問題,您不應該任意限制可以在叢集中使用的實例類型。

    Karpenter 將默認使用一組廣泛的兼容實例類型,並將根據待處理工作負載的要求、可用性和成本在配置時選擇一個實例。您可以通過 karpenter.k8s.aws/instance-category 密鑰在 NodePools 中擴展使用的實例類型列表。

    Kubernetes Cluster Autoscaler 要求節點組具有類似的大小,以便可以一致地縮放。您應該根據 CPU 和內存大小創建多個組,並獨立縮放它們。使用 ec2-instance-selector 來識別適合您節點組的類似大小的實例。

    ec2-instance-selector --service eks --vcpus-min 8 --memory-min 16
a1.2xlarge
a1.4xlarge
a1.metal
c4.4xlarge
c4.8xlarge
c5.12xlarge
c5.18xlarge
c5.24xlarge
c5.2xlarge
c5.4xlarge
c5.9xlarge
c5.metal

    更喜歡較大的節點以減少 API 服務器負載¶

    在決定使用哪些實例類型時,較少的大型節點將對 Kubernetes 控制平面施加較小的負載,因為將運行較少的 kubelet 和 DaemonSet。但是,大型節點可能無法像較小的節點那樣充分利用。節點大小應該根據您的工作負載可用性和縮放要求進行評估。

    一個由三個 u-24tb1.metal 實例(24 TB 內存和 448 個核心)組成的叢集有 3 個 kubelet,默認情況下每個節點最多可以有 110 個 pod。如果您的 pod 每個使用 4 個核心,那麼這可能是預期的(4 個核心 x 110 = 440 個核心/節點)。在一個 3 節點叢集中,您處理實例事故的能力將很低,因為 1 個實例故障可能會影響 1/3 的叢集。您應該在工作負載中指定節點要求和 pod 分散,以便 Kubernetes 調度程序可以正確放置工作負載。

    工作負載應該定義它們需要的資源和所需的可用性,通過污點、容忍度和 PodTopologySpread。它們應該首選可以完全利用並滿足可用性目標的最大節點,以減少控制平面負載、降低運營成本和降低成本。

    Kubernetes 調度程序將自動嘗試在可用資源的情況下將工作負載分散到可用性區域和主機上。如果沒有可用容量,Kubernetes Cluster Autoscaler 將嘗試在每個可用性區域均勻地添加節點。Karpenter 將嘗試盡快、盡可能便宜地添加節點,除非工作負載指定了其他要求。

    要強制工作負載使用調度程序分散,並在可用性區域中創建新節點,您應該使用 topologySpreadConstraints:

    spec:
  topologySpreadConstraints:
    - maxSkew: 3
      topologyKey: "topology.kubernetes.io/zone"
      whenUnsatisfiable: ScheduleAnyway
      labelSelector:
        matchLabels:
          dev: my-deployment
    - maxSkew: 2
      topologyKey: "kubernetes.io/hostname"
      whenUnsatisfiable: ScheduleAnyway
      labelSelector:
        matchLabels:
          dev: my-deployment

    使用類似大小的節點以獲得一致的工作負載性能¶

    工作負載應該定義它們需要在哪種大小的節點上運行,以實現一致的性能和可預測的縮放。一個請求 500m CPU 的工作負載在 4 個核心的實例上和 16 個核心的實例上的性能會有所不同。避免使用可突發 CPU 的實例類型,如 T 系列實例。

    為了確保您的工作負載獲得一致的性能,工作負載可以使用 Karpenter 支持的標籤來定位特定的實例大小。

    kind: deployment
...
spec:
  template:
    spec:
    containers:
    nodeSelector:
      karpenter.k8s.aws/instance-size: 8xlarge

    在使用 Kubernetes Cluster Autoscaler 的叢集中調度的工作負載應該根據標籤匹配來匹配節點組。

    spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: eks.amazonaws.com/nodegroup
            operator: In
            values:
            - 8-core-node-group    # match your node group name

    有效利用計算資源¶

    計算資源包括 EC2 實例和可用性區域。有效利用計算資源將提高您的可擴展性、可用性、性能,並降低總成本。在具有多個應用程序的自動縮放環境中,有效利用資源是非常困難預測的。Karpenter 就是為了根據工作負載需求即時配置實例,以最大化利用率和靈活性而創建的。

    Karpenter 允許工作負載聲明它需要什麼類型的計算資源,而無需先創建節點組或為特定節點配置標籤污點。請參閱 Karpenter 最佳實踐以了解更多信息。考慮在您的 Karpenter 配置器中啟用合併,以替換利用率低的節點。

    自動化 Amazon 機器映像(AMI)更新¶

    保持工作節點組件最新將確保您擁有最新的安全修復程序和與 Kubernetes API 兼容的功能。更新 kubelet 是 Kubernetes 功能最重要的組件,但自動化操作系統、內核和本地安裝的應用程序修補程序將在您擴展時減少維護工作。

    建議您使用最新的 Amazon EKS 優化 Amazon Linux 2 或 Amazon EKS 優化 Bottlerocket AMI 作為您的節點映像。Karpenter 將自動使用最新可用的 AMI來配置叢集中的新節點。Managed Node Groups 將在節點組更新期間更新 AMI,但不會在節點配置時更新 AMI ID。

    對於 Managed Node Groups,您需要在新的 AMI ID 可用時更新自動縮放組(ASG)啟動模板。次要版本的 AMI(例如 1.23.5 到 1.24.3)將作為節點組升級在 EKS 控制台和 API 中提供。修補程序版本(例如 1.23.5 到 1.23.6)將不會作為節點組的升級呈現。如果您想讓節點組保持最新的 AMI 修補程序版本,您需要創建新的啟動模板版本,並讓節點組使用新的 AMI 版本替換實例。

    您可以從此頁面或使用 AWS CLI 找到最新可用的 AMI。

    aws ssm get-parameter \
  --name /aws/service/eks/optimized-ami/1.24/amazon-linux-2/recommended/image_id \
  --query "Parameter.Value" \
  --output text

    為容器使用多個 EBS 卷¶

    EBS 卷的輸入/輸出(I/O)配額基於卷類型(例如 gp3)和磁盤大小。如果您的應用程序共享主機的單個 EBS 根卷,這可能會耗盡整個主機的磁盤配額,導致其他應用程序等待可用容量。應用程序會在將文件寫入其覆蓋分區、從主機掛載本地卷以及根據使用的日誌代理將日誌寫入標準輸出(STDOUT)時寫入磁盤。

    為了避免磁盤 I/O 耗盡,您應該將第二個卷掛載到容器狀態文件夾(例如 /run/containerd),為工作負載存儲使用單獨的 EBS 卷,並禁用不必要的本地日誌記錄。

    要使用 eksctl 將第二個卷掛載到您的 EC2 實例,您可以使用具有以下配置的節點組:

    managedNodeGroups:
  - name: al2-workers
    amiFamily: AmazonLinux2
    desiredCapacity: 2
    volumeSize: 80
    additionalVolumes:
      - volumeName: '/dev/sdz'
        volumeSize: 100
    preBootstrapCommands:
    - |
      "systemctl stop containerd"
      "mkfs -t ext4 /dev/nvme1n1"
      "rm -rf /var/lib/containerd/*"
      "mount /dev/nvme1n1 /var/lib/containerd/"
      "systemctl start containerd"

    如果您使用 terraform 來配置節點組,請參見 EKS Blueprints for terraform 中的示例。如果您使用 Karpenter 來配置節點,您可以使用 blockDeviceMappings 和節點用戶數據來添加其他卷。

    要直接將 EBS 卷掛載到您的 pod,您應該使用 AWS EBS CSI 驅動程序並使用存儲類消耗卷。

    ---
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-sc
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: ebs-claim
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ebs-sc
  resources:
    requests:
      storage: 4Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: public.ecr.aws/docker/library/nginx
    volumeMounts:
    - name: persistent-storage
      mountPath: /data
  volumes:
  - name: persistent-storage
    persistentVolumeClaim:
      claimName: ebs-claim

    如果工作負載使用 EBS 卷,請避免具有低 EBS 附加限制的實例¶

    EBS 是工作負載持久存儲的最簡單方式之一,但它也有可擴展性限制。每種實例類型都有最大數量的 EBS 卷可以附加。工作負載需要聲明它們應該在哪些實例類型上運行,並使用 Kubernetes 污點限制單個實例上的副本數量。

    禁用不必要的磁盤日誌記錄¶

    通過不在生產環境中以調試日誌記錄運行您的應用程序,並禁用經常讀寫磁盤的日誌記錄,可以避免不必要的本地日誌記錄。Journald 是本地日誌服務,它將日誌緩衝區保存在內存中,並定期刷新到磁盤。Journald 優於每行立即寫入磁盤的 syslog。禁用 syslog 還可以降低所需的總存儲量,並避免需要複雜的日誌輪換規則。要禁用 syslog,您可以在雲初始化配置中添加以下片段:

    runcmd:
  - [ systemctl, disable, --now, syslog.service ]

    當操作系統更新速度是必要時,就就地修補實例¶

    注意

    只有在需要時才應該就地修補實例。Amazon 建議將基礎設施視為不可變的,並以與應用程序相同的方式,通過較低環境對更新進行徹底測試。本節適用於這種情況無法實現的情況。

    在不中斷容器化工作負載的情況下,在現有 Linux 主機上安裝軟件包只需要幾秒鐘。可以安裝和驗證軟件包,而無需隔離、排空或替換實例。

    要替換實例,您首先需要創建、驗證和分發新的 AMI。實例需要有一個替換品創建,而舊實例需要被隔離和排空。然後需要在新實例上創建工作負載,驗證,並重複所有需要修補的實例。安全地替換實例而不中斷工作負載需要數小時、數天或數週。

    Amazon 建議使用從自動化聲明式系統構建、測試和提升的不可變基礎設施,但如果您需要快速修補系統,則需要就地修補系統並在新的 AMI 可用時替換它們。由於修補和替換系統之間存在巨大的時間差異,我們建議使用 AWS Systems Manager Patch Manager 來自動化修補節點。

    修補節點將允許您快速推出安全更新,並在 AMI 更新後定期替換實例。如果您使用具有只讀根文件系統的操作系統,如 Flatcar Container Linux 或 Bottlerocket OS,我們建議使用適用於這些操作系統的更新操作員。Flatcar Linux 更新操作員和 Bottlerocket 更新操作員將自動重新啟動實例,以保持節點最新。
