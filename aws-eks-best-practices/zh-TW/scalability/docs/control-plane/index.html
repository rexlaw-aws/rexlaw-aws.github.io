    Kubernetes 控制平面¶

    Kubernetes 控制平面由 Kubernetes API 伺服器、Kubernetes 控制器管理器、排程器和其他 Kubernetes 所需的元件組成。這些元件的可擴展性限制因集群中運行的內容而有所不同,但對擴展性影響最大的領域包括 Kubernetes 版本、利用率和個別節點的擴展。

    使用 EKS 1.24 或更高版本¶

    EKS 1.24 引入了一些變更,並將容器運行時切換到 containerd 而不是 Docker。Containerd 通過限制容器運行時功能以更好地符合 Kubernetes 的需求,從而幫助集群擴展並提高個別節點的性能。Containerd 在所有受支持的 EKS 版本中都可用,如果您想在 1.24 之前的版本中切換到 containerd,請使用 --container-runtime 引導標誌。

    限制工作負載和節點突發¶

    注意

    為了避免達到控制平面的 API 限制,您應該限制集群大小增加時的擴展高峰,例如一次增加 10% 以內的節點數量(例如從 1000 個節點增加到 1100 個節點,或從 4000 個 Pod 增加到 4500 個 Pod)。

    EKS 控制平面將隨著集群的增長而自動擴展,但擴展速度有限制。當您首次創建 EKS 集群時,控制平面將無法立即擴展到數百個節點或數千個 Pod。要了解 EKS 如何進行擴展改進,請參閱此博客文章。

    擴展大型應用程序需要基礎設施適應以完全準備就緒(例如預熱負載均衡器)。為了控制擴展速度,請確保您是根據應用程序的正確指標進行擴展。CPU 和內存擴展可能無法準確預測您的應用程序限制,在 Kubernetes Horizontal Pod Autoscaler (HPA) 中使用自定義指標可能是更好的擴展選擇。

    要使用自定義指標,請參見 Kubernetes 文檔中的示例。如果您有更高級的擴展需求或需要根據外部源(例如 AWS SQS 隊列)進行擴展,請使用 KEDA 進行基於事件的工作負載擴展。

    安全地縮減節點和 Pod¶

    替換長期運行的實例¶

    定期替換節點可以通過避免配置漂移和只在長時間運行後才出現的問題(例如緩慢的內存洩漏)來保持集群健康。自動替換將為您提供良好的流程和做法來進行節點升級和安全修補。如果集群中的每個節點都定期被替換,則維護單獨的持續維護過程就需要更少的工作。

    使用 Karpenter 的生存時間 (TTL) 設置在它們運行了指定時間後自動替換實例。自管理的節點組可以使用 max-instance-lifetime 設置自動循環節點。托管節點組目前沒有此功能,但您可以在 GitHub 上跟踪請求此處。

    刪除利用不足的節點¶

    當節點沒有正在運行的工作負載時,您可以使用 Kubernetes 集群自動縮放器中的縮減閾值和 --scale-down-utilization-threshold 或在 Karpenter 中使用 ttlSecondsAfterEmpty 配置器設置來刪除節點。

    使用 Pod 中斷預算和安全節點關閉¶

    從 Kubernetes 集群中刪除 Pod 和節點需要控制器對多個資源(例如 EndpointSlices)進行更新。頻繁或過快地執行此操作可能會導致 API 伺服器限流和應用程序中斷,因為更改會傳播到控制器。Pod 中斷預算是一種最佳做法,可以在節點被刪除或重新調度到集群中時減慢變更,以保護工作負載的可用性。

    在運行 Kubectl 時使用客戶端緩存¶

    不當地使用 kubectl 命令可能會增加 Kubernetes API 伺服器的負載。您應該避免運行重複使用 kubectl 的腳本或自動化(例如在 for 循環中)或運行沒有本地緩存的命令。

    kubectl 有一個客戶端緩存,可以緩存來自集群的發現信息,以減少所需的 API 調用數量。緩存默認啟用,每 10 分鐘刷新一次。

    如果您從容器中運行 kubectl 或沒有客戶端緩存,您可能會遇到 API 限流問題。建議保留您的集群緩存,通過掛載 --cache-dir 來避免進行不必要的 API 調用。

    禁用 kubectl 壓縮¶

    在您的 kubeconfig 文件中禁用 kubectl 壓縮可以減少 API 和客戶端 CPU 使用。默認情況下,伺服器會壓縮發送到客戶端的數據,以優化網絡帶寬。這會增加每個請求的客戶端和伺服器的 CPU 負載,禁用壓縮可以減少開銷和延遲,前提是您有足夠的帶寬。要禁用壓縮,您可以使用 --disable-compression=true 標誌或在您的 kubeconfig 文件中設置 disable-compression: true。

    apiVersion: v1
clusters:
- cluster:
    server: serverURL
    disable-compression: true
  name: cluster

    API 優先級和公平性¶

    概述¶

    為了在請求增加期間保護自己免受過載,API 伺服器會限制它在給定時間內可以有多少正在進行的請求。一旦超過此限制,API 伺服器將開始拒絕請求,並向客戶端返回 429 HTTP 響應代碼"請求過多"。伺服器丟棄請求並讓客戶端稍後重試要優於沒有伺服器端限制的請求數量,這可能會導致控制平面性能下降或不可用。

    Kubernetes 用來配置這些正在進行的請求如何在不同請求類型之間分配的機制稱為 API 優先級和公平性。API 伺服器配置它可以接受的總請求數,方法是將 --max-requests-inflight 和 --max-mutating-requests-inflight 標誌指定的值相加。EKS 使用 400 和 200 請求的默認值,允許在給定時間內分派總共 600 個請求。但是,隨著它根據增加的利用率和工作負載變化來擴展控制平面,它相應地將正在進行的請求配額增加到 2000(可能會變更)。APF 指定了這些正在進行的請求配額如何在不同請求類型之間進一步細分。請注意,EKS 控制平面是高度可用的,每個集群至少有 2 個註冊的 API 伺服器。這意味著您的集群可以處理的總正在進行的請求數是每個 kube-apiserver 設置的正在進行的請求配額的兩倍(或者如果水平擴展得更多,則更高)。這相當於最大 EKS 集群每秒處理數千個請求。

    Kubernetes 對象的兩種類型,稱為 PriorityLevelConfigurations 和 FlowSchemas,配置了如何將總請求數分配給不同的請求類型。這些對象由 API 伺服器自動維護,EKS 使用給定 Kubernetes 次版本的默認配置。PriorityLevelConfigurations 表示總允許請求數的一部分。例如,workload-high PriorityLevelConfiguration 被分配了 98 個,總共 600 個請求。分配給所有 PriorityLevelConfigurations 的請求數之和將等於 600(或略高於 600,因為 API 伺服器將向上舍入,如果給定級別被授予請求的一部分)。要檢查集群中的 PriorityLevelConfigurations 以及分配給每個 PriorityLevelConfigurations 的請求數,可以運行以下命令。這些是 EKS 1.24 上的默認值:

    $ kubectl get --raw /metrics | grep apiserver_flowcontrol_request_concurrency_limit
apiserver_flowcontrol_request_concurrency_limit{priority_level="catch-all"} 13
apiserver_flowcontrol_request_concurrency_limit{priority_level="global-default"} 49
apiserver_flowcontrol_request_concurrency_limit{priority_level="leader-election"} 25
apiserver_flowcontrol_request_concurrency_limit{priority_level="node-high"} 98
apiserver_flowcontrol_request_concurrency_limit{priority_level="system"} 74
apiserver_flowcontrol_request_concurrency_limit{priority_level="workload-high"} 98
apiserver_flowcontrol_request_concurrency_limit{priority_level="workload-low"} 245

    第二種對象是 FlowSchemas。具有給定一組屬性的 API 伺服器請求將被歸類到同一個 FlowSchema 下。這些屬性包括經過身份驗證的用戶或請求的屬性,例如 API 組、命名空間或資源。FlowSchema 還指定了此類請求應映射到哪個 PriorityLevelConfiguration。這兩個對象一起說,"我希望這種類型的請求計入這個正在進行的請求份額。"當請求命中 API 伺服器時,它將檢查每個 FlowSchema,直到找到一個與所有所需屬性匹配的 FlowSchema。如果多個 FlowSchema 與請求匹配,API 伺服器將選擇具有最小匹配優先級的 FlowSchema,該優先級在對象中指定為一個屬性。

    可以使用以下命令查看 FlowSchema 到 PriorityLevelConfigurations 的映射:

    $ kubectl get flowschemas

eks-exempt                     exempt            2                    <none>                7h19m   False
probes                         exempt            2                    <none>                7h19m   False
system-leader-election         leader-election   100                  ByUser                7h19m   False
endpoint-controller            workload-high     150                  ByUser                7h19m   False
workload-leader-election       leader-election   200                  ByUser                7h19m   False
system-node-high               node-high         400                  ByUser                7h19m   False
system-nodes                   system            500                  ByUser                7h19m   False
kube-controller-manager        workload-high     800                  ByNamespace           7h19m   False
kube-scheduler                 workload-high     800                  ByNamespace           7h19m   False
kube-system-service-accounts   workload-high     900                  ByNamespace           7h19m   False

global-default                 global-default    9900                 ByUser                7h19m   False
catch-all                      catch-all         10000                ByUser                7h19m   False

    PriorityLevelConfigurations 可以有 Queue、Reject 或 Exempt 類型。對於 Queue 和 Reject 類型,會對該優先級級別的最大正在進行的請求數強制實施限制,但當達到該限制時,行為會有所不同。例如,workload-high PriorityLevelConfiguration 使用 Queue 類型,並為控制器管理器、端點控制器、調度器、EKS 相關控制器和在 kube-system 命名空間中運行的 Pod 保留了 98 個請求。由於使用了 Queue 類型,API 伺服器將嘗試將請求保留在內存中,並希望在這些請求超時之前,正在進行的請求數量會降到 98 以下。如果給定請求在隊列中超時,或者已經排隊的請求太多,API 伺服器別無選擇,只能丟棄請求並向客戶端返回 429。請注意,排隊可能會防止請求收到 429,但它也會帶來更高的端到端延遲。

    現在考慮映射到 Reject 類型 catch-all PriorityLevelConfiguration 的 catch-all FlowSchema。如果客戶端達到 13 個正在進行的請求的限制,API 伺服器將不會進行排隊,而是立即丟棄請求並返回 429 響應代碼。最後,映射到 Exempt 類型 PriorityLevelConfiguration 的請求永遠不會收到 429,並且始終會立即分派。這用於高優先級請求,例如 healthz 請求或來自 system:masters 組的請求。

    監控 APF 和丟棄的請求¶

    要確認是否有任何請求由於 APF 而被丟棄,可以監控 API 伺服器指標 apiserver_flowcontrol_rejected_requests_total,以檢查受影響的 FlowSchemas 和 PriorityLevelConfigurations。例如,此指標顯示 100 個來自 service-accounts FlowSchema 的請求由於在 workload-low 隊列中超時而被丟棄:

    % kubectl get --raw /metrics | grep apiserver_flowcontrol_rejected_requests_total
apiserver_flowcontrol_rejected_requests_total{flow_schema="service-accounts",priority_level="workload-low",reason="time-out"} 100

    要檢查給定 PriorityLevelConfiguration 距離收到 429 或由於排隊而經歷增加的延遲有多近,您可以比較並發限制和正在使用的並發之間的差異。在此示例中,我們有 100 個請求的緩衝區。

    % kubectl get --raw /metrics | grep 'apiserver_flowcontrol_request_concurrency_limit.*workload-low'
apiserver_flowcontrol_request_concurrency_limit{priority_level="workload-low"} 245

% kubectl get --raw /metrics | grep 'apiserver_flowcontrol_request_concurrency_in_use.*workload-low'
apiserver_flowcontrol_request_concurrency_in_use{flow_schema="service-accounts",priority_level="workload-low"} 145

    要檢查給定 PriorityLevelConfiguration 是否正在經歷排隊但不一定丟棄請求,可以參考 apiserver_flowcontrol_current_inqueue_requests 指標:

    % kubectl get --raw /metrics | grep 'apiserver_flowcontrol_current_inqueue_requests.*workload-low'
apiserver_flowcontrol_current_inqueue_requests{flow_schema="service-accounts",priority_level="workload-low"} 10

    其他有用的 Prometheus 指標包括:

    • apiserver_flowcontrol_dispatched_requests_total
    • apiserver_flowcontrol_request_execution_seconds
    • apiserver_flowcontrol_request_wait_duration_seconds

    請參閱上游文檔以獲取APF 指標的完整列表。

    防止丟棄請求¶

    通過改變您的工作負載來防止 429¶

    當 APF 由於給定 PriorityLevelConfiguration 超過其允許的最大正在進行的請求數而丟棄請求時,受影響 FlowSchemas 中的客戶端可以減少在給定時間內執行的請求數量。這可以通過在出現 429 的期間減少總請求量來實現。請注意,長時間運行的請求(例如昂貴的列表調用)特別有問題,因為它們在整個執行時間內都被計為一個正在進行的請求。減少這些昂貴請求的數量或優化這些列表調用的延遲(例如,通過減少每個請求獲取的對象數量或切換到使用 watch 請求)可以幫助減少給定工作負載所需的總並發。

    通過改變您的 APF 設置來防止 429¶

    警告

    只有在您知道自己在做什麼的情況下才改變默認的 APF 設置。配置不當的 APF 設置可能會導致丟棄 API 伺服器請求和嚴重的工作負載中斷。

    防止丢棄请求的另一种方法是更改在 EKS 集群上安装的默认 FlowSchemas 或 PriorityLevelConfigurations。EKS 为给定的 Kubernetes 次版本安装了上游默认设置的 FlowSchemas 和 PriorityLevelConfigurations。除非在这些对象上设置以下注释为 false,否则 API 服务器将自动将它们恢复到默认值:

      metadata:
    annotations:
      apf.kubernetes.io/autoupdate-spec: "false"

    总的来说,可以通过以下方式修改 APF 设置:

    • 为您关心的请求分配更多的正在进行的请求容量。
    • 隔离非必要或昂贵的请求,以免占用其他请求类型的容量。

    这可以通过更改默认的 FlowSchemas 和 PriorityLevelConfigurations 或创建这些类型的新对象来实现。运营商可以增加相关 PriorityLevelConfigurations 对象的 assuredConcurrencyShares 值,以增加分配给它们的正在进行的请求份额。此外,如果应用程序可以处理更多的延迟,也可以增加在给定时间内可以排队的请求数量。

    另外,可以创建专门针对客户工作负载的新 FlowSchema 和 PriorityLevelConfigurations 对象。请注意,增加现有 PriorityLevelConfigurations 或新 PriorityLevelConfigurations 的 assuredConcurrencyShares 将导致其他桶可以处理的请求数量减少,因为总限制仍为每个 API 服务器 600 个正在进行的请求。

    在对 APF 默认值进行更改时,应该在非生产集群上监控这些指标,以确保更改设置不会导致意外的 429:

    1. 应该监控 apiserver_flowcontrol_rejected_requests_total 指标的所有 FlowSchemas,以确保没有桶开始丢弃请求。
    2. 应该比较 apiserver_flowcontrol_request_concurrency_limit 和 apiserver_flowcontrol_request_concurrency_in_use 的值,以确保使用中的并发不会危及该优先级级别的限制。

    定义新的 FlowSchema 和 PriorityLevelConfiguration 的一个常见用例是隔离。假设我们想将来自 Pod 的长时间运行的列表事件调用隔离到自己的请求份额。这将防止使用现有 service-accounts FlowSchema 的重要 Pod 请求收到 429 并被请求容量饿死。请记住,正在进行的请求总数是有限的,但是这个示例显示 APF 设置可以被修改以更好地划分给定工作负载的请求容量:

    隔离列表事件请求的示例 FlowSchema 对象:

    apiVersion: flowcontrol.apiserver.k8s.io/v1beta1
kind: FlowSchema
metadata:
  name: list-events-default-service-accounts
spec:
  distinguisherMethod:
    type: ByUser
  matchingPrecedence: 8000
  priorityLevelConfiguration:
    name: catch-all
  rules:
  - resourceRules:
    - apiGroups:
      - '*'
      namespaces:
      - default
      resources:
      - events
      verbs:
      - list
    subjects:
    - kind: ServiceAccount
      serviceAccount:
        name: default
        namespace: default
    • 此 FlowSchema 捕获 default 命名空间中 service accounts 发出的所有列表事件调用。
    • 匹配优先级 8000 低于现有 service-accounts FlowSchema 使用的 9000,因此这些列表事件调用将匹配 list-events-default-service-accounts 而不是 service-accounts。
    • 我们使用 catch-all PriorityLevelConfiguration 来隔离这些请求。此桶只允许 13 个正在进行的请求被这些长时间运行的列表事件调用使用。当 Pod 试图同时发出超过 13 个这样的请求时,它们将开始收到 429。

    从 API 服务器检索资源¶

    从 API 服务器获取信息是任何规模集群的预期行为。随着集群中资源数量的增加,请求频率和数据量可能会迅速成为控制平面的瓶颈,并导致 API 延迟和缓慢。根据延迟的严重程度,它可能会导致意外的停机,如果您不小心的话。

    了解您正在请求什么以及请求频率是避免这类问题的第一步。以下是基于扩展最佳实践的指导,以限制查询量。本节中的建议按顺序提供,从已知可最好扩展的选项开始。

    使用共享 Informer¶

    在构建与 Kubernetes API 集成的控制器和自动化时,您通常需要从 Kubernetes 资源中获取信息。如果您定期轮询这些资源,可能会对 API 服务器造成重大负载。

    使用 client-go 库中的 informer 将让您获得基于事件而不是轮询变更的好处。Informer 通过使用事件和变更的共享缓存进一步减少了负载,因此多个观察相同资源的控制器不会增加额外的负载。

    控制器应该避免轮询没有标签和字段选择器的集群范围资源,特别是在大型集群中。每个未过滤的轮询都需要从 etcd 通过 API 服务器发送大量不必要的数据,然后由客户端进行过滤。通过基于标签和命名空间进行过滤,您可以减少 API 服务器需要执行的工作量,以及发送到客户端的数据量。

    优化 Kubernetes API 使用¶

    使用自定义控制器或自动化调用 Kubernetes API 时,重要的是要限制对您需要的资源的调用。没有限制,您可能会对 API 服务器和 etcd 造成不必要的负载。

    建议您尽可能使用 watch 参数。默认行为是列出对象。要使用 watch 而不是列表,您可以在 API 请求的末尾附加 ?watch=true。例如,要在 default 命名空间中获取所有 Pod 并观察,请使用:

    /api/v1/namespaces/default/pods?watch=true

    如果您正在列出对象,您应该限制您正在列出的范围和返回的数据量。您可以通过向请求添加 limit=500 参数来限制返回的数据。fieldSelector 参数和 /namespace/ 路径可以用来确保您的列表范围尽可能窄。例如,要列出 default 命名空间中正在运行的 Pod,请使用以下 API 路径和参数。

    /api/v1/namespaces/default/pods?fieldSelector=status.phase=Running&limit=500

    或列出所有正在运行的 Pod:

    /api/v1/pods?fieldSelector=status.phase=Running&limit=500

    限制 watch 调用或列出对象的另一个选择是使用 resourceVersions,您可以在 Kubernetes 文档中阅读更多信息。没有 resourceVersion 参数,您将收到可用的最新版本,这需要 etcd 法定读取,这是数据库最昂贵和最慢的读取。resourceVersion 取决于您试图查询的资源,可以在 metadata.resourseVersion 字段中找到。在使用 watch 调用时,这也是推荐的,而不仅仅是列表调用。

    有一个特殊的 resourceVersion=0 可用,它将从 API 服务器缓存返回结果。这可以减少 etcd 负载,但不支持分页。

    /api/v1/namespaces/default/pods?resourceVersion=0
    建议使用 watch 并将 resourceVersion 设置为从前一个列表或 watch 接收的最新已知值。这在 client-go 中会自动处理。但如果您在其他语言中使用 k8s 客户端,建议仔细检查它。

    /api/v1/namespaces/default/pods?watch=true&resourceVersion=362812295
    如果您不带任何参数调用 API,这将是对 API 服务器和 etcd 最资源密集的。此调用将获取所有命名空间中的所有 Pod,而不进行分页或限制范围,并需要从 etcd 进行法定读取。
