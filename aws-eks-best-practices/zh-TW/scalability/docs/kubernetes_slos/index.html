    Kubernetes 上游 SLO¶

    Amazon EKS 運行與上游 Kubernetes 版本相同的程式碼,並確保 EKS 集群在 Kubernetes 社區定義的 SLO 內運行。 Kubernetes可擴展性特別興趣小組(SIG)定義了可擴展性目標,並通過 SLI 和 SLO 調查性能瓶頸。

    SLI 是我們衡量系統的方式,例如可用於確定系統"運行良好"程度的指標或度量,例如請求延遲或計數。 SLO 定義了系統"運行良好"時預期的值,例如請求延遲保持在 3 秒以內。 Kubernetes SLO 和 SLI 關注 Kubernetes 組件的性能,與專注於 EKS 集群端點可用性的 Amazon EKS 服務 SLA 完全獨立。

    Kubernetes 有許多功能允許用戶使用自定義插件或驅動程序擴展系統,如 CSI 驅動程序、准入 Webhook 和自動縮放器。 這些擴展可能會以不同方式大幅影響 Kubernetes 集群的性能,例如,如果 Webhook 目標不可用,則具有 failurePolicy=Ignore 的准入 Webhook 可能會增加 K8s API 請求的延遲。 Kubernetes Scalability SIG 使用"你承諾,我們承諾"框架定義可擴展性:

    如果你承諾: - 正確配置您的集群 - 合理使用擴展性功能 - 將集群中的負載保持在建議限制內

    那麼我們承諾您的集群可以擴展,即: - 所有 SLO 都得到滿足。

    Kubernetes SLO¶

    Kubernetes SLO 並未考慮可能影響集群的所有插件和外部限制,例如工作節點縮放或准入 Webhook。 這些 SLO 集中在Kubernetes 組件上,並確保 Kubernetes 操作和資源在預期範圍內運行。 SLO 有助於 Kubernetes 開發人員確保對 Kubernetes 代碼的更改不會降低整個系統的性能。

    Kuberntes Scalability SIG 定義了以下官方 SLO/SLI。 Amazon EKS 團隊定期對 EKS 集群運行可擴展性測試,以監控隨著變更和新版本發布而出現的性能降級。

    目標 定義 SLO
    API 請求延遲(變更) 處理每個(資源,動詞)對的變更 API 調用延遲,以最近 5 分鐘的第 99 百分位數為度量 在默認的 Kubernetes 安裝中,對於每個(資源,動詞)對,不包括虛擬和聚合資源以及自定義資源定義,每個集群日的第 99 百分位數 <= 1 秒
    API 請求延遲(只讀) 處理每個(資源,範圍)對的非流式只讀 API 調用延遲,以最近 5 分鐘的第 99 百分位數為度量 在默認的 Kubernetes 安裝中,對於每個(資源,範圍)對,不包括虛擬和聚合資源以及自定義資源定義,(a) 如果 scope=resource 則第 99 百分位數 <= 1 秒 (b) 否則(如果 scope=namespace 或 scope=cluster) 則 <= 30 秒
    Pod 啟動延遲 可調度無狀態 Pod 的啟動延遲,不包括拉取映像和運行初始化容器的時間,從 Pod 創建時間戳到所有容器報告為已啟動並通過 watch 觀察到,以最近 5 分鐘的第 99 百分位數為度量 在默認的 Kubernetes 安裝中,每個集群日的第 99 百分位數 <= 5 秒

    API 請求延遲¶

    kube-apiserver 默認將 --request-timeout 定義為 1m0s,這意味著請求最多可以運行一分鐘(60 秒)才會超時並被取消。 定義的 SLO 是根據請求的類型(變更或只讀)來區分的:

    變更¶

    Kubernetes 中的變更請求會修改資源,例如創建、刪除或更新。 這些請求很昂貴,因為這些變更必須先寫入etcd 後端,然後才能返回更新後的對象。 Etcd是一個分布式的 key-value 存儲,用於存儲所有 Kubernetes 集群數據。

    這個延遲是以 5 分鐘內的第 99 百分位數來衡量的,針對 Kubernetes 資源的(資源,動詞)對,例如這將測量創建 Pod 請求和更新節點請求的延遲。 請求延遲必須 <= 1 秒才能滿足 SLO。

    只讀¶

    只讀請求檢索單個資源(如獲取 Pod X)或集合(如"獲取命名空間 X 中的所有 Pod")。 kube-apiserver 維護了一個對象緩存,因此所請求的資源可能來自緩存,也可能需要先從 etcd 檢索。 這些延遲也是以 5 分鐘內的第 99 百分位數來衡量的,但只讀請求可以有不同的範圍。 SLO 定義了兩個不同的目標:

    • 對於對單個資源的請求(即 kubectl get pod -n mynamespace my-controller-xxx),請求延遲應保持在 <= 1 秒。
    • 對於請求命名空間或集群中的多個資源(例如 kubectl get pods -A)的延遲應保持在 <= 30 秒

    SLO 對不同的請求範圍有不同的目標值,因為對 Kubernetes 資源集合的請求希望在 SLO 內返回所有對象的詳細信息。 在大型集群或大量資源集合中,這可能會導致大型響應大小,需要一些時間才能返回。 例如,在運行數萬個 Pod 的集群中,每個 Pod 在 JSON 編碼時大約為 1 KiB,返回集群中的所有 Pod 將包括 10MB 或更多。 Kubernetes 客戶端可以通過使用 APIListChunking 分塊獲取大型資源集合來減少這個響應大小。

    Pod 啟動延遲¶

    這個 SLO 主要關注從 Pod 創建到容器實際開始執行的時間。 為了測量這一點,計算從記錄在 Pod 上的創建時間戳到WATCH 該 Pod報告容器已啟動的時間差(不包括容器映像拉取和初始化容器執行的時間)。 要滿足 SLO,每個集群日的第 99 百分位數 Pod 啟動延遲必須保持在 <= 5 秒。

    請注意,這個 SLO 假設工作節點已經處於就緒狀態,可以調度 Pod。 這個 SLO 不考慮映像拉取或初始化容器的執行,也限制了測試對象為"無狀態 Pod",不使用持久存儲插件。

    Kubernetes SLI 指標¶

    Kubernetes 還在通過向 Kubernetes 組件添加 Prometheus 指標來改善對 SLI 的可觀察性。 使用 Prometheus 查詢語言(PromQL)我們可以構建查詢,在 Prometheus 或 Grafana 儀表板中顯示 SLI 性能隨時間的變化,下面是上述 SLO 的一些示例。

    API 服務器請求延遲¶

    指標 定義
    apiserver_request_sli_duration_seconds 每個動詞、組、版本、資源、子資源、範圍和組件的響應延遲分佈(不包括 Webhook 持續時間和優先級和公平性隊列等待時間),以秒為單位。
    apiserver_request_duration_seconds 每個動詞、dry run 值、組、版本、資源、子資源、範圍和組件的響應延遲分佈,以秒為單位。

    注意:從 Kubernetes 1.27 開始提供 apiserver_request_sli_duration_seconds 指標。

    您可以使用這些指標來調查 API 服務器的響應時間,以及 Kubernetes 組件或其他插件/組件是否存在瓶頸。 下面的查詢基於社區 SLO 儀表板。

    API 請求延遲 SLI(變更) - 這個時間不包括 Webhook 執行或等待在隊列中的時間。 histogram_quantile(0.99, sum(rate(apiserver_request_sli_duration_seconds_bucket{verb=~"CREATE|DELETE|PATCH|POST|PUT", subresource!~"proxy|attach|log|exec|portforward"}[5m])) by (resource, subresource, verb, scope, le)) > 0

    API 請求延遲總計(變更) - 這是請求在 API 服務器上的總時間,這個時間可能比 SLI 時間長,因為它包括 Webhook 執行和 API 優先級和公平性等待時間。 histogram_quantile(0.99, sum(rate(apiserver_request_duration_seconds_bucket{verb=~"CREATE|DELETE|PATCH|POST|PUT", subresource!~"proxy|attach|log|exec|portforward"}[5m])) by (resource, subresource, verb, scope, le)) > 0

    在這些查詢中,我們排除了流式 API 請求,例如 kubectl port-forward 或 kubectl exec 請求(subresource!~"proxy|attach|log|exec|portforward")。 我們只過濾修改對象的 Kubernetes 動詞(verb=~"CREATE|DELETE|PATCH|POST|PUT")。 然後我們計算最近 5 分鐘內的 99 百分位延遲。

    我們可以使用類似的查詢來獲取只讀 API 請求,我們只需修改要過濾的動詞,包括只讀操作 LIST 和 GET。 根據請求的範圍,也有不同的 SLO 閾值,即獲取單個資源或列出多個資源。

    API 請求延遲 SLI(只讀) - 這個時間不包括 Webhook 執行或等待在隊列中的時間。 對於單個資源(scope=resource, threshold=1s) histogram_quantile(0.99, sum(rate(apiserver_request_sli_duration_seconds_bucket{verb=~"GET", scope=~"resource"}[5m])) by (resource, subresource, verb, scope, le))

    對於同一命名空間中的資源集合(scope=namespace, threshold=5s) histogram_quantile(0.99, sum(rate(apiserver_request_sli_duration_seconds_bucket{verb=~"LIST", scope=~"namespace"}[5m])) by (resource, subresource, verb, scope, le))

    對於整個集群中的資源集合(scope=cluster, threshold=30s) histogram_quantile(0.99, sum(rate(apiserver_request_sli_duration_seconds_bucket{verb=~"LIST", scope=~"cluster"}[5m])) by (resource, subresource, verb, scope, le))

    API 請求延遲總計(只讀) - 這是請求在 API 服務器上的總時間,這個時間可能比 SLI 時間長,因為它包括 Webhook 執行和等待時間。 對於單個資源(scope=resource, threshold=1s) histogram_quantile(0.99, sum(rate(apiserver_request_duration_seconds_bucket{verb=~"GET", scope=~"resource"}[5m])) by (resource, subresource, verb, scope, le))

    對於同一命名空間中的資源集合(scope=namespace, threshold=5s) histogram_quantile(0.99, sum(rate(apiserver_request_duration_seconds_bucket{verb=~"LIST", scope=~"namespace"}[5m])) by (resource, subresource, verb, scope, le))

    對於整個集群中的資源集合(scope=cluster, threshold=30s) histogram_quantile(0.99, sum(rate(apiserver_request_duration_seconds_bucket{verb=~"LIST", scope=~"cluster"}[5m])) by (resource, subresource, verb, scope, le))

    SLI 指標提供了 Kubernetes 組件性能的洞察,因為它們排除了請求在 API 優先級和公平性隊列中等待的時間、通過准入 Webhook 處理的時間,或其他 Kubernetes 擴展的時間。 總計指標提供了更全面的視圖,因為它反映了您的應用程序等待 API 服務器響應的時間。 比較這些指標可以提供洞察請求處理延遲的引入位置。

    Pod 啟動延遲¶

    指標 定義
    kubelet_pod_start_sli_duration_seconds 啟動 Pod 的持續時間(以秒為單位),不包括拉取映像和運行初始化容器的時間,從 Pod 創建時間戳到觀察到所有容器已啟動
    kubelet_pod_start_duration_seconds 從 kubelet 第一次看到 Pod 到 Pod 開始運行的持續時間(以秒為單位)。 這不包括調度 Pod 或擴展工作節點容量的時間。

    注意:從 Kubernetes 1.27 開始提供 kubelet_pod_start_sli_duration_seconds 指標。

    與上述查詢類似,您可以使用這些指標來洞察節點縮放、映像拉取和初始化容器如何延遲 Pod 啟動。

    Pod 啟動延遲 SLI - 這是從 Pod 創建到應用程序容器報告為運行的時間。 這包括工作節點容量可用和 Pod 被調度的時間,但不包括拉取映像或運行初始化容器的時間。 histogram_quantile(0.99, sum(rate(kubelet_pod_start_sli_duration_seconds_bucket[5m])) by (le))

    Pod 啟動延遲總計 - 這是 kubelet 第一次啟動 Pod 的時間。 這是從 kubelet 通過 WATCH 接收 Pod 開始測量的,不包括工作節點縮放或調度的時間。 這包括拉取映像和運行初始化容器的時間。 histogram_quantile(0.99, sum(rate(kubelet_pod_start_duration_seconds_bucket[5m])) by (le))

    您集群上的 SLO¶

    如果您正在收集 EKS 集群中 Kubernetes 資源的 Prometheus 指標,您可以深入了解 Kubernetes 控制平面組件的性能。

    perf-tests 存儲庫包含 Grafana 儀表板,在測試期間顯示集群的延遲和關鍵性能指標。 perf-tests 配置利用了kube-prometheus-stack,這是一個開源項目,預先配置好收集 Kubernetes 指標,但您也可以使用 Amazon Managed Prometheus 和 Amazon Managed Grafana。

    如果您使用 kube-prometheus-stack 或類似的 Prometheus 解決方案,您可以安裝相同的儀表板來實時觀察集群上的 SLO。

    1. 您首先需要使用 kubectl apply -f prometheus-rules.yaml 安裝用於儀表板的 Prometheus 規則。 您可以在此處下載規則副本:https://github.com/kubernetes/perf-tests/blob/master/clusterloader2/pkg/prometheus/manifests/prometheus-rules.yaml
      1. 確保檢查文件中的命名空間是否與您的環境匹配
      2. 驗證標籤是否與 kube-prometheus-stack 的 prometheus.prometheusSpec.ruleSelector Helm 值匹配
    2. 然後您可以在 Grafana 中安裝儀表板。 json 儀表板和生成它們的 python 腳本可在此處獲得:https://github.com/kubernetes/perf-tests/tree/master/clusterloader2/pkg/prometheus/manifests/dashboards
      1. slo.json 儀表板顯示集群在 Kubernetes SLO 方面的性能

    請考慮到 SLO 專注於集群中 Kubernetes 組件的性能,但您可以查看其他指標,這些指標提供不同的觀點或洞察力。 Kubernetes 社區項目如 Kube-state-metrics可以幫助您快速分析集群中的趨勢。 大多數來自 Kubernetes 社區的常見插件和驅動程序也會發出 Prometheus 指標,允許您調查自動縮放器或自定義調度器等內容。

    Observability Best Practices Guide提供了其他 Kubernetes 指標的示例,您可以使用它們獲得更深入的洞察。
