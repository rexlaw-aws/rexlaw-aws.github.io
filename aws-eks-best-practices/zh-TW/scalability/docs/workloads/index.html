    工作負載¶

    工作負載對集群可擴展的大小有影響。大量使用 Kubernetes API 的工作負載將限制單個集群中可以運行的總工作負載數量,但您可以更改一些預設值來幫助減少負載。

    Kubernetes 集群中的工作負載可以訪問與 Kubernetes API 集成的功能(例如 Secrets 和 ServiceAccounts),但這些功能並非總是必需的,如果未使用,應將其禁用。限制工作負載對 Kubernetes 控制平面的訪問和依賴將增加集群中可以運行的工作負載數量,並通過刪除對工作負載的不必要訪問和實施最小特權實踐來提高集群的安全性。請閱讀安全最佳實踐以獲取更多信息。

    使用 IPv6 進行 pod 網絡¶

    您無法將 VPC 從 IPv4 過渡到 IPv6,因此在配置集群之前啟用 IPv6 很重要。如果您在 VPC 中啟用 IPv6,這並不意味著您必須使用它,如果您的 pod 和服務使用 IPv6,您仍然可以將流量路由到和從 IPv4 地址。請參閱EKS 網絡最佳實踐以獲取更多信息。

    在您的集群中使用IPv6可以避免一些最常見的集群和工作負載擴展限制。IPv6 可以避免 IP 地址耗盡,這種情況下 pod 和節點無法創建,因為沒有可用的 IP 地址。它還可以提高每個節點的性能,因為 pod 可以更快地獲得 IP 地址,減少了每個節點的 ENI 附加數量。您可以通過在 VPC CNI 中使用IPv4 前綴模式來實現類似的節點性能,但您仍然需要確保 VPC 中有足夠的 IP 地址可用。

    限制每個命名空間的服務數量¶

    每個命名空間最多 5,000 個服務,集群最多 10,000 個服務。為了幫助組織工作負載和服務、提高性能,並避免命名空間範圍內資源的級聯影響,我們建議將每個命名空間的服務數量限制在 500 個以內。

    隨著集群中服務總數的增加,kube-proxy 每個節點創建的 IP 表規則數量也會增加。生成成千上萬的 IP 表規則並通過這些規則路由數據包會對節點產生性能影響,並增加網絡延遲。

    創建包含單個應用程序環境的 Kubernetes 命名空間,只要每個命名空間的服務數量不超過 500 個。這將使服務發現足夠小,避免服務發現限制,也可以幫助您避免服務命名衝突。應用程序環境(例如開發、測試、生產)應該使用單獨的 EKS 集群,而不是命名空間。

    了解彈性負載均衡器配額¶

    在創建您的服務時,請考慮您將使用的負載均衡類型(例如網絡負載均衡器(NLB)或應用程序負載均衡器(ALB))。每種負載均衡器類型提供不同的功能,並有不同的配額。一些預設配額可以調整,但有一些配額最大值是不能更改的。要查看您的帳戶配額和使用情況,請在 AWS 控制台中查看服務配額儀表板。

    例如,默認 ALB 目標是 1000。如果您的服務有超過 1000 個端點,您將需要增加配額或將服務分散到多個 ALB 上,或使用 Kubernetes Ingress。默認 NLB 目標是 3000,但每個可用區(AZ)限制為 500 個目標。如果您的集群為 NLB 服務運行超過 500 個 pod,您將需要使用多個 AZ 或請求提高配額限制。

    使用與服務耦合的負載均衡器的另一種替代方法是使用入口控制器。AWS Load Balancer 控制器可以為入口資源創建 ALB,但您可能考慮在集群中運行專用的控制器。集群內的入口控制器允許您通過在集群內運行反向代理來公開多個 Kubernetes 服務。控制器具有不同的功能,例如對Gateway API的支持,這可能會根據您的工作負載數量和大小而有所不同。

    使用 Route 53、Global Accelerator 或 CloudFront¶

    要使用多個負載均衡器公開服務,您需要使用Amazon CloudFront、AWS Global Accelerator或Amazon Route 53將所有負載均衡器公開為單個面向客戶的端點。每個選項都有不同的優點,可以單獨或組合使用,具體取決於您的需求。

    Route 53 可以在一個通用名稱下公開多個負載均衡器,並根據分配的權重將流量路由到每個負載均衡器。您可以在文檔中了解更多關於 DNS 權重的信息,並在AWS Load Balancer Controller 文檔中了解如何使用Kubernetes external DNS 控制器實現它們。

    Global Accelerator 可以根據請求 IP 地址將工作負載路由到最近的區域。這對於部署到多個區域的工作負載可能很有用,但它不會改善單個區域內集群的路由。將 Route 53 與 Global Accelerator 結合使用還有其他好處,如健康檢查和自動故障轉移(如果可用區不可用)。您可以在這篇博客文章中看到使用 Global Accelerator 和 Route 53 的示例。

    CloudFront 可以與 Route 53 和 Global Accelerator 一起使用,也可以單獨使用,將流量路由到多個目的地。CloudFront 會緩存來自源的資產,這可能會根據您要提供的內容減少帶寬需求。

    使用 EndpointSlices 而不是 Endpoints¶

    在發現與服務標籤匹配的 pod 時,您應該使用EndpointSlices而不是 Endpoints。Endpoints 是一種在小規模時公開服務的簡單方式,但自動縮放或更新的大型服務會在 Kubernetes 控制平面上產生大量流量。EndpointSlices 具有自動分組功能,可以啟用拓撲感知提示等功能。

    並非所有控制器都默認使用 EndpointSlices。您應該驗證您的控制器設置並在需要時啟用它。對於AWS Load Balancer 控制器,您應該啟用可選標誌--enable-endpoint-slices來使用 EndpointSlices。

    如果可能,使用不可變和外部 Secrets¶

    kubelet 會保留 pod 在該節點上使用的 Secrets 的當前密鑰和值的緩存。kubelet 會設置一個監視 Secrets 的監視器,以檢測更改。隨著集群的擴展,不斷增加的監視數量會對 API 服務器的性能產生負面影響。

    有兩種策略可以減少對 Secrets 的監視:

    • 對於不需要訪問 Kubernetes 資源的應用程序,您可以通過設置 automountServiceAccountToken: false 來禁用自動掛載服務帳戶 Secrets
    • 如果您的應用程序 Secrets 是靜態的,將來不會修改,請將Secret 標記為不可變。kubelet 不會為不可變的 Secrets 維護 API 監視。

    要禁用自動將服務帳戶掛載到 pod,您可以在工作負載中使用以下設置。如果特定工作負載需要服務帳戶,您可以覆蓋這些設置。

    apiVersion: v1
kind: ServiceAccount
metadata:
  name: app
automountServiceAccountToken: true

    監控集群中 Secrets 的數量,在超過 10,000 個的限制之前。您可以使用以下命令查看集群中 Secrets 的總數。您應該通過集群監控工具監控此限制。

    kubectl get secrets -A | wc -l

    您應該設置監控,在達到此限制之前向集群管理員發出警報。考慮使用外部 Secrets 管理選項,如AWS Key Management Service (AWS KMS)或Hashicorp Vault與Secrets Store CSI 驅動程序。

    限制部署歷史記錄¶

    創建、更新或刪除 pod 可能會很慢,因為舊對象仍然在集群中被跟踪。您可以減少部署的revisionHistoryLimit,以清理較舊的 ReplicaSets,這將降低 Kubernetes 控制器管理器跟踪的總對象數量。Deployments 的默認歷史記錄限制為 10。

    如果您的集群通過 CronJobs 或其他機制創建大量 job 對象,您應該使用ttlSecondsAfterFinished 設置來自動清理集群中完成的 pod。這將在指定的時間後從 job 歷史記錄中刪除成功執行的 job。

    默認禁用 enableServiceLinks¶

    當 Pod 在節點上運行時,kubelet 會為每個活動服務添加一組環境變量。Linux 進程對其環境的最大大小有限制,如果您的命名空間中有太多服務,這個限制可能會被達到。每個命名空間的服務數量不應超過 5,000。超過這個數量後,服務環境變量的數量會超過 shell 限制,導致 pod 在啟動時崩潰。

    還有其他原因 pod 不應該使用服務環境變量進行服務發現。環境變量名稱衝突、洩露服務名稱和總環境大小都是一些問題。您應該使用 CoreDNS 來發現服務端點。

    限制每個資源的動態准入 Webhooks¶

    動態准入 Webhooks包括准入 Webhooks 和變更 Webhooks。它們是 Kubernetes 控制平面之外的 API 端點,在資源發送到 Kubernetes API 時按順序調用。每個 Webhook 的默認超時時間為 10 秒,如果您有多個 Webhook 或任何一個 Webhook 超時,都可能增加 API 請求的時間。

    確保您的 Webhook 高度可用,特別是在 AZ 故障期間,並正確設置失敗策略以拒絕資源或忽略失敗。不要在不需要時調用 Webhook,允許 --dry-run kubectl 命令繞過 Webhook。

    apiVersion: admission.k8s.io/v1
kind: AdmissionReview
request:
  dryRun: False

    變更 Webhook 可以頻繁地修改資源。如果您有 5 個變更 Webhook 並部署 50 個資源,etcd 將存儲每個資源的所有版本,直到壓縮運行(每 5 分鐘)才刪除修改資源的舊版本。在這種情況下,當 etcd 刪除過時的資源時,將有 200 個資源版本被刪除,並且取決於資源的大小,可能會在碎片整理運行(每 15 分鐘)之前在 etcd 主機上佔用大量空間。

    這種碎片整理可能會導致 etcd 暫停,這可能會對 Kubernetes API 和控制器產生其他影響。您應該避免頻繁修改大型資源或快速連續修改數百個資源。
