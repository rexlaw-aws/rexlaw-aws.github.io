    Kubernetes 叢集自動調整器¶

    概覽¶

    Kubernetes 叢集自動調整器是由 SIG Autoscaling 維護的熱門叢集自動調整解決方案。它負責確保您的叢集有足夠的節點來調度您的 pod,而不浪費資源。它會監視無法調度的 pod 和利用不足的節點。然後,它模擬添加或刪除節點,然後將變更應用於您的叢集。Cluster Autoscaler 中的 AWS 雲端提供者實現控制您 EC2 自動縮放群組的 .DesiredReplicas 欄位。

    本指南將提供配置 Cluster Autoscaler 的心智模型,並選擇最佳的權衡方案以滿足您組織的需求。雖然沒有單一最佳配置,但有一組配置選項可以讓您在性能、可擴展性、成本和可用性之間進行權衡。此外,本指南還將提供優化 AWS 配置的技巧和最佳實踐。

    詞彙表¶

    本文檔中將頻繁使用以下術語。這些術語可能有廣泛的含義,但在本文檔的目的中限定為以下定義。

    可擴展性指 Cluster Autoscaler 在 Kubernetes 叢集中的 pod 和節點數量增加時的性能。隨著可擴展性限制的達到,Cluster Autoscaler 的性能和功能會降低。當 Cluster Autoscaler 超過其可擴展性限制時,它可能無法再在您的叢集中添加或刪除節點。

    性能指 Cluster Autoscaler 做出和執行縮放決策的速度。完美的 Cluster Autoscaler 會立即做出決定並觸發縮放操作,以響應 pod 無法調度等刺激。

    可用性意味著 pod 可以快速調度而不會中斷。這包括當新創建的 pod 需要調度時,以及當縮減節點時終止分配到該節點的任何剩餘 pod。

    成本由擴展和縮減事件背後的決策決定。如果現有節點利用不足或添加了太大的新節點,資源就會浪費。根據用例的不同,由於過於激進的縮減決策而過早終止 pod 也可能會產生成本。

    節點組是 Kubernetes 中節點組的抽象概念。它不是真正的 Kubernetes 資源,而是存在於 Cluster Autoscaler、Cluster API 和其他組件中的抽象概念。同一節點組內的節點共享標籤和污點等屬性,但可能由多個可用性區域或實例類型組成。

    EC2 自動縮放群組可用作 EC2 上節點組的實現。EC2 自動縮放群組配置為啟動實例,這些實例會自動加入其 Kubernetes 叢集並將標籤和污點應用於 Kubernetes API 中相應的節點資源。

    EC2 托管節點組是 EC2 上節點組的另一種實現。它抽象了手動配置 EC2 自動縮放群組的複雜性,並提供了節點版本升級和優雅節點終止等其他管理功能。

    操作 Cluster Autoscaler¶

    Cluster Autoscaler 通常作為部署安裝在您的叢集中。它使用領導者選舉來確保高可用性,但工作由單個副本完成。它不能水平擴展。對於基本設置,使用提供的安裝說明應該可以開箱即用,但需要注意一些事項。

    確保:

    • Cluster Autoscaler 的版本與叢集版本匹配。跨版本兼容性未經測試或支持。
    • 啟用自動發現,除非您有特定的高級用例阻止使用此模式。

    採用最小權限訪問 IAM 角色¶

    使用自動發現時,我們強烈建議您採用最小權限訪問,將 autoscaling:SetDesiredCapacity 和 autoscaling:TerminateInstanceInAutoScalingGroup 操作限制在當前叢集的自動縮放群組範圍內。

    這將防止在一個叢集中運行的 Cluster Autoscaler 修改另一個叢集中的節點組,即使 --node-group-auto-discovery 參數沒有使用標籤(例如 k8s.io/cluster-autoscaler/<cluster-name>)將其範圍縮小到叢集的節點組。

    {
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "autoscaling:SetDesiredCapacity",
                "autoscaling:TerminateInstanceInAutoScalingGroup"
            ],
            "Resource": "*",
            "Condition": {
                "StringEquals": {
                    "aws:ResourceTag/k8s.io/cluster-autoscaler/enabled": "true",
                    "aws:ResourceTag/k8s.io/cluster-autoscaler/<my-cluster>": "owned"
                }
            }
        },
        {
            "Effect": "Allow",
            "Action": [
                "autoscaling:DescribeAutoScalingGroups",
                "autoscaling:DescribeAutoScalingInstances",
                "autoscaling:DescribeLaunchConfigurations",
                "autoscaling:DescribeScalingActivities",
                "autoscaling:DescribeTags",
                "ec2:DescribeImages",
                "ec2:DescribeInstanceTypes",
                "ec2:DescribeLaunchTemplateVersions",
                "ec2:GetInstanceTypesFromInstanceRequirements",
                "eks:DescribeNodegroup"
            ],
            "Resource": "*"
        }
    ]
}

    配置您的節點組¶

    有效的自動調整從正確配置一組節點組開始。為您的叢集選擇正確的節點組集合是最大化可用性和降低成本的關鍵。AWS 使用 EC2 自動縮放群組實現節點組,這對於大量使用案例都很靈活。但是,Cluster Autoscaler 對您的節點組做出一些假設。保持您的 EC2 自動縮放群組配置與這些假設一致將最小化不需要的行為。

    確保:

    • 同一節點組中的每個節點都有相同的調度屬性,如標籤、污點和資源。
    • 對於 MixedInstancePolicies,實例類型的 CPU、內存和 GPU 必須相同
    • 策略中指定的第一種實例類型將用於模擬調度。
    • 如果您的策略有更多資源的其他實例類型,擴展後可能會浪費資源。
    • 如果較小,您的 pod 可能無法調度到新實例上,因為容量不足。
    • 節點數量較多的節點組優於節點數量較少的多個節點組。這將對可擴展性產生最大影響。
    • 盡可能使用 EC2 功能,而不是同時支持的系統(如區域、MixedInstancePolicy)

    注意:我們建議使用 EKS 托管節點組。托管節點組附帶強大的管理功能,包括 Cluster Autoscaler 的功能,如自動 EC2 自動縮放群組發現和優雅節點終止。

    優化性能和可擴展性¶

    了解自動調整算法的時間複雜度將有助於調整 Cluster Autoscaler,使其在擁有超過 1,000 個節點的大型叢集中繼續平穩運行。

    調整 Cluster Autoscaler 可擴展性的主要控制旋鈕是提供給該進程的資源、算法的掃描間隔以及叢集中的節點組數量。還有其他涉及算法真實時間複雜度的因素,如調度插件複雜性和 pod 數量。這些被視為不可配置的參數,因為它們是叢集工作負載的自然特性,無法輕易調整。

    Cluster Autoscaler 將整個叢集的狀態(包括 Pod、節點和節點組)加載到內存中。在每個掃描間隔中,算法識別無法調度的 pod 並為每個節點組模擬調度。調整這些因素會有不同的權衡,應該仔細考慮您的用例。

    垂直自動調整 Cluster Autoscaler¶

    擴展 Cluster Autoscaler 以適應更大的叢集的最簡單方法是增加其部署的資源請求。對於大型叢集,應同時增加內存和 CPU,但這會因叢集大小而有很大差異。自動調整算法將所有 pod 和節點存儲在內存中,這可能會導致內存占用超過 1 GB。通常手動增加資源。如果發現持續調整資源會造成運營負擔,請考慮使用 Addon Resizer 或 Vertical Pod Autoscaler。

    減少節點組數量¶

    最小化節點組數量是確保 Cluster Autoscaler 在大型叢集上繼續良好運行的一種方法。對於某些組織來說,這可能很具挑戰性,因為他們會根據團隊或應用程序來構造節點組。雖然這在 Kubernetes API 中得到完全支持,但這被認為是 Cluster Autoscaler 的反模式,會影響可擴展性。使用多個節點組有很多原因(例如 Spot 或 GPU),但在許多情況下,有替代設計可以實現相同的效果,同時使用較少的組。

    確保:

    • 使用命名空間而不是節點組來隔離 pod。
    • 在低信任的多租戶叢集中可能無法做到這一點。
    • 正確設置 Pod ResourceRequests 和 ResourceLimits 以避免資源爭用。
    • 更大的實例類型將導致更優化的裝箱和減少系統 pod 開銷。
    • 使用 NodeTaints 或 NodeSelectors 來調度 pod 作為例外,而不是規則。
    • 區域資源定義為單個 EC2 自動縮放群組,具有多個可用性區域。

    減少掃描間隔¶

    低掃描間隔(例如 10 秒)將確保 Cluster Autoscaler 在 pod 無法調度時能夠快速響應。但是,每次掃描都會導致對 Kubernetes API 和 EC2 自動縮放群組或 EKS 托管節點組 API 的大量 API 調用。這些 API 調用可能會導致速率限制或甚至 Kubernetes 控制平面的服務不可用。

    默認掃描間隔為 10 秒,但在 AWS 上,啟動節點需要明顯更長的時間。這意味著您可以增加間隔而不會顯著增加整體擴展時間。例如,如果啟動節點需要 2 分鐘,將間隔更改為 1 分鐘將以 6 倍減少 API 調用為代價,縮放速度降低 38%。

    跨節點組分片¶

    Cluster Autoscaler 可以配置為僅對特定節點組進行操作。使用此功能,可以部署多個 Cluster Autoscaler 實例,每個實例配置為對不同的節點組進行操作。此策略允許您使用任意數量的節點組,以成本換取可擴展性。我們只建議在提高性能的最後手段中使用這種方法。

    Cluster Autoscaler 最初並未設計用於此配置,因此會產生一些副作用。由於分片之間不會通信,多個自動調整器可能會嘗試調度同一個無法調度的 pod。這可能會導致不必要地擴展多個節點組。這些額外的節點將在 scale-down-delay 後縮減。

    metadata:
  name: cluster-autoscaler
  namespace: cluster-autoscaler-1

...

--nodes=1:10:k8s-worker-asg-1
--nodes=1:10:k8s-worker-asg-2

---

metadata:
  name: cluster-autoscaler
  namespace: cluster-autoscaler-2

...

--nodes=1:10:k8s-worker-asg-3
--nodes=1:10:k8s-worker-asg-4

    確保:

    • 每個分片都配置為指向唯一的 EC2 自動縮放群組
    • 每個分片都部署到一個單獨的命名空間,以避免領導者選舉衝突

    優化成本和可用性¶

    Spot 實例¶

    您可以在節點組中使用 Spot 實例,並節省高達 90% 的按需價格,代價是 Spot 實例可能會在 EC2 需要容量時隨時被中斷。當您的 EC2 自動縮放群組無法由於缺乏可用容量而擴展時,將發生容量不足錯誤。通過選擇多個實例系列來最大化多樣性,可以增加實現所需規模的機會,並減少 Spot 實例中斷對叢集可用性的影響。使用 Spot 實例的混合實例策略是增加多樣性而不增加節點組數量的好方法。請記住,如果您需要保證資源,請使用按需實例而不是 Spot 實例。

    在配置混合實例策略時,所有實例類型的資源容量必須相似。自動調整器的調度模擬器使用 MixedInstancePolicy 中指定的第一個 InstanceType。如果後續的實例類型更大,擴展後可能會浪費資源。如果更小,您的 pod 可能無法調度到新實例上,因為容量不足。例如,M4、M5、M5a 和 M5n 實例的 CPU 和內存都非常相似,是混合實例策略的良好候選者。EC2 實例選擇器工具可以幫助您識別相似的實例類型。

    建議將按需和 Spot 容量隔離到單獨的 EC2 自動縮放群組中。這優於使用基本容量策略,因為調度屬性根本不同。由於 Spot 實例可能隨時被中斷(當 EC2 需要容量時),用戶通常會污染其可搶占節點,要求對搶占行為有明確的 pod 容忍。這些污點導致節點的調度屬性不同,因此它們應該被分離到多個 EC2 自動縮放群組中。

    Cluster Autoscaler 有一個擴展器的概念,它提供了不同的策略來選擇擴展哪個節點組。--expander=least-waste 是一個很好的通用默認值,如果您打算使用多個節點組來實現 Spot 實例多樣化(如上圖所述),它可以通過擴展後最佳利用的節點組來進一步優化節點組的成本。

    優先考慮節點組/ASG¶

    您還可以使用優先級擴展器配置基於優先級的自動調整。--expander=priority 使您的叢集能夠優先考慮節點組/ASG,如果由於任何原因無法擴展,它將選擇優先列表中的下一個節點組。這在某些情況下很有用,例如,您希望使用 P3 實例類型,因為它們的 GPU 為您的工作負載提供了最佳性能,但作為第二選擇,您也可以使用 P2 實例類型。

    apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-autoscaler-priority-expander
  namespace: kube-system
data:
  priorities: |-
    10:
      - .*p2-node-group.*
    50:
      - .*p3-node-group.*

    Cluster Autoscaler 將嘗試擴展與名稱 p3-node-group 匹配的 EC2 自動縮放群組。如果此操作在 --max-node-provision-time 內不成功,它將嘗試擴展與名稱 p2-node-group 匹配的 EC2 自動縮放群組。 此值默認為 15 分鐘,可以減少以獲得更快的節點組選擇,但如果值太低,可能會導致不必要的擴展。

    過度配置¶

    Cluster Autoscaler 通過確保只在需要時添加節點並在未使用時刪除節點來最大限度地降低成本。這會大大影響部署延遲,因為許多 pod 將被迫等待節點擴展才能調度。節點可能需要數分鐘才能可用,這可能會將 pod 調度延遲提高一個數量級。

    這可以通過過度配置來緩解,這會以成本換取調度延遲。過度配置是使用具有負優先級的臨時 pod 實現的,這些 pod 佔用叢集中的空間。當新創建的 pod 無法調度且具有更高優先級時,臨時 pod 將被搶占以騰出空間。然後,臨時 pod 變得無法調度,觸發 Cluster Autoscaler 擴展新的過度配置節點。

    過度配置還有其他不太明顯的好處。沒有過度配置,高利用率叢集的一個副作用是 pod 將使用 Pod 或節點親和性的 preferredDuringSchedulingIgnoredDuringExecution 規則做出更不優化的調度決策。一個常見的用例是使用 AntiAffinity 跨可用性區域分離高可用性應用程序的 pod。過度配置可以顯著提高找到正確區域節點的機會。

    過度配置容量的數量是您組織的一個謹慎的業務決策。從根本上說,這是性能和成本之間的權衡。一種決策方式是確定您的平均擴展頻率,並將其除以新節點的供應時間。例如,如果平均每 30 秒需要一個新節點,而 EC2 需要 30 秒來配置一個新節點,一個節點的過度配置將確保始終有一個額外的節點可用,將調度延遲減少 30 秒,代價是一個額外的 EC2 實例。為了改善區域調度決策,過度配置等於您 EC2 自動縮放群組中可用性區域數量的節點數,以確保調度器可以為傳入的 pod 選擇最佳區域。

    防止縮減驅逐¶

    某些工作負載驅逐起來代價很高。大數據分析、機器學習任務和測試運行最終會完成,但如果中斷則必須重新啟動。Cluster Autoscaler 將嘗試縮減任何低於 scale-down-utilization-threshold 的節點,這將中斷節點上的任何剩餘 pod。通過確保被認為昂貴驅逐的 pod 受到 Cluster Autoscaler 識別的標籤保護,可以防止這種情況發生。

    確保:

    • 昂貴驅逐的 pod 具有註釋 cluster-autoscaler.kubernetes.io/safe-to-evict=false

    高級用例¶

    EBS 卷¶

    持久存儲對於構建有狀態應用程序(如數據庫或分佈式緩存)至關重要。EBS 卷在 Kubernetes 上實現了這一用例,但限於特定區域。如果使用不同區域的單獨 EBS 卷對這些應用程序進行分片,它們可以實現高可用性。Cluster Autoscaler 然後可以平衡 EC2 自動縮放群組的縮放。

    確保:

    • 啟用節點組平衡,設置 balance-similar-node-groups=true。
    • 節點組配置相同,除了不同的可用性區域和 EBS 卷。

    協同調度¶

    機器學習分佈式培訓作業從同區節點配置的最小延遲中獲益很多。這些工作負載部署多個 pod 到特定區域。這可以通過為所有協調 pod 設置 Pod 親和性或使用 topologyKey: failure-domain.beta.kubernetes.io/zone 設置節點親和性來實現。Cluster Autoscaler 然後將擴展特定區域以滿足需求。您可能希望分配多個 EC2 自動縮放群組,每個可用性區域一個,以實現整個協調工作負載的故障轉移。

    確保:

    • 禁用節點組平衡,設置 balance-similar-node-groups=false
    • 當叢集包含區域和區域節點組時,使用節點親和性和/或Pod 搶占。
    • 使用節點親和性來強制或鼓勵區域 pod 避免區域節點組,反之亦然。
    • 如果區域 pod 調度到區域節點組,這將導致區域 pod 的容量不平衡。
    • 如果您的區域工作負載可以容忍中斷和重新定位,請配置Pod 搶占以使區域擴展的 pod 強制搶占和重新調度到競爭較少的區域。

    加速器¶

    一些叢集利用專用硬件加速器,如 GPU。在擴展時,加速器設備插件可能需要數分鐘才能向叢集宣告資源。Cluster Autoscaler 已模擬該節點將具有加速器,但直到加速器準備就緒並更新節點的可用資源,待定的 pod 才無法調度到該節點。這可能導致重複不必要的擴展。

    此外,具有加速器和高 CPU 或內存利用率的節點將不會被考慮縮減,即使加速器未被使用。這種行為可能由於加速器相對成本較高而變得昂貴。相反,Cluster Autoscaler 可以應用特殊規則,如果節點有未使用的加速器,則將其視為縮減候選。

    為了確保這些情況下的正確行為,您可以配置 GPU 節點上的 kubelet,使其在加入叢集之前標記節點。Cluster Autoscaler 將使用此標籤選擇器來觸發加速器優化行為。

    確保:

    • GPU 節點的 Kubelet 配置有 --node-labels k8s.amazonaws.com/accelerator=$ACCELERATOR_TYPE
    • 具有加速器的節點遵守上述相同的調度屬性規則。

    從 0 開始縮放¶

    Cluster Autoscaler 能夠將節點組縮放到 0 並從 0 開始,這可以帶來顯著的成本節約。它通過檢查自動縮放組的 LaunchConfiguration 或 LaunchTemplate 中指定的 InstanceType 來發現 CPU、內存和 GPU 資源。一些 pod 需要額外的資源,如 WindowsENI 或 PrivateIPv4Address,或特定的 NodeSelectors 或 Taints,這些無法從 LaunchConfiguration 中發現。Cluster Autoscaler 可以通過從 EC2 自動縮放群組的標籤中發現這些因素來考慮它們。例如:

    Key: k8s.io/cluster-autoscaler/node-template/resources/$RESOURCE_NAME
Value: 5
Key: k8s.io/cluster-autoscaler/node-template/label/$LABEL_KEY
Value: $LABEL_VALUE
Key: k8s.io/cluster-autoscaler/node-template/taint/$TAINT_KEY
Value: NoSchedule

    注意:請記住,當縮放到 0 時,您的容量將返回 EC2,並可能在未來無法使用。

    其他參數¶

    有許多配置選項可用於調整 Cluster Autoscaler 的行為和性能。 完整的參數列表可在 GitHub 上找到。
    參數 描述 默認值
    scan-interval 重新評估叢集進行擴展或縮減的頻率 10 秒
    max-empty-bulk-delete 可以同時刪除的空節點的最大數量 10
    scale-down-delay-after-add 擴展後恢復縮減評估的延遲時間 10 分鐘
    scale-down-delay-after-delete 節點刪除後恢復縮減評估的延遲時間,默認為 scan-interval scan-interval
    scale-down-delay-after-failure 縮減失敗後恢復縮減評估的延遲時間 3 分鐘
    scale-down-unneeded-time 節點未使用的時間長度,超過後可以縮減 10 分鐘
    scale-down-unready-time 未就緒節點未使用的時間長度,超過後可以縮減 20 分鐘
    scale-down-utilization-threshold 節點利用率水平,定義為請求資源之和除以容量,低於此水平的節點可被視為縮減候選 0.5
    scale-down-non-empty-candidates-count 一次迭代中考慮作為縮減候選的非空節點的最大數量。較低的值意味著更好的 CA 響應能力,但可能會導致較慢的縮減延遲。較高的值可能會影響具有大型叢集(數百個節點)的 CA 性能。將其設置為非正值以關閉此啟發式方法 - CA 將不限制它考慮的節點數量。 30
    scale-down-candidates-pool-ratio 當前迭代中的一些候選項不再有效時,作為額外非空候選項考慮的節點比例。較低的值意味著更好的 CA 響應能力,但可能會導致較慢的縮減延遲。較高的值可能會影響具有大型叢集(數百個節點)的 CA 性能。將其設置為 1.0 以關閉此啟發式方法 - CA 將把所有節點作為額外候選項。 0.1
    scale-down-candidates-pool-min-count 當前迭代中的一些候選項不再有效時,作為額外非空候選項考慮的最小節點數。在計算額外候選項池大小時,我們採用 max(#nodes * scale-down-candidates-pool-ratio, scale-down-candidates-pool-min-count) 50
