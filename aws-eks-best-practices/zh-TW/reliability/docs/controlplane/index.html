    EKS 控制平面¶

    Amazon Elastic Kubernetes Service (EKS) 是一項受管理的 Kubernetes 服務,可讓您在 AWS 上輕鬆運行 Kubernetes,而無需安裝、操作和維護自己的 Kubernetes 控制平面或工作節點。它運行上游 Kubernetes,並經過 Kubernetes 一致性認證。此一致性確保 EKS 支持 Kubernetes API,就像您可以在 EC2 或內部部署安裝的開源社區版本一樣。在上游 Kubernetes 上運行的現有應用程序與 Amazon EKS 兼容。

    EKS 自動管理 Kubernetes 控制平面節點的可用性和可擴展性,並自動替換不健康的控制平面節點。

    EKS 架構¶

    EKS 架構旨在消除可能危及 Kubernetes 控制平面可用性和持久性的任何單點故障。

    由 EKS 管理的 Kubernetes 控制平面在 EKS 管理的 VPC 內運行。EKS 控制平面包括 Kubernetes API 伺服器節點和 etcd 集群。運行 API 伺服器、調度程序和 kube-controller-manager 等組件的 Kubernetes API 伺服器節點在自動縮放組中運行。EKS 在不同的可用區 (AZ) 中至少運行兩個 API 伺服器節點。同樣,為了持久性,etcd 伺服器節點也在跨越三個 AZ 的自動縮放組中運行。EKS 在每個 AZ 中運行一個 NAT 網關,API 伺服器和 etcd 伺服器在私有子網中運行。此架構可確保單個 AZ 中的事件不會影響 EKS 集群的可用性。

    創建新集群時,Amazon EKS 會創建一個高可用性的托管 Kubernetes API 伺服器端點,您可以使用它與集群進行通信(使用 kubectl 等工具)。托管端點使用 NLB 來平衡 Kubernetes API 伺服器的負載。EKS 還配置了兩個位於不同 AZ 的 ENI,以促進與您的工作節點的通信。

    您可以配置您的 Kubernetes 集群 API 伺服器是否可從公共互聯網(使用公共端點)或通過您的 VPC(使用 EKS 管理的 ENI)或兩者都可訪問。

    無論用戶和工作節點使用公共端點還是 EKS 管理的 ENI 連接到 API 伺服器,都有冗餘的連接路徑。

    建議¶

    監控控制平面指標¶

    監控 Kubernetes API 指標可以提供有關控制平面性能的洞見,並識別問題。不健康的控制平面可能會影響集群內運行的工作負載的可用性。例如,編寫不佳的控制器可能會使 API 伺服器過載,從而影響您應用程序的可用性。

    Kubernetes 在 /metrics 端點公開控制平面指標。

    您可以使用 kubectl 查看公開的指標:

    kubectl get --raw /metrics

    這些指標以 Prometheus 文本格式表示。

    您可以使用 Prometheus 收集和存儲這些指標。 2020 年 5 月,CloudWatch 增加了對在 CloudWatch Container Insights 中監控 Prometheus 指標的支持。因此,您也可以使用 Amazon CloudWatch 監控 EKS 控制平面。您可以使用教程:添加新的 Prometheus 採集目標:Prometheus KPI 伺服器指標收集指標並創建 CloudWatch 儀表板來監控集群的控制平面。

    您可以在這裡找到 Kubernetes API 伺服器指標。例如,apiserver_request_duration_seconds 可以指示 API 請求運行的時間長度。

    考慮監控以下控制平面指標:

    API 伺服器¶

    指標 描述
    apiserver_request_total 按動詞、乾運行值、組、版本、資源、範圍、組件和 HTTP 響應代碼分類的 apiserver 請求計數器。
    apiserver_request_duration_seconds* 每個動詞、乾運行值、組、版本、資源、子資源、範圍和組件的響應延遲分佈(以秒為單位)。
    apiserver_admission_controller_admission_duration_seconds 按名稱、操作和 API 資源以及類型(驗證或允許)識別的准入控制器延遲直方圖(以秒為單位)。
    apiserver_admission_webhook_rejection_count 准入 webhook 拒絕計數。按名稱、操作、拒絕代碼、類型(驗證或允許)、錯誤類型(調用 webhook 錯誤、apiserver 內部錯誤、無錯誤)識別。
    rest_client_request_duration_seconds 請求延遲(以秒為單位)。按動詞和 URL 細分。
    rest_client_requests_total HTTP 請求數,按狀態代碼、方法和主機分區。

    etcd¶

    指標 描述
    etcd_request_duration_seconds 每個操作和對象類型的 Etcd 請求延遲(以秒為單位)。
    etcd_db_total_size_in_bytes 或
    apiserver_storage_db_total_size_in_bytes(從 EKS v1.26 開始) 或
    apiserver_storage_size_bytes(從 EKS v1.28 開始)    		 Etcd 資料庫大小。

    考慮使用Kubernetes 監控概述儀表板來可視化和監控 Kubernetes API 伺服器請求和延遲以及 etcd 延遲指標。

    可以使用以下 Prometheus 查詢來監控 etcd 的當前大小。該查詢假設有一個名為 kube-apiserver 的作業來採集 API 指標端點的指標,並且 EKS 版本低於 v1.26。

    max(etcd_db_total_size_in_bytes{job="kube-apiserver"} / (8 * 1024 * 1024 * 1024))

    Attention

    當資料庫大小限制被超過時,etcd 會發出沒有空間的警報並停止接受進一步的寫入請求。換句話說,集群變為只讀,所有修改對象(如創建新 pod、擴展部署等)的請求都將被集群的 API 伺服器拒絕。

    集群身份驗證¶

    EKS 目前支持兩種身份驗證方式:持有者/服務帳戶令牌和 IAM 身份驗證,後者使用Webhook 令牌身份驗證。當用戶調用 Kubernetes API 時,Webhook 會將請求中包含的身份驗證令牌傳遞給 IAM。該令牌是一個 base 64 簽名的 URL,由 AWS 命令行界面(AWS CLI)生成。

    創建 EKS 集群的 IAM 用戶或角色會自動獲得對集群的完全訪問權限。您可以通過編輯 aws-auth configmap來管理對 EKS 集群的訪問。

    如果您錯誤配置了 aws-auth configmap 並失去了對集群的訪問權限,您仍然可以使用集群創建者的用戶或角色訪問您的 EKS 集群。

    在 AWS 區域中無法使用 IAM 服務的極少數情況下,您也可以使用 Kubernetes 服務帳戶的持有者令牌來管理集群。

    創建一個"超級管理員"帳戶,該帳戶被允許在集群中執行所有操作:

    kubectl -n kube-system create serviceaccount super-admin

    創建一個角色綁定,將 super-admin 角色賦予 super-admin:

    kubectl create clusterrolebinding super-admin-rb --clusterrole=cluster-admin --serviceaccount=kube-system:super-admin

    獲取服務帳戶的密鑰:

    SECRET_NAME=`kubectl -n kube-system get serviceaccount/super-admin -o jsonpath='{.secrets[0].name}'`

    獲取與密鑰關聯的令牌:

    TOKEN=`kubectl -n kube-system get secret $SECRET_NAME -o jsonpath='{.data.token}'| base64 --decode`

    將服務帳戶和令牌添加到 kubeconfig:

    kubectl config set-credentials super-admin --token=$TOKEN

    將 kubeconfig 中的當前上下文設置為使用 super-admin 帳戶:

    kubectl config set-context --current --user=super-admin

    最終的 kubeconfig 應該如下所示:

    apiVersion: v1
clusters:
- cluster:
    certificate-authority-data:<REDACTED>
    server: https://<CLUSTER>.gr7.us-west-2.eks.amazonaws.com
  name: arn:aws:eks:us-west-2:<account number>:cluster/<cluster name>
contexts:
- context:
    cluster: arn:aws:eks:us-west-2:<account number>:cluster/<cluster name>
    user: super-admin
  name: arn:aws:eks:us-west-2:<account number>:cluster/<cluster name>
current-context: arn:aws:eks:us-west-2:<account number>:cluster/<cluster name>
kind: Config
preferences: {}
users:
#- name: arn:aws:eks:us-west-2:<account number>:cluster/<cluster name>
#  user:
#    exec:
#      apiVersion: client.authentication.k8s.io/v1alpha1
#      args:
#      - --region
#      - us-west-2
#      - eks
#      - get-token
#      - --cluster-name
#      - <<cluster name>>
#      command: aws
#      env: null
- name: super-admin
  user:
    token: <<super-admin sa's secret>>

    准入 Webhook¶

    Kubernetes 有兩種類型的准入 Webhook:驗證准入 Webhook 和變更准入 Webhook。它們允許用戶擴展 Kubernetes API,並在對象被 API 接受之前對其進行驗證或變更。這些 Webhook 的配置不當可能會破壞 EKS 控制平面,阻塞集群關鍵操作。

    為了避免影響集群關鍵操作,請避免設置像以下這樣的"catch-all" Webhook:

    - name: "pod-policy.example.com"
  rules:
  - apiGroups:   ["*"]
    apiVersions: ["*"]
    operations:  ["*"]
    resources:   ["*"]
    scope: "*"

    或確保 Webhook 有一個 fail open 策略,超時時間短於 30 秒,以確保在 Webhook 不可用時不會損害集群關鍵工作負載。

    阻止帶有不安全 sysctls 的 Pod¶

    Sysctl 是一個 Linux 實用程序,允許用戶在運行時修改內核參數。這些內核參數控制操作系統行為的各個方面,如網絡、文件系統、虛擬內存和進程管理。

    Kubernetes 允許為 Pod 分配 sysctl 配置文件。Kubernetes 將 systcls 分為安全和不安全。安全的 sysctls 在容器或 Pod 中是命名空間化的,設置它們不會影響節點上的其他 Pod 或節點本身。相反,不安全的 sysctls 默認被禁用,因為它們可能會破壞其他 Pod 或使節點不穩定。

    由於不安全的 sysctls 默認被禁用,kubelet 將不會創建具有不安全 sysctl 配置文件的 Pod。如果您創建了這樣的 Pod,調度程序將反復將這些 Pod 分配到節點,而節點無法啟動它。這種無限循環最終會使集群控制平面承受壓力,使集群不穩定。

    考慮使用 OPA Gatekeeper 或 Kyverno 拒絕具有不安全 sysctls 的 Pod。

    處理集群升級¶

    自 2021 年 4 月以來,Kubernetes 的發布週期已從每年四次(每季度一次)變更為每年三次。新的次版本(如 1.21 或 1.22)大約每 15 週發布一次。從 Kubernetes 1.19 開始,每個次版本在首次發布後大約 12 個月內受支持。隨著 Kubernetes v1.28 的出現,控制平面和工作節點之間的兼容性偏差已從 n-2 擴展到 n-3 個次版本。要了解更多信息,請參見集群升級最佳實踐。

    運行大型集群¶

    EKS 主動監控控制平面實例的負載,並自動擴展它們以確保高性能。但是,在運行大型集群時,您應該考慮 Kubernetes 和 AWS 服務中的潛在性能問題和限制。

    • 擁有超過 1000 個服務的集群可能會在 kube-proxy 使用 iptables 模式時遇到網絡延遲,根據 ProjectCalico 團隊的測試。解決方案是切換到在 ipvs 模式下運行 kube-proxy。
    • 如果 CNI 需要為 Pod 請求 IP 地址,或者您需要頻繁創建新的 EC2 實例,您也可能遇到EC2 API 請求節流。您可以通過配置 CNI 緩存 IP 地址來減少對 EC2 API 的調用。您可以使用更大的 EC2 實例類型來減少 EC2 縮放事件。
