    EKS 資料平面¶

    為了運行高可用性和有彈性的應用程式,您需要一個高可用性和有彈性的資料平面。彈性的資料平面確保 Kubernetes 可以自動擴展和修復您的應用程式。一個有彈性的資料平面由兩個或更多工作節點組成,可以隨工作負載的變化而增長和收縮,並自動從故障中恢復。

    您有兩種選擇用於 EKS 的工作節點: EC2 實例和 Fargate。如果您選擇 EC2 實例,您可以自己管理工作節點,或使用 EKS 托管節點群組。您可以有一個由托管、自管理工作節點和 Fargate 混合的集群。

    EKS on Fargate 提供了最簡單的方式來獲得一個有彈性的資料平面。Fargate 在一個隔離的計算環境中運行每個 Pod。在 Fargate 上運行的每個 Pod 都有自己的工作節點。Fargate 會根據 Kubernetes 擴展 Pod 的情況自動擴展資料平面。您可以使用 水平 Pod 自動縮放器來擴展資料平面和您的工作負載。

    使用 Kubernetes Cluster Autoscaler、EC2 Auto Scaling 群組或社區項目如 Atlassian's Escalator是擴展 EC2 工作節點的首選方式。

    建議¶

    使用 EC2 Auto Scaling 群組創建工作節點¶

    使用 EC2 Auto Scaling 群組創建工作節點,而不是創建單個 EC2 實例並將它們加入到集群中,這是一個最佳實踐。Auto Scaling 群組將自動替換任何終止或失敗的節點,確保集群始終有足夠的容量來運行您的工作負載。

    使用 Kubernetes Cluster Autoscaler 擴展節點¶

    Cluster Autoscaler 會在集群沒有足夠資源運行 Pod 時調整資料平面的大小,並添加另一個工作節點。雖然 Cluster Autoscaler 是一個被動的過程,它會等待 Pod 處於 Pending 狀態是因為集群沒有足夠的容量。當發生這種情況時,它會添加 EC2 實例到集群。每當集群耗盡容量時,新的副本或新的 Pod 將處於 Pending 狀態,直到添加了工作節點。如果資料平面無法足夠快地擴展以滿足工作負載的需求,這種延遲可能會影響您應用程式的可靠性。如果一個工作節點一直被低利用,並且它的所有 Pod 都可以被調度到其他工作節點上,Cluster Autoscaler 會終止它。

    使用 Cluster Autoscaler 配置過度配置¶

    Cluster Autoscaler 在集群中的 Pod 已經處於 Pending 狀態時觸發資料平面的擴展。因此,從您的應用程式需要更多副本開始到實際獲得更多副本之間可能會有一段延遲。一種解決這種可能延遲的方法是添加更多所需的副本,增加應用程式的副本數量。

    Cluster Autoscaler 建議的另一種模式使用 pause Pod 和 Priority Preemption 功能。pause Pod 運行一個 pause 容器,顧名思義,它什麼也不做,只是作為集群中可以被其他 Pod 使用的計算容量的佔位符。因為它以 非常低的優先級 運行,當另一個 Pod 需要被創建而集群沒有可用容量時,pause Pod 會被驅逐出節點。Kubernetes 調度器注意到 pause Pod 被驅逐,並試圖重新調度它。但是,由於集群正在運行在容量限制內,pause Pod 仍然處於 Pending 狀態,這就觸發了 Cluster Autoscaler 添加節點。

    有一個 Helm 圖表可以安裝 cluster overprovisioner。

    將 Cluster Autoscaler 與多個 Auto Scaling 群組一起使用¶

    運行 Cluster Autoscaler 時,啟用 --node-group-auto-discovery 標誌。這樣做將允許 Cluster Autoscaler 找到包含特定定義標籤的所有自動縮放群組,並防止需要在清單中定義和維護每個自動縮放群組。

    將 Cluster Autoscaler 與本地存儲一起使用¶

    默認情況下,Cluster Autoscaler 不會縮減具有附加本地存儲的節點。將 --skip-nodes-with-local-storage 標誌設置為 false,以允許 Cluster Autoscaler 縮減這些節點。

    在多個可用區中分散工作節點和工作負載¶

    您可以通過在多個可用區中運行工作節點和 Pod 來保護您的工作負載免受單個可用區故障的影響。您可以使用創建節點的子網來控制工作節點創建的可用區。

    如果您使用 Kubernetes 1.18 及更高版本,推薦的跨可用區分散 Pod 的方法是使用 Topology Spread Constraints for Pods。

    下面的部署會在可能的情況下跨可用區分散 Pod,如果不可能,則仍然讓這些 Pod 運行:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-server
  template:
    metadata:
      labels:
        app: web-server
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          whenUnsatisfiable: ScheduleAnyway
          topologyKey: topology.kubernetes.io/zone
          labelSelector:
            matchLabels:
              app: web-server
      containers:
      - name: web-app
        image: nginx
        resources:
          requests:
            cpu: 1

    Note

    kube-scheduler 只能通過具有這些標籤的節點感知拓撲域。如果上述部署部署到只有單個區域中節點的集群,所有 Pod 都會調度到這些節點上,因為 kube-scheduler 不知道其他區域。這個問題將在 Kubernetes 1.24 中通過添加 MinDomainsInPodToplogySpread feature gate 得到解決,它允許指定 minDomains 屬性來告知調度器可用域的數量。

    Warning

    將 whenUnsatisfiable 設置為 DoNotSchedule 將導致 Pod 在無法滿足拓撲擴散約束時無法調度。只有在不運行 Pod 比違反拓撲擴散約束更好時,才應該設置它。

    在較舊版本的 Kubernetes 中,您可以使用 Pod 反親和性規則在多個可用區中調度 Pod。下面的清單告訴 Kubernetes 調度器首選在不同的可用區中調度 Pod。

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
  labels:
    app: web-server
spec:
  replicas: 4
  selector:
    matchLabels:
      app: web-server
  template:
    metadata:
      labels:
        app: web-server
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - web-server
              topologyKey: failure-domain.beta.kubernetes.io/zone
            weight: 100
      containers:
      - name: web-app
        image: nginx

    Warning

    不要要求 Pod 被調度到不同的可用區,否則部署中的 Pod 數量永遠不會超過可用區的數量。

    在使用 EBS 卷時確保每個可用區的容量¶

    如果您使用 Amazon EBS 提供持久卷,那麼您需要確保 Pod 和相關的 EBS 卷位於同一個可用區。目前,EBS 卷只在單個可用區內可用。Pod 無法訪問位於不同可用區的 EBS 支持的持久卷。Kubernetes 調度器知道工作節點所在的可用區。Kubernetes 將始終在與卷相同的可用區中調度需要 EBS 卷的 Pod。但是,如果在卷所在的可用區沒有可用的工作節點,則無法調度 Pod。

    為每個可用區創建一個 Auto Scaling 群組,並確保其具有足夠的容量,以確保集群始終有能力在與 EBS 卷相同的可用區中調度 Pod。此外,您應該在 Cluster Autoscaler 中啟用 --balance-similar-node-groups 功能。

    如果您運行一個使用 EBS 卷但沒有高可用性要求的應用程式,那麼您可以將應用程式的部署限制在單個可用區。在 EKS 中,工作節點自動添加 failure-domain.beta.kubernetes.io/zone 標籤,其中包含可用區的名稱。您可以通過運行 kubectl get nodes --show-labels 來查看附加到節點的標籤。有關內置節點標籤的更多信息,請參見此處。您可以使用節點選擇器在特定可用區調度 Pod。

    在下面的示例中,Pod 將只在 us-west-2c 可用區調度:

    apiVersion: v1
kind: Pod
metadata:
  name: single-az-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: failure-domain.beta.kubernetes.io/zone
            operator: In
            values:
            - us-west-2c
  containers:
  - name: single-az-container
    image: kubernetes/pause

    持久卷(由 EBS 支持)也自動標記有可用區的名稱;您可以通過運行 kubectl get pv -L topology.ebs.csi.aws.com/zone 來查看您的持久卷所屬的可用區。當創建 Pod 並聲明一個卷時,Kubernetes 將在與該卷相同的可用區中調度 Pod。

    考慮以下場景:您有一個具有一個節點群組的 EKS 集群。這個節點群組有三個工作節點,分佈在三個可用區。您有一個使用 EBS 支持的持久卷的應用程式。當您創建這個應用程式和相應的卷時,它的 Pod 會在第一個可用區中創建。然後,運行這個 Pod 的工作節點變得不健康並最終不可用。Cluster Autoscaler 將用一個新的工作節點替換不健康的節點;但是,因為自動縮放群組跨越三個可用區,新的工作節點可能會在第二個或第三個可用區啟動,而不是第一個可用區。由於 AZ 受限的 EBS 卷只存在於第一個可用區,但該區域沒有可用的工作節點,因此無法調度 Pod。因此,您應該在每個可用區創建一個節點群組,以便始終有足夠的容量可用於在其他可用區無法調度的 Pod。

    另一種選擇是使用 EFS,這可以簡化運行需要持久存儲的應用程式的集群自動縮放。客戶可以從該區域的所有可用區並發地訪問 EFS 文件系統。即使使用 EFS 支持的持久卷的 Pod 被終止並在不同的可用區重新調度,它也能掛載該卷。

    運行 node-problem-detector¶

    工作節點的故障可能會影響應用程式的可用性。node-problem-detector是一個 Kubernetes 插件,您可以在集群中安裝它來檢測工作節點問題。您可以使用 npd 的補救系統自動排空和終止節點。

    為系統和 Kubernetes 守護進程保留資源¶

    您可以通過為操作系統和 Kubernetes 守護進程保留計算容量來提高工作節點的穩定性。Pod - 特別是那些沒有聲明 limits 的 Pod - 可能會耗盡系統資源,使節點陷入操作系統進程和 Kubernetes 守護進程(kubelet、容器運行時等)與 Pod 競爭系統資源的情況。您可以使用 kubelet 標誌 --system-reserved 和 --kube-reserved 分別為系統進程(udev、sshd等)和 Kubernetes 守護進程保留資源。

    如果您使用 EKS 優化的 Linux AMI,CPU、內存和存儲默認情況下會為系統和 Kubernetes 守護進程保留。當基於此 AMI 啟動工作節點時,EC2 用戶數據會配置為觸發 bootstrap.sh 腳本。該腳本根據 EC2 實例上可用的 CPU 核心數和總內存計算 CPU 和內存保留。計算出的值被寫入位於 /etc/kubernetes/kubelet/kubelet-config.json 的 KubeletConfiguration 文件。

    如果您在節點上運行自定義守護進程,並且默認保留的 CPU 和內存量不足,您可能需要增加系統資源保留。

    eksctl 提供了最簡單的方式來自定義系統和 Kubernetes 守護進程的資源保留。

    實現 QoS¶

    對於關鍵應用程式,請考慮在 Pod 中為容器定義 requests=limits。這將確保如果另一個 Pod 請求資源,該容器不會被終止。

    為所有容器實現 CPU 和內存限制是一個最佳實踐,因為它可以防止容器無意中消耗系統資源,影響其他共同位置進程的可用性。

    為所有工作負載配置和調整資源請求/限制¶

    可以應用一些一般性的指導原則來調整工作負載的資源請求和限制:

    • 不要在 CPU 上指定資源限制。在沒有限制的情況下,請求充當容器獲得相對 CPU 時間的權重。這允許您的工作負載使用全部 CPU,而不受人工限制或饑餓的影響。

    • 對於非 CPU 資源,配置 requests=limits 提供了最可預測的行為。如果 requests!=limits,容器的 QOS也會從 Guaranteed 降低到 Burstable,這使它更有可能在節點壓力事件中被驅逐。

    • 對於非 CPU 資源,不要指定一個遠大於請求的限制。limits 相對於 requests 配置得越大,節點被過度承諾的可能性就越大,導致工作負載中斷的可能性也越高。

    • 正確調整請求對於使用節點自動縮放解決方案(如 Karpenter或 Cluster AutoScaler)非常重要。這些工具根據您的工作負載請求來確定要配置的節點數量和大小。如果您的請求太小而限制較大,您可能會發現您的工作負載被驅逐或 OOM 殺死,因為它們被緊密地打包在節點上。

    確定資源請求很困難,但是像 Vertical Pod Autoscaler 這樣的工具可以通過觀察運行時容器資源使用情況來"正確調整"請求。其他可能有用的確定請求大小的工具包括:

    • Goldilocks
    • Parca
    • Prodfiler
    • rsg

    為命名空間配置資源配額¶

    命名空間旨在用於擁有許多用戶分佈在多個團隊或項目的環境中。它們提供了名稱的作用域,是在多個團隊、項目和工作負載之間劃分集群資源的一種方式。您可以限制命名空間中的總體資源消耗。ResourceQuota對象可以限制可以在命名空間中按類型創建的對象的數量,以及該項目中的資源可以請求的總計算(CPU 和內存)資源。您可以限制給定命名空間中可以請求的存儲和/或計算(CPU 和內存)資源的總和。

    如果為命名空間啟用了資源配額,用戶必須為該命名空間中的每個容器指定請求或限制。

    考慮為每個命名空間配置配額。考慮使用 LimitRanges 自動將預先配置的限制應用於命名空間內的容器。

    限制命名空間內容器的資源使用¶

    資源配額有助於限制命名空間使用的資源量。LimitRange 對象可以幫助您實施容器可以請求的最小和最大資源。使用 LimitRange 您可以為容器設置默認請求和限制,這在您的組織中設置計算資源限制不是標準做法時很有幫助。顧名思義, LimitRange 可以在命名空間中強制執行每個 Pod 或容器的最小和最大計算資源使用。以及在命名空間中強制執行每個 PersistentVolumeClaim 的最小和最大存儲請求。

    考慮同時使用 LimitRange 和 ResourceQuota 來在容器和命名空間級別強制實施限制。設置這些限制將確保容器或命名空間不會侵犯集群中其他租戶使用的資源。

    CoreDNS¶

    CoreDNS 在 Kubernetes 中履行名稱解析和服務發現功能。它默認安裝在 EKS 集群上。為了互操作性,Kubernetes 服務 CoreDNS 仍然命名為 kube-dns。CoreDNS Pod 作為部署的一部分在 kube-system 命名空間中運行,在 EKS 中,默認情況下它運行兩個副本,並聲明了請求和限制。DNS 查詢被發送到 kube-system 命名空間中運行的 kube-dns 服務。

    建議¶

    監控 CoreDNS 指標¶

    CoreDNS 內置了對 Prometheus 的支持。您應該特別考慮監控 CoreDNS 延遲(coredns_dns_request_duration_seconds_sum,在 1.7.0 版本之前,該指標被稱為 core_dns_response_rcode_count_total)、錯誤(coredns_dns_responses_total, NXDOMAIN, SERVFAIL, FormErr)和 CoreDNS Pod 的內存消耗。

    出於故障排查目的,您可以使用 kubectl 查看 CoreDNS 日誌:

    for p in $(kubectl get pods -n kube-system -l k8s-app=kube-dns -o jsonpath='{.items[*].metadata.name}'); do kubectl logs $p -n kube-system; done

    使用 NodeLocal DNSCache¶

    您可以通過運行 NodeLocal DNSCache 來提高集群 DNS 性能。這個特性在集群節點上以 DaemonSet 的形式運行一個 DNS 緩存代理。所有 Pod 都使用節點上運行的 DNS 緩存代理進行名稱解析,而不是使用 kube-dns 服務。

    為 CoreDNS 配置 cluster-proportional-scaler¶

    提高集群 DNS 性能的另一種方法是根據節點數量和集群中的 CPU 核心自動水平擴展 CoreDNS 部署。Horizontal cluster-proportional-autoscaler 是一個容器,它根據可調度的數據平面的大小調整部署的副本數量。

    節點和節點中的總 CPU 核心是您可以用來擴展 CoreDNS 的兩個指標。您可以同時使用這兩個指標。如果您使用較大的節點,CoreDNS 縮放基於 CPU 核心數。而如果您使用較小的節點,CoreDNS 副本的數量取決於您的數據平面中的 CPU 核心。比例自動縮放器配置如下所示:

    linear: '{"coresPerReplica":256,"min":1,"nodesPerReplica":16}'

    選擇節點群組的 AMI¶

    EKS 提供了經過優化的 EC2 AMI,客戶用它們來創建自管理和托管節點群組。這些 AMI 在每個區域和每個受支持的 Kubernetes 版本中都有發佈。當發現任何 CVE 或錯誤時,EKS 會將這些 AMI 標記為已棄用。因此,建議在為節點群組選擇 AMI 時不要使用已棄用的 AMI。

    您可以使用以下命令使用 Ec2 describe-images API 過濾已棄用的 AMI:

    aws ec2 describe-images --image-id ami-0d551c4f633e7679c --no-include-deprecated

    您還可以通過檢查 describe-image 輸出是否包含 DeprecationTime 字段來識別已棄用的 AMI。例如:
