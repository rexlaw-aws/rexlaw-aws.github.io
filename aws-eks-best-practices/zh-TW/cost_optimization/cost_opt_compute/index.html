    成本優化 - 計算和自動擴展¶

    作為開發人員,您會對應用程式的資源需求(如 CPU 和記憶體)做出估計,但如果您不持續調整它們,它們可能會過時,這可能會增加成本並降低性能和可靠性。持續調整應用程式的資源需求比第一次就正確更重要。

    以下提到的最佳實踐將幫助您構建和操作成本意識的工作負載,同時實現業務目標,最大限度地降低成本,並使您的組織最大化投資回報。優化集群計算成本的重要性順序如下:

    1. 合適調整工作負載
    2. 減少未使用的容量
    3. 優化計算容量類型(如 Spot)和加速器(如 GPU)

    合適調整您的工作負載¶

    在大多數 EKS 集群中,大部分成本來自用於運行容器化工作負載的 EC2 實例。如果不了解您的工作負載需求,您將無法合適調整您的計算資源。這就是為什麼使用適當的請求和限制並根據需要調整這些設置非常重要。此外,依賴項(如實例大小和存儲選擇)可能會影響工作負載性能,從而對成本和可靠性產生各種意外後果。

    請求應與實際利用率保持一致。如果容器的請求過高,將會有未使用的容量,這是總集群成本的一個重要因素。每個 pod 中的容器(如應用程式和 sidecar)都應設置自己的請求和限制,以確保 pod 限制的總和盡可能準確。

    利用諸如 Goldilocks、KRR 和 Kubecost 等工具來估計容器的資源請求和限制。根據應用程式的性質、性能/成本要求和複雜性,您需要評估哪些指標最適合擴展,應用程式性能在什麼時候會下降(飽和點),以及如何相應地調整請求和限制。請參考應用程式合適調整以獲取更多指導。

    我們建議使用水平 Pod 自動縮放器 (HPA) 來控制應該運行的應用程式副本數量,使用垂直 Pod 自動縮放器 (VPA) 來調整每個副本所需的請求和限制,以及使用 Karpenter 或 Cluster Autoscaler 等節點自動縮放器來持續調整集群中的總節點數量。本文檔後面部分記錄了使用 Karpenter 和 Cluster Autoscaler 進行成本優化的技術。

    垂直 Pod 自動縮放器可以調整分配給容器的請求和限制,使工作負載以最佳方式運行。您應該以審核模式運行 VPA,這樣它就不會自動進行更改並重新啟動您的 pod。它將根據觀察到的指標提出更改建議。對於任何影響生產工作負載的更改,您都應該先在非生產環境中進行審查和測試,因為這些更改可能會影響應用程式的可靠性和性能。

    減少消耗¶

    節省資金的最佳方法是配置更少的資源。一種方法是根據當前需求調整工作負載。您應該從確保您的工作負載定義了它們的需求並動態擴展開始任何成本優化工作。這將需要從您的應用程式獲取指標,並設置配置,如 PodDisruptionBudgets 和 Pod Readiness Gates，以確保您的應用程式可以安全地動態擴展和縮減。

    水平 Pod 自動縮放器是一個靈活的工作負載自動縮放器,可以根據性能和可靠性要求調整所需的副本數量。它有一個靈活的模型,可以根據 CPU、記憶體或自定義指標(如隊列深度、連接到 pod 的數量等)來定義何時擴展和縮減。

    Kubernetes Metrics Server 可以啟用對內置指標(如 CPU 和記憶體使用率)的縮放,但如果您想根據其他指標(如 Amazon CloudWatch 或 SQS 隊列深度)進行縮放,您應該考慮使用事件驅動自動縮放項目,如 KEDA。請參考此博客文章了解如何將 KEDA 與 CloudWatch 指標一起使用。如果您不確定要監控和縮放哪些指標,請查看關於監控重要指標的最佳實踐。

    減少工作負載消耗會在集群中產生多餘的容量,並通過適當的自動縮放配置允許您自動縮減節點並減少總支出。我們建議您不要嘗試手動優化計算容量。Kubernetes 調度器和節點自動縮放器被設計來為您處理這個過程。

    減少未使用的容量¶

    在確定了應用程式的正確大小並減少了過多的請求後,您可以開始減少配置的計算容量。如果您已經按照上述部分採取了正確的措施,您應該能夠動態地做到這一點。AWS 中使用的兩個主要節點自動縮放器如下。

    Karpenter 和 Cluster Autoscaler¶

    Karpenter 和 Kubernetes Cluster Autoscaler 都將根據創建或刪除 pod 以及計算需求的變化來調整集群中的節點數量。它們的主要目標是相同的,但 Karpenter 採取了不同的節點管理配置和取消配置方法,這可以幫助降低成本並優化整個集群的使用情況。

    隨著集群規模的增長和工作負載種類的增加,預先配置節點組和實例變得越來越困難。就像調整工作負載請求一樣,設置一個初始基線並根據需要持續調整非常重要。

    Cluster Autoscaler 優先擴展¶

    Kubernetes Cluster Autoscaler 通過根據應用程式的擴展和縮減來擴展和縮減節點組(稱為節點組)來工作。如果您不動態縮放工作負載,Cluster Autoscaler 將無法幫助您節省資金。Cluster Autoscaler 要求集群管理員事先為工作負載創建節點組。節點組需要配置為使用具有相同"配置文件"的實例,即大致相同的 CPU 和記憶體。

    您可以有多個節點組,Cluster Autoscaler 可以配置為設置優先縮放級別,每個節點組可以包含不同大小的節點。節點組可以具有不同的容量類型,優先擴展器可用於首先縮放較便宜的組。

    以下是使用 ConfigMap 優先使用保留容量而不是按需實例的集群配置示例片段。您可以使用相同的技術來優先使用 Graviton 或 Spot 實例而不是其他類型。

    apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: my-cluster
managedNodeGroups:
  - name: managed-ondemand
    minSize: 1
    maxSize: 7
    instanceType: m5.xlarge
  - name: managed-reserved
    minSize: 2
    maxSize: 10
    instanceType: c5.2xlarge

    apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-autoscaler-priority-expander
  namespace: kube-system
data:
  priorities: |-
    10:
      - .*ondemand.*
    50:
      - .*reserved.*

    使用節點組可以幫助底層計算資源按預期執行,例如跨 AZ 分佈節點,但並非所有工作負載都有相同的要求或期望,最好讓應用程式明確聲明其要求。有關 Cluster Autoscaler 的更多信息,請參見最佳實踐部分。

    Descheduler¶

    Cluster Autoscaler 可以根據需要調度新 pod 或節點利用率下降來增加或減少集群容量。它不會在 pod 調度到節點後採取整體性的 pod 放置視角。如果您正在使用 Cluster Autoscaler,您還應該查看 Kubernetes descheduler,以避免浪費集群中的容量。

    如果您的集群中有 10 個節點,每個節點的利用率為 60%,則您沒有使用集群中 40% 的配置容量。使用 Cluster Autoscaler,您可以將每個節點的利用率閾值設置為 60%,但這只會在利用率低於 60% 時嘗試縮減單個節點。

    使用 descheduler,它可以在 pod 已經調度或節點已經添加到集群後查看集群容量和利用率。它試圖保持集群的總容量高於指定的閾值。它還可以根據節點污點或新加入集群的節點刪除 pod,以確保 pod 在其最佳計算環境中運行。請注意,descheduler 不會調度被驅逐 pod 的替換,而是依賴默認調度器來完成這項工作。

    Karpenter 整合¶

    Karpenter 採取了一種"無組"的節點管理方法。這種方法對於不同類型的工作負載更加靈活,需要集群管理員進行更少的前期配置。與預先定義組並根據工作負載需求擴展每個組不同,Karpenter 使用配置器和節點模板來定義可以創建的 EC2 實例類型以及實例的設置。

    裝箱是利用更多實例資源的做法,通過將更多工作負載打包到更少的、最佳大小的實例上。雖然這有助於通過只配置工作負載使用的資源來降低計算成本,但它也有一些權衡。它可能需要更長的時間來啟動新的工作負載,因為在大規模擴展事件期間需要向集群添加容量。在設置裝箱時,請考慮成本優化、性能和可用性之間的平衡。

    Karpenter 可以持續監控和裝箱以提高實例資源利用率並降低計算成本。Karpenter 還可以為您的工作負載選擇更具成本效益的工作節點。這可以通過在配置器中打開"整合"標誌來實現(下面的示例代碼片段)。下面的示例顯示了一個啟用整合的配置器。在撰寫本指南時,Karpenter 不會用更便宜的 Spot 實例替換正在運行的 Spot 實例。有關 Karpenter 整合的更多詳細信息,請參考此博客。

    apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: enable-binpacking
spec:
  consolidation:
    enabled: true

    對於可能不可中斷的工作負載,例如沒有檢查點的長時間批處理作業,請考慮使用 do-not-evict 註釋對 pod 進行註釋。通過選擇不驅逐 pod,您告訴 Karpenter 它不應該自願刪除包含此 pod 的節點。但是,如果在節點正在排空時添加了 do-not-evict pod,剩餘的 pod 仍將被驅逐,但該 pod 將阻止終止,直到將其刪除。在任何情況下,該節點都將被隔離,以防止將其他工作調度到該節點。以下是一個示例,展示如何設置註釋:

    apiVersion: v1
kind: Pod
metadata:
  name: label-demo
  labels:
    environment: production
  annotations:  
    "karpenter.sh/do-not-evict": "true"
spec:
  containers:
  - name: nginx
    image: nginx
    ports:
    - containerPort: 80

    通過調整 Cluster Autoscaler 參數來刪除利用不足的節點¶

    節點利用率定義為請求資源之和除以容量。默認情況下,scale-down-utilization-threshold 設置為 50%。此參數可以與 scale-down-unneeded-time 一起使用,後者確定節點在被縮減之前應該保持未使用的時間 - 默認為 10 分鐘。縮減節點上仍在運行的 pod 將由 kube-scheduler 調度到其他節點。調整這些設置可以幫助刪除利用不足的節點,但重要的是您要先測試這些值,以免過早強制集群縮減。

    您可以通過確保受昂貴驅逐保護的 pod 具有 Cluster Autoscaler 識別的標籤來防止縮減發生。為此,請確保具有昂貴驅逐 pod 具有註釋 cluster-autoscaler.kubernetes.io/safe-to-evict=false。以下是設置註釋的示例 yaml:

    apiVersion: v1
kind: Pod
metadata:
  name: label-demo
  labels:
    environment: production
  annotations:  
    "cluster-autoscaler.kubernetes.io/safe-to-evict": "false"
spec:
  containers:
  - name: nginx
    image: nginx
    ports:
    - containerPort: 80

    使用 Cluster Autoscaler 和 Karpenter 標記節點¶

    AWS 資源標籤用於組織您的資源,並在詳細級別跟踪您的 AWS 成本。它們與成本跟踪的 Kubernetes 標籤沒有直接關係。建議從 Kubernetes 資源標籤開始,並利用像 Kubecost 這樣的工具來獲取基於 Kubernetes 標籤在 pod、命名空間等上的基礎設施成本報告。

    工作節點需要有標籤,以在 AWS Cost Explorer 中顯示計費信息。對於 Cluster Autoscaler,使用啟動模板在托管節點組內標記您的工作節點。對於自管理節點組,使用 EC2 自動縮放組標記您的實例。對於 Karpenter 配置的實例,使用節點模板中的 spec.tags進行標記。

    多租戶集群¶

    在與不同團隊共享的集群上工作時,您可能無法看到在同一節點上運行的其他工作負載。雖然資源請求可以幫助隔離一些"吵雜鄰居"問題,如 CPU 共享,但它們可能無法隔離所有資源邊界,如磁盤 I/O 耗盡。不能隔離或限制工作負載使用的每個可消耗資源。消耗共享資源速度高於其他工作負載的工作負載應該通過節點污點和容忍進行隔離。這種工作負載的另一種高級技術是CPU 固定,它可以確保容器獲得專有 CPU 而不是共享 CPU。

    在節點級別隔離工作負載可能更昂貴,但您可能能夠調度最佳努力作業,或利用預留實例、Graviton 處理器或Spot等額外節省。

    共享集群也可能有集群級別的資源限制,如 IP 耗盡、Kubernetes 服務限制或 API 縮放請求。您應該查看可擴展性最佳實踐指南以確保您的集群避免這些限制。

    您可以在命名空間或 Karpenter 配置器級別隔離資源。資源配額提供了一種設置命名空間中工作負載可以使用的資源限制的方法。這可以作為一個很好的初步防護措施,但應該不斷評估,以確保它不會人為地限制工作負載的擴展。

    Karpenter 配置器可以設置集群中某些可消耗資源的限制(如 CPU、GPU),但您需要配置租戶應用程式以使用適當的配置器。這可以防止單個配置器在集群中創建太多節點,但應該不斷評估,以確保限制設置得不太低,從而阻止工作負載擴展。

    計劃的自動縮放¶

    您可能需要在週末和非工作時間縮減集群。這對於測試和非生產集群特別相關,在它們不使用時,您希望將其縮減到零。像 cluster-turndown 和 kube-downscaler 這樣的解決方案可以根據 cron 計劃將副本縮減到零。

    優化計算容量類型¶

    在優化集群中的總計算容量並利用裝箱之後,您應該查看您在集群中配置的計算類型以及您如何支付這些資源。AWS 有計算節省方案可以降低您的計算成本,我們將其歸類為以下容量類型:

    • Spot
    • 節省方案
    • 按需
    • Fargate

    每種容量類型都有不同的管理開銷、可用性和長期承諾權衡,您需要決定哪種適合您的環境。任何環境都不應該依賴單一容量類型,您可以在單個集群中混合使用多種運行類型,以優化特定工作負載要求和成本。

    Spot¶

    Spot 容量類型從可用區的備用容量中配置 EC2 實例。Spot 提供最大的折扣 - 高達 90% - 但這些實例可能會在需要它們的其他地方中斷。此外,可能並非始終有容量來配置新的 Spot 實例,並且現有的 Spot 實例可能會在2 分鐘中斷通知後被收回。如果您的應用程式有長時間的啟動或關閉過程,Spot 實例可能不是最佳選擇。

    Spot 計算應該使用各種實例類型,以降低沒有 Spot 容量可用的可能性。需要處理實例中斷,以安全地關閉節點。使用 Karpenter 或托管節點組配置的節點會自動支持實例中斷通知。如果您使用自管理節點,您將需要單獨運行節點終止處理程序來優雅地關閉 Spot 實例。

    在單個集群中平衡 Spot 和按需實例是可能的。使用 Karpenter,您可以創建加權配置器來實現不同容量類型的平衡。使用 Cluster Autoscaler,您可以創建混合節點組,包含 Spot 和按需或預留實例。

    以下是使用 Karpenter 優先使用 Spot 實例而不是按需實例的示例。在創建配置器時,您可以指定 Spot、按需或兩者(如下所示)。當您同時指定兩者,並且 pod 沒有明確指定是否需要使用 Spot 或按需時,Karpenter 會優先使用 Spot 實例,並使用價格容量優化分配策略進行配置。

    apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: spot-prioritized
spec:
  requirements:
    - key: "karpenter.sh/capacity-type" 
      operator: In
        values: ["spot", "on-demand"]

    節省方案、預留實例和 AWS EDP¶

    您可以通過使用計算節省方案來降低計算支出。節省方案提供了 1 年或 3 年計算使用承諾的折扣價格。使用範圍可以涵蓋 EKS 集群中的 EC2 實例,也可以涵蓋 Lambda 和 Fargate 等任何計算使用。使用節省方案,您可以在承諾期內降低成本,並仍然可以選擇任何 EC2 實例類型。

    計算節省方案可以將您的 EC2 成本降低高達 66%,而無需承諾使用哪些實例類型、系列或區域。節省會自動應用於您使用的實例。

    EC2 實例節省方案提供高達 72% 的計算節省,承諾在特定區域和 EC2 系列(例如 C 系列實例)中使用。您可以在該區域內的任何 AZ 之間轉移使用,使用該系列的任何一代實例(例如 c5 或 c6),並使用該系列中的任何大小的實例。折扣將自動應用於您的帳戶中符合節省方案標準的任何實例。

    預留實例類似於 EC2 實例節省方案,但它們還可以保證可用區或區域的容量,並相比按需實例降低成本 - 高達 72%。一旦計算出所需的預留容量,您就可以選擇保留它們的時間長度(1 年或 3 年)。折扣將自動應用於您在帳戶中運行這些 EC2 實例。

    客戶還可以選擇與 AWS 簽訂企業協議。企業協議使客戶能夠量身定制最適合其需求的協議。客戶可以根據 AWS EDP(企業折扣計劃)享受定價折扣。有關企業協議的更多信息,請聯繫您的 AWS 銷售代表。

    按需¶

    按需 EC2 實例具有不受中斷(相比 Spot)和無長期承諾(相比節省方案)的可用性。如果您正在尋求降低集群中的成本,您應該減少使用按需 EC2 實例。

    在優化您的工作負載需求後,您應該能夠計算出集群的最小和最大容量。隨時間這個數字可能會變化,但很少會下降。考慮將最小值以下的所有內容用於節省方案,將超出最小值的容量用於 Spot。其他可能不會持續使用或需要可用性的內容可以使用按需。

    如本節所述,減少資源消耗並最大限度地利用您配置的資源是節省資金的最佳方法。使用 Cluster Autoscaler,您可以通過 scale-down-utilization-threshold 設置刪除利用不足的節點。使用 Karpenter,建議啟用整合。

    要手動識別可與您的工作負載一起使用的 EC2 實例類型,您應該使用 ec2-instance-selector,它可以顯示每個區域可用的實例以及與 EKS 兼容的實例。以下是在 us-east-1 區域中具有 x86 處理器架構、4 Gb 內存和 2 個 vCPU 的實例的使用示例。

    ec2-instance-selector --memory 4 --vcpus 2 --cpu-architecture x86_64 \
  -r us-east-1 --service eks
c5.large
c5a.large
c5ad.large
c5d.large
c6a.large
c6i.large
t2.medium
t3.medium
t3a.medium

    對於非生產環境,您可以在夜間和週末等閒置時間自動縮減集群。Kubecost 項目 cluster-turndown 是一個可以根據設定的時間表自動縮減集群的控制器示例。

    Fargate 計算¶

    Fargate 計算是 EKS 集群的一種完全托管的計算選項。它通過在 Kubernetes 集群中每個節點上調度一個 pod 來提供 pod 隔離。它允許您根據工作負載的 CPU 和 RAM 需求調整計算節點的大小,以嚴格控制集群中的工作負載使用情況。

    Fargate 可以擴展到最小 .25 vCPU 和 0.5 GB 內存,最大 16 vCPU 和 120 GB 內存的工作負載。pod 大小變化有限制,您需要了解您的工作負載如何最適合 Fargate 配置。例如,如果您的工作負載需要 1 vCPU 和 0.5 GB 內存,Fargate pod 的最小大小將為 1 vCPU 和 2 GB 內存。

    雖然 Fargate 有許多好處,如無需管理 EC2 實例或操作系統,但由於每個部署的 pod 都被隔離為集群中的單獨節點,它可能需要比傳統 EC2 實例更多的計算容量。這需要對 Kubelet、日誌代理和您通常部署到節點的任何 DaemonSet 等進行更多複製。DaemonSet 在 Fargate 上不受支持,需要將其轉換為 pod "sidecar",並與應用程式一起運行。

    Fargate 無法從裝箱或 CPU 過度配置中獲益,因為工作負載的邊界是不可突發和不可共享的節點。Fargate 將為您節省 EC2 實例管理時間,這本身就有成本,但 CPU 和內存成本可能比其他 EC2 容量類型更高。Fargate pod 可以利用計算節省方案來降低按需成本。

    優化計算使用¶

    節省計算基礎設施成本的另一種方法是使用更高效的計算。這可以來自更高性能的通用計算,如Graviton 處理器,它的成本最高可降低 20%,能源效率提高 60% - 或特定工作負載的加速器,如 GPU 和 FPGA。您需要構建可以在 arm 架構上運行的容器,並為您的工作負載設置具有正確加速器的節點。

    EKS 可以運行具有混合架構(例如 amd64 和 arm64)的集群,如果您的容器編譯為多個架構,您可以通過在配置器中允許兩種架構來利用 Karpenter 上的 Graviton 處理器。但為了保持一致的性能,建議您將每個工作負載保持在單一計算架構上,只有在沒有其他可用容量時才使用不同的架構。

    配置器可以配置為具有多個架構,工作負載也可以在其工作負載規範中請求特定的架構。

    apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: default
spec:
  requirements:
  - key: "kubernetes.io/arch"
    operator: In
    values: ["arm64", "amd64"]

    使用 Cluster Autoscaler,您將需要為 Graviton 實例創建一個節點組,並在您的工作負載上設置節點容忍來利用新的容量。

    GPU 和 FPGA 可以大大提高工作負載的性能,但工作負載需要優化以使用加速器。許多機器學習和人工智能類型的工作負載可以使用 GPU 進行計算,並且可以將實例添加到集群中並使用資源請求掛載到工作負載中。

    spec:
  template:
    spec:
    - containers:
      ...
      resources:
          limits:
            nvidia.com/gpu: "1"

    一些 GPU 硬件可以跨多個工作負載共享,因此可以配置和使用單個 GPU。要了解如何配置工作負載 GPU 共享,請參考虛擬 GPU 設備插件以獲取更多信息。您也可以參考以下博客:

    • 使用 NVIDIA 的多實例 GPU (MIG) 在 Amazon EKS 上最大化 GPU 利用率:為增強性能運行每個 GPU 上的更多 pod
