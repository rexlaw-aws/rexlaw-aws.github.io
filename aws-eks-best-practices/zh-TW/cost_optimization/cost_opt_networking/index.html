    網絡

    成本優化 - 網路

    為了實現高可用性(HA)和容錯能力,構建高可用系統是最佳實踐。實際上,這意味著將您的工作負載和底層基礎設施分散在給定AWS區域的多個可用區(AZ)中。確保您的Amazon EKS環境具有這些特性將提高系統的整體可靠性。與此同時,您的EKS環境可能還由各種構件(即VPC)、組件(即ELB)和集成(即ECR和其他容器註冊表)組成。

    高可用系統和其他特定用例組件的組合可能會對數據傳輸和處理產生重大影響。這反過來會影響由於數據傳輸和處理而產生的成本。

    以下詳述的做法將幫助您設計和優化您的EKS環境,以實現不同領域和用例的成本效益。

    Pod到Pod通信¶

    根據您的設置,Pod之間的網絡通信和數據傳輸可能會對運行Amazon EKS工作負載的整體成本產生重大影響。本節將介紹不同的概念和方法,以緩解與Pod間通信相關的成本,同時考慮高可用(HA)架構、應用程序性能和恢復能力。

    限制流量到可用區¶

    跨區域流量(在AZ之間分配的流量)頻繁可能會對您的網絡相關成本產生重大影響。以下是一些策略,介紹如何控制EKS集群中Pod之間的跨區流量。

    使用拓撲感知路由(以前稱為拓撲感知提示)

    當使用拓撲感知路由時,需要了解服務、EndpointSlice和kube-proxy如何協同工作來路由流量。如上圖所示,服務是接收流量的穩定網絡抽象層。創建服務時,會創建多個EndpointSlice。每個EndpointSlice都有一個端點列表,其中包含一些Pod地址以及它們運行所在的節點和任何其他拓撲信息。kube-proxy是一個守護進程,在集群中的每個節點上運行,也承擔內部路由的作用,但它是基於它從創建的EndpointSlice中獲取的內容來執行的。

    當在Kubernetes服務上拓撲感知路由被啟用並實現時,EndpointSlice控制器將根據集群分散的不同區域,將端點按比例分配。對於這些端點,EndpointSlice控制器還將設置一個_提示_,用於指示端點應該為哪個區域提供流量服務。kube-proxy將根據這些_提示_將流量從一個區域路由到一個端點。

    下圖顯示了EndpointSlice如何根據提示進行組織,以便kube-proxy可以知道應該根據它們的區域起源將流量路由到哪個目的地。如果沒有提示,就不會有這種分配或組織,流量將被代理到不同區域的目的地,而不管它的來源是哪裡。

    在某些情況下,EndpointSlice控制器可能會為不同的區域應用_提示_,這意味著端點最終可能會為不同區域的流量提供服務。這樣做的原因是試圖在不同區域的端點之間保持流量的均衡分佈。

    以下是啟用服務的_拓撲感知路由_的代碼片段。

    apiVersion: v1
kind: Service
metadata:
  name: orders-service
  namespace: ecommerce
    annotations:
      service.kubernetes.io/topology-mode: Auto
spec:
  selector:
    app: orders
  type: ClusterIP
  ports:
  - protocol: TCP
    port: 3003
    targetPort: 3003

    下面的截圖顯示了EndpointSlice控制器成功將提示應用於在AZ eu-west-1a中運行的Pod副本的端點。

    使用自動縮放器:在特定AZ配置節點

    我們強烈建議在多個AZ中運行您的工作負載,以提高應用程序的可靠性,特別是在AZ發生問題時。如果您願意以犧牲可靠性為代價來降低網絡相關成本,您可以將節點限制在單個AZ。

    要在同一AZ中運行所有Pod,可以在同一AZ的子網中配置工作節點,或使用Cluster Autoscaler (CA)配置工作節點組。對於Karpenter,使用"topology.kubernetes.io/zone"並指定要在其中創建工作節點的AZ。例如,下面的Karpenter配置器片段在us-west-2a AZ中配置節點。

    Karpenter

    apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
name: single-az
spec:
  requirements:
  - key: "topology.kubernetes.io/zone"
    operator: In
    values: ["us-west-2a"]

    Cluster Autoscaler (CA)

    apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: my-ca-cluster
  region: us-east-1
  version: "1.21"
availabilityZones:
- us-east-1a
managedNodeGroups:
- name: managed-nodes
  labels:
    role: managed-nodes
  instanceType: t3.medium
  minSize: 1
  maxSize: 10
  desiredCapacity: 1
...

    使用Pod分配和節點親和性

    另外,如果您有在多個AZ中運行的工作節點,每個節點都會有一個_topology.kubernetes.io/zone_標籤,其值為其AZ(如us-west-2a或us-west-2b)。您可以利用nodeSelector或nodeAffinity將Pod調度到單個AZ中的節點。例如,以下清單文件將Pod調度到在AZ us-west-2a中運行的節點內。

    apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    env: test
spec:
  nodeSelector:
    topology.kubernetes.io/zone: us-west-2a
  containers:
  - name: nginx
    image: nginx 
    imagePullPolicy: IfNotPresent

    限制流量到節點¶

    在某些情況下,將流量限制在區域級別是不夠的。除了降低成本外,您可能還需要降低某些應用程序之間頻繁內部通信的網絡延遲。為了實現最佳網絡性能並降低成本,您需要一種方法來將流量限制到特定節點。例如,微服務A應該始終與位於節點1上的微服務B通信,即使在高可用(HA)設置中也是如此。微服務A在節點1上與微服務B在節點2上通信可能會對此類應用程序的所需性能產生負面影響,特別是如果節點2位於另一個AZ的話。

    使用服務內部流量策略

    為了限制Pod網絡流量到節點,您可以利用_服務內部流量策略_。默認情況下,發送到工作負載服務的流量將隨機分佈到不同生成的端點。因此,在HA架構中,這意味著來自微服務A的流量可能會流向微服務B在不同AZ中的任何副本。但是,通過將服務的內部流量策略設置為Local,流量將被限制到從該流量發起的節點上的端點。此策略規定專門使用節點本地端點。由此推論,該工作負載的網絡流量相關成本將低於集群範圍內分佈的情況。同時,延遲也會更低,使您的應用程序更加高效。

    以下是設置服務_內部流量策略_的代碼片段。

    apiVersion: v1
kind: Service
metadata:
  name: orders-service
  namespace: ecommerce
spec:
  selector:
    app: orders
  type: ClusterIP
  ports:
  - protocol: TCP
    port: 3003
    targetPort: 3003
  internalTrafficPolicy: Local

    為了避免由於流量下降而導致應用程序出現意外行為,您應該考慮以下方法:

    • 為每個通信Pod運行足夠的副本
    • 使用拓撲傳播約束實現相對均勻的Pod分佈
    • 對於通信Pod的共同位置,使用pod-affinity規則

    在這個例子中,您有微服務A的2個副本和微服務B的3個副本。如果微服務A的副本分散在節點1和2之間,而微服務B的3個副本都在節點3上,那麼由於Local內部流量策略,它們將無法通信,因為沒有可用的節點本地端點,流量將被丟棄。

    如果微服務B確實有2個副本在節點1和2上,那麼它們之間就會有通信。但是,您仍然會有一個微服務B的孤立副本,沒有任何對等副本可以通信。

    使用服務內部流量策略和拓撲傳播約束

    將_內部流量策略_與_拓撲傳播約束_結合使用可以確保您為不同節點上的通信微服務提供正確數量的副本。

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: express-test
spec:
  replicas: 6
  selector:
    matchLabels:
      app: express-test
  template:
    metadata:
      labels:
        app: express-test
        tier: backend
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: "topology.kubernetes.io/zone"
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: express-test

    使用服務內部流量策略和Pod親和性規則

    另一種方法是在使用服務內部流量策略時利用Pod親和性規則。使用Pod親和性,您可以影響調度程序,因為它們頻繁通信而將某些Pod共同放置。通過對某些Pod應用嚴格的調度約束(requiredDuringSchedulingIgnoredDuringExecution),這將在調度器將Pod放置在節點上時為Pod共同位置提供更好的結果。

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: graphql
  namespace: ecommerce
  labels:
    app.kubernetes.io/version: "0.1.6"
    ...
    spec:
      serviceAccountName: graphql-service-account
      affinity:
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - orders
            topologyKey: "kubernetes.io/hostname"

    負載均衡器到Pod通信¶

    EKS工作負載通常由負載均衡器前置,該負載均衡器將流量分配到EKS集群中的相關Pod。您的架構可能包括內部和/或外部面向的負載均衡器。根據您的架構和網絡流量配置,負載均衡器和Pod之間的通信可能會佔據大部分數據傳輸費用。

    您可以使用AWS Load Balancer Controller自動管理ELB資源(ALB和NLB)的創建。您在這種設置中產生的數據傳輸費用將取決於網絡流量的路徑。AWS Load Balancer Controller支持兩種網絡流量模式,instance mode_和_ip mode。

    使用_instance mode_時,將在EKS集群中的每個節點上打開一個NodePort。然後,負載均衡器將在節點之間均勻代理流量。如果節點上有目標Pod運行,則不會產生任何數據傳輸費用。但是,如果目標Pod位於不同於接收流量的NodePort所在節點的另一個AZ,則會有一個額外的網絡跳躍,從kube-proxy到目標Pod。在這種情況下,將產生跨AZ數據傳輸費用。由於流量在節點之間的均勻分佈,很可能會有額外的數據傳輸費用,與從kube-proxy到相關目標Pod的跨區網絡流量跳躍有關。

    下圖描述了流量從負載均衡器流向NodePort,然後從kube-proxy流向位於不同AZ的另一個節點上的目標Pod的網絡路徑。這是_instance mode_設置的一個示例。

    當使用_ip mode_時,流量直接從負載均衡器代理到目標Pod。因此,這種方法不涉及_任何數據傳輸費用_。

    建議您將負載均衡器設置為_ip traffic mode_以降低數據傳輸費用。對於這種設置,還重要的是確保您的負載均衡器部署在VPC中的所有子網上。

    從容器註冊表的數據傳輸¶

    Amazon ECR¶

    向Amazon ECR私有註冊表的數據傳輸是免費的。同區域內的數據傳輸不收費,但向互聯網和跨區域的數據傳輸將按互聯網數據傳輸費率在傳輸的兩端收費。

    您應該利用ECR內置的鏡像複製功能將相關的容器鏡像複制到與您的工作負載相同的區域。這樣,複制就只收取一次費用,而所有同區域(區域內)的鏡像拉取都是免費的。

    您可以通過_使用Interface VPC Endpoints連接到同區域的ECR存儲庫_,進一步降低從ECR拉取鏡像(數據傳出)的相關成本。連接到ECR的公共AWS端點(通過NAT網關和Internet網關)的替代方法將產生更高的數據處理和傳輸成本。下一節將更詳細地介紹如何降低工作負載與AWS服務之間的數據傳輸成本。

    如果您運行的工作負載有特別大的鏡像,您可以構建自己的自定義Amazon Machine Images (AMIs),預先緩存容器鏡像。這可以減少初始鏡像拉取時間和從容器註冊表到EKS工作節點的潛在數據傳輸成本。

    向互聯網和AWS服務的數據傳輸¶

    將Kubernetes工作負載與其他AWS服務或第三方工具和平台集成是一種常見做法,通過互聯網進行。用於路由流量到達相關目的地的底層網絡基礎設施可能會影響數據傳輸過程中產生的成本。

    使用NAT網關¶

    NAT網關是執行網絡地址轉換(NAT)的網絡組件。下圖描述了EKS集群中的Pod與其他AWS服務(Amazon ECR、DynamoDB和S3)以及第三方平台進行通信。在這個例子中,Pod在不同AZ的私有子網中運行。為了發送和接收來自互聯網的流量,在一個AZ的公有子網中部署了一個NAT網關,允許任何具有私有IP地址的資源共享單個公有IP地址來訪問互聯網。這個NAT網關又與Internet網關組件通信,允許數據包被發送到最終目的地。

    為了最小化這種情況下的數據傳輸成本,您應該在每個AZ部署一個NAT網關。這樣,路由到互聯網的流量將通過同一AZ的NAT網關,避免了跨AZ的數據傳輸。但是,儘管您將節省跨AZ數據傳輸的成本,這種設置的影響是您的架構中會產生額外的NAT網關成本。

    這種推薦的方法如下圖所示。

    使用VPC端點¶

    為了進一步降低這種架構中的成本,您應該使用VPC端點在您的工作負載和AWS服務之間建立連接。VPC端點允許您在不通過互聯網傳輸數據/網絡數據包的情況下,從VPC內部訪問AWS服務。所有流量都是內部的,並保持在AWS網絡內。有兩種類型的VPC端點:Interface VPC端點(許多AWS服務支持)和Gateway VPC端點(只有S3和DynamoDB支持)。

    Gateway VPC端點

    Gateway VPC端點沒有每小時或數據傳輸費用。使用Gateway VPC端點時,需要注意它們不能跨VPC邊界擴展。它們不能用於VPC對等、VPN網絡或通過Direct Connect。

    Interface VPC端點

    VPC端點有每小時收費,並且根據AWS服務的不同,可能會或可能不會有與基礎ENI相關的額外數據處理費用。為了降低與Interface VPC端點相關的跨AZ數據傳輸成本,您可以在每個AZ創建一個VPC端點。即使它們指向同一個AWS服務,您也可以在同一VPC中創建多個VPC端點。

    跨VPC的數據傳輸¶

    在某些情況下,您可能有位於不同VPC(在同一AWS區域內)的工作負載需要相互通信。這可以通過允許流量通過連接到各自VPC的Internet網關來跨越公共互聯網來實現。這種通信可以通過在公有子網中部署EC2實例、NAT網關或NAT實例來實現。但是,包含這些組件的設置將產生處理/傳輸VPC內外數據的費用。如果來自不同VPC的流量跨AZ移動,則在數據傳輸過程中會產生額外的費用。下圖描述了使用NAT網關和Internet網關在不同VPC之間建立通信的設置。

    VPC對等連接¶

    為了降低此類用例的成本,您可以使用VPC對等。使用VPC對等連接,在同一AZ內的網絡流量不會產生任何數據傳輸費用。如果流量跨AZ,將產生費用。儘管如此,VPC對等方法仍然是在同一AWS區域內跨VPC工作負載通信的推薦方法,因為它可以降低成本。但是,需要注意的是,VPC對等主要適用於1:1 VPC連接,因為它不支持傳遞網絡。

    下圖是通過VPC對等連接進行工作負載通信的高級表示。

    傳遞網絡連接¶

    正如前一節所指出的,VPC對等連接不支持傳遞網絡連接。如果您想要連接3個或更多具有傳遞網絡要求的VPC,那麼您應該使用Transit Gateway(TGW)。這將使您能夠克服VPC對等或多個VPC對等連接之間的操作開銷的限制。您按小時收費並根據發送到TGW的數據收費。通過TGW流動的跨AZ流量沒有目的地成本。

    使用服務網格¶

    服務網格提供強大的網絡功能,可用於降低EKS集群環境中的網絡相關成本。但是,您應該仔細考慮採用服務網格會為您的環境帶來的操作任務和複雜性。

    限制流量到可用區¶

    使用Istio的區域加權分佈

    Istio可以在路由發生後應用網絡策略。這是使用目標規則完成的,例如區域加權分佈。使用這個功能,您可以控制流量到某個目的地的權重(以百分比表示),這取決於它的起源。這個流量的源頭可以是外部(或面向公眾)的負載均衡器,也可以是集群內部的Pod。當所有Pod端點可用時,區域將根據加權輪詢負載均衡算法進行選擇。如果某些端點不健康或不可用,區域權重將自動調整以反映可用端點的變化。

    在實現區域加權分佈之前,您應該首先了解您的網絡流量模式以及目標規則策略可能對您的應用程序行為產生的影響。因此,建議您使用諸如AWS X-Ray或Jaeger等工具建立分佈式跟蹤機制。

    上述Istio目標規則也可以應用於管理從負載均衡器到EKS集群中Pod的流量。區域加權分佈規則可以應用於接收來自高可用負載均衡器(特別是Ingress網關)流量的服務。這些規則允許您根據流量的區域起源來控制流量的分佈 - 在這種情況下是負載均衡器。如果配置正確,與隨機或均勻分佈到不同AZ中Pod副本的負載均衡器相比,產生的跨區流量將更少。

    以下是Istio中目標規則資源的代碼示例。如下所示,此資源為來自3個不同AZ的傳入流量指定了加權配置。這些配置聲明,來自給定AZ的大部分傳入流量(在本例中為70%)應該代理到與其起源相同的AZ中的目標。

    apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: express-test-dr
spec:
  host: express-test.default.svc.cluster.local
  trafficPolicy:
    loadBalancer:                        
      localityLbSetting:
        distribute:
        - from: eu-west-1/eu-west-1a/    
          to:
            "eu-west-1/eu-west-1a/*": 70 
            "eu-west-1/eu-west-1b/*": 20
            "eu-west-1/eu-west-1c/*": 10
        - from: eu-west-1/eu-west-1b/*    
          to:
            "eu-west-1/eu-west-1a/*": 20 
            "eu-west-1/eu-west-1b/*": 70
            "eu-west-1/eu-west-1c/*": 10
        - from: eu-west-1/eu-west-1c/*    
          to:
            "eu-west-1/eu-west-1a/*": 20 
            "eu-west-1/eu-west-1b/*": 10
            "eu-west-1/eu-west-1c/*": 70**
    connectionPool:
      http:
        http2MaxRequests: 10
        maxRequestsPerConnection: 10
    outlierDetection:
      consecutiveGatewayErrors: 1
      interval: 1m
      baseEjectionTime: 30s

    下圖描述了一個場景,其中有一個高可用負載均衡器位於_eu-west-1_區域,並應用了區域加權分佈。對於此圖,目標規則配置為將60%來自_eu-west-1a_的流量發送到同一AZ中的Pod,而40%來自_eu-west-1a_的流量應該發送到eu-west-1b中的Pod。

    限制流量到可用區和節點¶

    使用服務內部流量策略與Istio

    為了緩解與_外部_傳入流量和Pod之間_內部_流量相關的網絡成本,您可以結合Istio的目標規則和Kubernetes服務_內部流量策略_。將Istio目標規則與服務內部流量策略結合的方式將主要取決於3個因素:

    • 微服務的角色
    • 跨微服務的網絡流量模式
    • 微服務應如何部署在Kubernetes集群拓撲中

    下圖顯示了嵌套請求的網絡流量,以及上述策略如何控制流量。

    1. 最終用戶向APP A發出請求,APP A又向APP C發出嵌套請求。這個請求首先發送到一個高可用的負載均衡器,該負載均衡器在AZ 1和AZ 2都有實例,如上圖所示。
    2. 外部傳入的請求然後由Istio虛擬服務路由到正確的目的地。
    3. 請求被路由後,Istio目標規則控制了基於流量起源(AZ 1或AZ 2)的流量分佈。
    4. 流量然後進入APP A的服務,並被代理到相應的Pod端點。如圖所示,80%的傳入流量被發送到AZ 1中的Pod端點,20%的傳入流量被發送到AZ 2。
    5. APP A然後向APP C發出內部請求。APP C的服務啟用了內部流量策略(internalTrafficPolicy``: Local)。
    6. 來自APP A(在NODE 1上)到APP C的內部請求成功,因為APP C有可用的節點本地端點。
    7. 來自APP A(在NODE 3)到*APP C*的內部請求失敗,因為APP C**沒有可用的_節點本地端點_。如圖所示,APP C在NODE 3上沒有副本。

    以下截圖是從一個實時示例中捕獲的。第一組截圖演示了成功的外部請求到graphql以及從graphql到共同位於節點ip-10-0-0-151.af-south-1.compute.internal上的orders副本的成功嵌套請求。

    使用Istio,您可以驗證和導出代理感知的任何上游集群和端點的統計信息。這可以提供網絡流量的圖景,以及工作負載中各服務的分佈份額。繼續使用相同的示例,可以使用以下命令獲取graphql代理感知的orders端點:

    kubectl exec -it deploy/graphql -n ecommerce -c istio-proxy -- curl localhost:15000/clusters | grep orders 

    ...

orders-service.ecommerce.svc.cluster.local::10.0.1.33:3003::**rq_success::119**

    在這種情況下,graphql代理只知道與它共享節點的orders端點。如果您從orders服務中刪除internalTrafficPolicy: Local設置,並重新運行類似的命令,則結果將返回分佈在不同節點上的所有副本端點。此外,通過檢查相應端點的rq_total,您會注意到網絡分佈相對均勻。因此,如果這些端點與運行在不同AZ中的上游服務相關聯,那麼這種跨區域的網絡分佈將導致更高的成本。

    如上一節所提到的,您可以使用pod-affinity來共同放置頻繁通信的Pod。

    ...
spec:
...
  template:
    metadata:
      labels:
        app: graphql
        role: api
        workload: ecommerce
    spec:
      affinity:
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - orders
            topologyKey: "kubernetes.io/hostname"
      nodeSelector:
        managedBy: karpenter
        billing-team: ecommerce
...

    當graphql和orders副本不在同一節點(ip-10-0-0-151.af-south-1.compute.internal)上共存時,對graphql的第一個請求成功,如Postman截圖中的200 response code所示,而從graphql到orders的嵌套請求則失敗,返回503 response code。
