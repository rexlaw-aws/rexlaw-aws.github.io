    成本優化 - 儲存¶

    概述¶

    有些情況下,您可能需要運行需要短期或長期保存數據的應用程序。對於此類用例,可以定義卷並由 Pod 掛載,以便其容器可以利用不同的存儲機制。Kubernetes 支持不同類型的卷用於短暫和持久性存儲。存儲的選擇主要取決於應用程序的需求。對於每種方法,都有成本影響,以下詳述的做法將有助於您在 EKS 環境中需要某種形式存儲的工作負載實現成本效率。

    短暫卷¶

    短暫卷適用於需要短暫本地卷但不需要在重啟後持久保存數據的應用程序。例子包括對臨時空間、緩存和只讀輸入數據(如配置數據和密鑰)的需求。您可以在此處找到更多關於 Kubernetes 短暫卷的詳細信息。大多數短暫卷(如 emptyDir、configMap、downwardAPI、secret、hostpath)都由本地附加的可寫設備(通常為根磁盤)或 RAM 支持,因此選擇最具成本效益和性能的主機卷非常重要。

    使用 EBS 卷¶

    我們建議從gp3作為主機根卷開始。它是 Amazon EBS 提供的最新通用 SSD 卷,每 GB 價格也比 gp2 卷低(最高 20%)。

    使用 Amazon EC2 實例存儲¶

    Amazon EC2 實例存儲為您的 EC2 實例提供臨時塊級存儲。EC2 實例存儲提供的存儲通過物理連接到主機的磁盤進行訪問。與 Amazon EBS 不同,您只能在啟動實例時附加實例存儲卷,這些卷僅在實例的生命週期內存在。它們無法分離並重新附加到其他實例。您可以在此處了解更多關於 Amazon EC2 實例存儲的信息。實例存儲卷沒有任何額外費用。這使它們(實例存儲卷)比帶有大型 EBS 卷的通用 EC2 實例更具成本效益。

    要在 Kubernetes 中使用本地存儲卷,您應該使用 Amazon EC2 用戶數據對磁盤進行分區、配置和格式化,以便可以將其掛載為 pod 規格中的HostPath。或者,您可以利用本地持久卷靜態供應商來簡化本地存儲管理。本地持久卷靜態供應商允許您通過標準的 Kubernetes PersistentVolumeClaim (PVC) 接口訪問本地實例存儲卷。此外,它將配置包含節點關聯信息的 PersistentVolumes (PVs),以將 Pod 調度到正確的節點。儘管它使用 Kubernetes PersistentVolumes,但 EC2 實例存儲卷本質上是短暫的。寫入短暫磁盤的數據只在實例的生命週期內可用。當實例終止時,數據也會消失。更多詳情請參考此博客。

    請注意,使用 Amazon EC2 實例存儲卷時,總 IOPS 限制與主機共享,並將 Pod 綁定到特定主機。在採用 Amazon EC2 實例存儲卷之前,您應該仔細審查您的工作負載要求。

    持久卷¶

    Kubernetes 通常與運行無狀態應用程序相關聯。但是,也有一些情況下您可能想要運行需要保存持久數據或信息的微服務,從一個請求到下一個請求。數據庫是此類用例的常見示例。但是,Pod 及其內部的容器或進程都是短暫的。為了在 Pod 生命週期之外持久保存數據,您可以使用 PV 來定義對特定位置存儲的訪問,該位置獨立於 Pod。PV 相關的成本高度依賴於所使用的存儲類型以及應用程序的使用方式。

    在 Amazon EKS 上,有多種支持 Kubernetes PV 的存儲選項,列在此處。以下介紹的存儲選項包括 Amazon EBS、Amazon EFS、Amazon FSx for Lustre 和 Amazon FSx for NetApp ONTAP。

    Amazon Elastic Block Store (EBS) 卷¶

    Amazon EBS 卷可以作為 Kubernetes PV 使用,提供塊級存儲卷。它們非常適合依賴隨機讀寫和吞吐量密集型應用程序的數據庫。Amazon Elastic Block Store Container Storage Interface (CSI) 驅動程序允許 Amazon EKS 集群管理 Amazon EBS 卷的生命週期作為持久卷。Container Storage Interface 使 Kubernetes 與存儲系統之間的交互更加容易。當在您的 EKS 集群中部署 CSI 驅動程序時,您可以通過本地 Kubernetes 存儲資源(如 Persistent Volumes (PVs)、Persistent Volume Claims (PVCs) 和 Storage Classes (SCs))訪問其功能。此鏈接提供了如何使用 Amazon EBS CSI 驅動程序與 Amazon EBS 卷交互的實際示例。

    選擇合適的卷¶

    我們建議使用最新一代的塊存儲(gp3),因為它在價格和性能之間提供了合適的平衡。它還允許您獨立擴展卷 IOPS 和吞吐量,而無需配置額外的塊存儲容量。如果您目前正在使用 gp2 卷,我們強烈建議您遷移到 gp3 卷。此博客解釋了如何在 Amazon EKS 集群上從 gp2 遷移到 gp3。

    當您有需要更高性能且需要的卷大小超過單個 gp3 卷的支持時,您應該考慮使用io2 block express。這種存儲類型非常適合您最大、最 I/O 密集和最關鍵的部署,如 SAP HANA 或其他具有低延遲要求的大型數據庫。請注意,實例的 EBS 性能受實例性能限制的約束,因此並非所有實例都支持 io2 block express 卷。您可以在此文檔中查看支持的實例類型和其他注意事項。

    單個 gp3 卷最多可支持 16,000 個最大 IOPS、1,000 MiB/s 最大吞吐量和 16TiB 最大容量。最新一代的 Provisioned IOPS SSD 卷最多可提供 256,000 IOPS、4,000 MiB/s 吞吐量和 64TiB 容量。

    在這些選項中,您應該根據應用程序的需求最好地調整存儲性能和成本。

    持續監控和優化¶

    了解您應用程序的基線性能並監控所選卷是很重要的,以檢查它是否滿足您的要求/期望,或者是否過度配置(例如,配置的 IOPS 未完全利用)。

    不要一開始就分配一個大卷,而是可以逐步增加卷的大小,因為您累積了更多數據。您可以使用 Amazon Elastic Block Store CSI 驅動程序(aws-ebs-csi-driver)中的卷調整功能動態調整卷大小。請注意,您只能增加 EBS 卷大小。

    為了識別和刪除任何遺留的 EBS 卷,您可以使用 AWS 可信顧問的成本優化類別。此功能可幫助您識別未附加的卷或在一段時間內寫入活動很低的卷。還有一個名為 Popeye 的雲原生開源只讀工具,可掃描實時 Kubernetes 集群並報告已部署資源和配置的潛在問題。例如,它可以掃描未使用的 PV 和 PVC,並檢查它們是否已綁定或是否存在任何卷掛載錯誤。

    有關深入監控的詳細信息,請參考EKS 成本優化可觀察性指南。

    您可以考慮的另一個選項是 AWS Compute Optimizer Amazon EBS 卷建議。此工具可自動識別最佳卷配置和所需的正確性能級別。例如,它可用於基於過去 14 天的最大利用率對配置的 IOPS、卷大小和 EBS 卷類型進行優化設置。它還量化了其建議帶來的潛在每月成本節省。您可以查看此博客了解更多詳情。

    備份保留策略¶

    您可以通過拍攝時間點快照來備份 Amazon EBS 卷上的數據。Amazon EBS CSI 驅動程序支持卷快照。您可以按照此處概述的步驟學習如何創建快照並恢復 EBS PV。

    後續快照是增量備份,這意味著只保存設備上自最近一次快照以來發生變化的塊。這可以最大限度地減少創建快照所需的時間,並通過不重複數據來節省存儲成本。但是,在沒有適當保留策略的情況下,增加舊的 EBS 快照數量會在大規模運營時造成意外成本。如果您直接通過 AWS API 備份 Amazon EBS 卷,可以利用 Amazon Data Lifecycle Manager(DLM)提供的自動化、基於策略的生命週期管理解決方案來管理 Amazon Elastic Block Store (EBS) 快照和 EBS 支持的 Amazon Machine Images (AMIs)。控制台使自動化 EBS 快照和 AMI 的創建、保留和刪除變得更加容易。

    Note

    目前無法通過 Amazon EBS CSI 驅動程序使用 Amazon DLM。

    在 Kubernetes 環境中,您可以利用名為 Velero 的開源工具來備份您的 EBS 持久卷。您可以在安排作業時設置 TTL 標誌來過期備份。Velero 提供了此指南作為示例。

    Amazon Elastic File System (EFS)¶

    Amazon Elastic File System (EFS)是一個無服務器、完全彈性的文件系統,允許您使用標準文件系統接口和文件系統語義共享文件數據,適用於各種工作負載和應用程序,如 Wordpress 和 Drupal、JIRA 和 Git 等開發工具,以及 Jupyter 等共享筆記本系統以及家目錄。

    Amazon EFS 的主要優勢之一是它可以被多個節點和多個可用區域中的多個容器掛載。另一個優勢是您只需為使用的存儲付費。EFS 文件系統將自動隨著您添加和刪除文件而增長和縮小,消除了容量規劃的需要。

    要在 Kubernetes 中使用 Amazon EFS,您需要使用 Amazon Elastic File System Container Storage Interface (CSI) 驅動程序 aws-efs-csi-driver。目前,該驅動程序可以動態創建訪問點。但是,必須先配置 Amazon EFS 文件系統,並將其作為輸入提供給 Kubernetes 存儲類參數。

    選擇合適的 EFS 存儲類¶

    Amazon EFS 提供四種存儲類。

    兩種標準存儲類:

    • Amazon EFS 標準
    • Amazon EFS 標準-不頻繁訪問(EFS 標準-IA)

    兩種單區域存儲類:

    • Amazon EFS 單區域
    • Amazon EFS 單區域-不頻繁訪問(EFS 單區域-IA)

    不頻繁訪問(IA)存儲類針對未每天訪問的文件進行了成本優化。通過 Amazon EFS 生命週期管理,您可以將在生命週期策略(7、14、30、60 或 90 天)期間未訪問的文件移動到 IA 存儲類,這可將存儲成本降低高達 92% 與 EFS 標準和 EFS 單區域存儲類相比。

    通過 EFS Intelligent-Tiering,生命週期管理會監控您文件系統的訪問模式,並自動將文件移動到最佳存儲類。

    Note

    aws-efs-csi-driver 目前無法控制更改存儲類、生命週期管理或 Intelligent-Tiering。這些應該手動在 AWS 控制台或通過 EFS API 設置。

    Note

    aws-efs-csi-driver 與基於 Windows 的容器映像不兼容。

    Note

    啟用 vol-metrics-opt-in (發出卷指標)時存在已知的內存問題,因為 DiskUsage 函數消耗的內存量與您的文件系統大小成正比。目前,我們建議在大型文件系統上禁用 --vol-metrics-opt-in 選項,以避免消耗過多內存。有關更多詳細信息,請參見 github 問題鏈接。

    Amazon FSx for Lustre¶

    Lustre 是一個高性能的並行文件系統,通常用於需要高達數百 GB/s 的吞吐量和亞毫秒級延遲的工作負載。它用於機器學習培訓、金融建模、HPC 和視頻處理等場景。Amazon FSx for Lustre提供了完全托管的共享存儲,具有可擴展性和性能,並與 Amazon S3 無縫集成。

    您可以使用 Amazon EKS 或您在 AWS 上自管理的 Kubernetes 集群上的 FSx for Lustre CSI 驅動程序來使用 FSx for Lustre 支持的 Kubernetes 持久存儲卷。請參閱Amazon EKS 文檔了解更多詳細信息和示例。

    連接到 Amazon S3¶

    建議將位於 Amazon S3 上的高耐用性長期數據存儲庫與您的 FSx for Lustre 文件系統相連接。連接後,大型數據集將按需從 Amazon S3 延遲加載到 FSx for Lustre 文件系統。您還可以將分析結果寫回 S3,然後刪除您的[Lustre]文件系統。

    選擇合適的部署和存儲選項¶

    FSx for Lustre 提供不同的部署選項。第一個選項稱為臨時,它不複制數據,而第二個選項稱為持久,顧名思義,它持久保存數據。

    第一個選項(臨時)可用於降低短期數據處理的成本。持久部署選項旨在用於長期存儲,它會自動在 AWS 可用區域內複制數據。它還支持 SSD 和 HDD 存儲。

    您可以在 FSx for lustre 文件系統的 Kubernetes StorageClass 的參數中配置所需的部署類型。此鏈接提供了示例模板。

    Note

    對於對延遲敏感的工作負載或需要最高 IOPS/吞吐量的工作負載,您應該選擇 SSD 存儲。對於不對延遲敏感但注重吞吐量的工作負載,您應該選擇 HDD 存儲。

    啟用數據壓縮¶

    您可以通過在參數中指定"LZ4"作為數據壓縮類型來啟用文件系統的數據壓縮。啟用後,所有新寫入的文件將在寫入磁盤之前自動壓縮到 FSx for Lustre 上,在讀取時解壓縮。LZ4 數據壓縮算法是無損的,因此可以完全從壓縮數據中重建原始數據。

    您可以在 FSx for lustre 文件系統的 Kubernetes StorageClass 的參數中配置 LZ4 作為數據壓縮類型。當值設置為 NONE(默認)時,壓縮將被禁用。此鏈接提供了示例模板。

    Note

    Amazon FSx for Lustre 與基於 Windows 的容器映像不兼容。

    Amazon FSx for NetApp ONTAP¶

    Amazon FSx for NetApp ONTAP是一個完全托管的共享存儲,構建在 NetApp 的 ONTAP 文件系統之上。FSx for ONTAP 提供功能豐富、快速和靈活的共享文件存儲,可從在 AWS 或本地運行的 Linux、Windows 和 macOS 計算實例廣泛訪問。

    Amazon FSx for NetApp ONTAP 支持兩層存儲:1/主存儲層和2/容量池層。

    主存儲層是一個配置的高性能 SSD 基礎層,用於活動、對延遲敏感的數據。完全彈性的容量池層是針對不常訪問的數據進行成本優化的,隨著數據被分層到其中而自動擴展,並提供虛乎無限的 PB 級容量。您可以在容量池存儲上啟用數據壓縮和重複數據刪除,進一步減少存儲容量的使用。NetApp 的本機策略驅動 FabricPool 功能持續監控數據訪問模式,自動雙向地在存儲層之間轉移數據,以優化性能和成本。

    NetApp 的 Astra Trident 提供了使用 CSI 驅動程序的動態存儲編排,允許 Amazon EKS 集群管理由 Amazon FSx for NetApp ONTAP 文件系統支持的 PV 的生命週期。要開始使用,請參見 Astra Trident 文檔中的將 Astra Trident 與 Amazon FSx for NetApp ONTAP 一起使用。

    其他注意事項¶

    最小化容器映像大小¶

    部署容器後,容器映像作為多個層被緩存在主機上。通過減小映像大小,可以減少主機上所需的存儲空間。

    通過使用精簡的基礎映像,如scratch映像或distroless容器映像(僅包含您的應用程序及其運行時依賴項),您不僅可以降低存儲成本,還可以獲得其他好處,如減少攻擊面積和更短的映像拉取時間。

    您還應該考慮使用開源工具,如 Slim.ai,它提供了一種簡單、安全的方式來創建最小映像。

    多層包、工具、應用程序依賴項和庫很容易使容器映像臃腫。通過使用多階段構建,您可以選擇性地將工件從一個階段複製到另一個階段,從最終映像中排除不必要的所有內容。您可以在此處查看更多映像構建最佳實踐。

    另一個要考慮的是保留緩存映像的時間。當磁盤使用量達到一定程度時,您可能需要清理陳舊的映像。這樣做將有助於確保主機操作有足夠的空間。默認情況下,kubelet每五分鐘對未使用的映像進行一次垃圾收集,每分鐘對未使用的容器進行一次垃圾收集。

    要配置未使用的容器和映像垃圾收集的選項,請使用配置文件調整 kubelet,並使用 KubeletConfiguration資源類型更改與垃圾收集相關的參數。
