    Windows 網路¶

    Windows 容器網路概述¶

    Windows 容器與 Linux 容器有根本性的不同。Linux 容器使用 Linux 構造，如命名空間、聯合檔案系統和 cgroups。在 Windows 上，這些構造由 Host Compute Service (HCS) 抽象化。HCS 充當位於 Windows 容器實現之上的 API 層。Windows 容器還利用 Host Network Service (HNS) 來定義節點上的網路拓撲。

    從網路的角度來看，HCS 和 HNS 使 Windows 容器的功能類似於虛擬機。例如，每個容器都有一個連接到 Hyper-V 虛擬交換機 (vSwitch) 的虛擬網路介面卡 (vNIC)，如上圖所示。

    IP 位址管理¶

    Amazon EKS 中的節點使用其彈性網路介面 (ENI) 連接到 AWS VPC 網路。目前僅支援每個 Windows 工作節點一個 ENI。Windows 節點的 IP 位址管理由 VPC Resource Controller 在控制平面上執行。有關 Windows 節點 IP 位址管理工作流程的更多詳細信息，請參見 此處。

    Windows 工作節點可支援的 Pod 數量由節點的大小和可用 IPv4 位址的數量決定。您可以按以下方式計算節點上可用的 IPv4 位址: - 默認情況下，只會將次要 IPv4 位址分配給 ENI。在這種情況下: 

    可供 Pod 使用的 IPv4 位址總數 = 主介面支持的 IPv4 位址數 - 1
    我們從總數中減去 1，因為 1 個 IPv4 位址將用作 ENI 的主要地址，因此無法分配給 Pod。

    • 如果集群已通過啟用 前綴委派功能 進行了高 Pod 密度配置,則- 
      可供 Pod 使用的 IPv4 位址總數 = (主介面支持的 IPv4 位址數 - 1) * 16
      在這裡,VPC Resource Controller 將分配 /28 前綴,而不是分配次要 IPv4 位址,因此可用 IPv4 位址的總數將提高 16 倍。

    使用上述公式,我們可以根據 m5.large 實例計算 Windows 工作節點的最大 Pod 數: - 默認情況下,在次要 IP 模式下- 

    每個 ENI 有 10 個次要 IPv4 位址 - 1 = 9 個可用 IPv4 位址
    - 使用 前綴委派- 
    (每個 ENI 有 10 個次要 IPv4 位址 - 1) * 16 = 144 個可用 IPv4 位址

    有關實例類型可支持的 IP 位址數量的更多信息,請參見 每個實例類型每個網路介面的 IP 位址。

    另一個關鍵考慮因素是網路流量的流向。在 Windows 上,如果節點上有超過 100 個服務,就存在端口耗盡的風險。出現這種情況時,節點將開始拋出以下錯誤消息:

    "Policy creation failed: hcnCreateLoadBalancer failed in Win32: The specified port already exists."

    為了解決這個問題,我們利用了直接服務返回 (DSR)。DSR 是非對稱網路負載分配的一種實現。換句話說,請求和響應流量使用不同的網路路徑。這個功能加快了 Pod 之間的通信,並降低了端口耗盡的風險。因此,我們建議在 Windows 節點上啟用 DSR。

    在 Windows Server SAC EKS 優化 AMI 中,DSR 默認已啟用。對於 Windows Server 2019 LTSC EKS 優化 AMI,您需要在實例配置期間使用以下腳本和將 amiFamily 設置為 WindowsServer2019FullContainer 或 WindowsServer2019CoreContainer 來啟用它。請參見 eksctl 自定義 AMI 以獲取更多信息。

    nodeGroups:
- name: windows-ng
  instanceType: c5.xlarge
  minSize: 1
  volumeSize: 50
  amiFamily: WindowsServer2019CoreContainer
  ssh:
    allow: false
    為了在 Windows Server 2019 及更高版本中利用 DSR,您需要在實例啟動期間指定以下 kube-proxy 標誌。您可以通過調整與 自管理節點組啟動模板 關聯的 userdata 腳本來實現。

    <powershell>
[string]$EKSBinDir = "$env:ProgramFiles\Amazon\EKS"
[string]$EKSBootstrapScriptName = 'Start-EKSBootstrap.ps1'
[string]$EKSBootstrapScriptFile = "$EKSBinDir\$EKSBootstrapScriptName"
(Get-Content $EKSBootstrapScriptFile).replace('"--proxy-mode=kernelspace",', '"--proxy-mode=kernelspace", "--feature-gates WinDSR=true", "--enable-dsr",') | Set-Content $EKSBootstrapScriptFile 
& $EKSBootstrapScriptFile -EKSClusterName "eks-windows" -APIServerEndpoint "https://<REPLACE-EKS-CLUSTER-CONFIG-API-SERVER>" -Base64ClusterCA "<REPLACE-EKSCLUSTER-CONFIG-DETAILS-CA>" -DNSClusterIP "172.20.0.10" -KubeletExtraArgs "--node-labels=alpha.eksctl.io/cluster-name=eks-windows,alpha.eksctl.io/nodegroup-name=windows-ng-ltsc2019 --register-with-taints=" 3>&1 4>&1 5>&1 6>&1
</powershell>

    可以按照 Microsoft Networking 博客 和 Windows Containers on AWS Lab 中的說明來驗證 DSR 的啟用情況。

    容器網路介面 (CNI) 選項¶

    AWSVPC CNI 是 Windows 和 Linux 工作節點的事實標準 CNI 插件。雖然 AWSVPC CNI 滿足了許多客戶的需求,但仍可能出現需要考慮使用覆蓋網絡等替代方案以避免 IP 耗盡的情況。在這些情況下,可以使用 Calico CNI 代替 AWSVPC CNI。Project Calico 是由 Tigera 開發的開源軟件,其中包括一個適用於 EKS 的 CNI。有關在 EKS 上安裝 Calico CNI 的說明,請參見 Project Calico EKS 安裝頁面。

    網絡策略¶

    將 Kubernetes 集群從默認的 Pod 之間開放通信模式更改為基於網絡策略的訪問限制被認為是最佳實踐。開源 Project Calico 對網絡策略提供了強大的支持,適用於 Linux 和 Windows 節點。此功能是獨立的,不依賴於使用 Calico CNI。因此,我們建議安裝 Calico 並將其用於網絡策略管理。

    有關在 EKS 上安裝 Calico 的說明,請參見 在 Amazon EKS 上安裝 Calico。

    此外,Amazon EKS 最佳實踐指南-網絡部分中提供的建議同樣適用於具有 Windows 工作節點的 EKS 集群,但是一些功能,如"Pod 的安全組"目前不支持 Windows。
