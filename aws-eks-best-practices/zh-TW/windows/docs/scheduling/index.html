    在異構工作負載上運行¶¶

    Kubernetes 支持異構集群,您可以在同一個集群中混合使用 Linux 和 Windows 節點。在該集群中,您可以混合使用在 Linux 上運行的 Pod 和在 Windows 上運行的 Pod。您甚至可以在同一個集群中運行多個版本的 Windows。但是,在做出這個決定時,需要考慮幾個因素(如下所述)。

    將 POD 分配到最佳實踐節點¶

    為了將 Linux 和 Windows 工作負載保持在各自的 OS 特定節點上,您需要使用節點選擇器和污點/容忍的某種組合。在異構環境中調度工作負載的主要目標是避免破壞現有 Linux 工作負載的兼容性。

    確保 OS 特定的工作負載落在適當的容器主機上¶

    用戶可以使用 nodeSelectors 確保 Windows 容器可以被調度到適當的主機上。所有當前的 Kubernetes 節點都有以下默認標籤:

    kubernetes.io/os = [windows|linux]
kubernetes.io/arch = [amd64|arm64|...]

    如果 Pod 規範不包含像 "kubernetes.io/os": windows 的 nodeSelector,則 Pod 可能會被調度到任何主機,無論是 Windows 還是 Linux。這可能會出現問題,因為 Windows 容器只能在 Windows 上運行,Linux 容器只能在 Linux 上運行。

    在企業環境中,通常會有大量預先存在的 Linux 容器部署,以及現成的配置生態系統,如 Helm charts。在這些情況下,您可能會hesitate 對部署的 nodeSelectors 進行更改。替代方案是使用污點。

    例如: --register-with-taints='os=windows:NoSchedule'

    如果您正在使用 EKS,eksctl 提供了通過 clusterConfig 應用污點的方法:

    NodeGroups:
  - name: windows-ng
    amiFamily: WindowsServer2022FullContainer
    ...
    labels:
      nodeclass: windows2022
    taints:
      os: "windows:NoSchedule"

    為所有 Windows 節點添加污點,調度程序將不會在這些節點上調度 pod,除非它們容忍污點。Pod 清單示例:

    nodeSelector:
    kubernetes.io/os: windows
tolerations:
    - key: "os"
      operator: "Equal"
      value: "windows"
      effect: "NoSchedule"

    處理同一集群中的多個 Windows 版本¶

    每個 pod 使用的 Windows 容器基礎映像必須與節點的內核版本相匹配。如果您想在同一個集群中使用多個 Windows Server 版本,那麼您應該設置其他節點標籤、nodeSelectors 或利用名為 windows-build 的標籤。

    Kubernetes 1.17 自動添加了一個新的標籤 node.kubernetes.io/windows-build 來簡化在同一個集群中管理多個 Windows 版本。如果您正在運行較舊的版本,則建議手動將此標籤添加到 Windows 節點。

    此標籤反映了需要匹配兼容性的 Windows 主要版本、次版本和內部版本號。以下是當前每個 Windows Server 版本使用的值。

    需要注意的是,Windows Server 正在轉向長期服務通道(LTSC)作為主要發行渠道。Windows Server 半年度通道(SAC)已於2022年8月9日退役。將不會有未來的 SAC 版本的 Windows Server。

    產品名稱 版本號
    Server full 2022 LTSC 10.0.20348
    Server core 2019 LTSC 10.0.17763

    kubectl get nodes -o wide

    下面的示例將額外的 nodeSelector 應用於 pod 清單,以便在運行不同 Windows 節點組 OS 版本時匹配正確的 Windows 版本。

    nodeSelector:
    kubernetes.io/os: windows
    node.kubernetes.io/windows-build: '10.0.20348'
tolerations:
    - key: "os"
    operator: "Equal"
    value: "windows"
    effect: "NoSchedule"

    使用 RuntimeClass 簡化 Pod 清單中的 NodeSelector 和容忍¶

    您還可以利用 RuntimeClass 來簡化使用污點和容忍的過程。這可以通過創建一個 RuntimeClass 對象來實現,該對象用於封裝這些污點和容忍。

    運行以下清單創建一個 RuntimeClass:

    apiVersion: node.k8s.io/v1beta1
kind: RuntimeClass
metadata:
  name: windows-2022
handler: 'docker'
scheduling:
  nodeSelector:
    kubernetes.io/os: 'windows'
    kubernetes.io/arch: 'amd64'
    node.kubernetes.io/windows-build: '10.0.20348'
  tolerations:
  - effect: NoSchedule
    key: os
    operator: Equal
    value: "windows"

    創建 Runtimeclass 後,將其作為 Spec 分配給 Pod 清單:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: iis-2022
  labels:
    app: iis-2022
spec:
  replicas: 1
  template:
    metadata:
      name: iis-2022
      labels:
        app: iis-2022
    spec:
      runtimeClassName: windows-2022
      containers:
      - name: iis

    托管節點組支持¶

    為了幫助客戶以更簡化的方式運行其 Windows 應用程序,AWS 於2022年12月15日推出了對 Amazon EKS 托管節點組(MNG)對 Windows 容器的支持。為了幫助運營團隊保持一致,Windows MNGs使用與Linux MNGs相同的工作流程和工具啟用。支持 Windows Server 2019 和 2022 的完整和核心 AMI (Amazon Machine Image) 系列版本。

    以下 AMI 系列支持托管節點組(MNG):

    AMI 系列
    WINDOWS_CORE_2019_x86_64
    WINDOWS_FULL_2019_x86_64
    WINDOWS_CORE_2022_x86_64
    WINDOWS_FULL_2022_x86_64
