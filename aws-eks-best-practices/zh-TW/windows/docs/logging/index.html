    記錄¶

    容器化應用程式通常將應用程式日誌定向到 STDOUT。容器運行時捕獲這些日誌並對其執行某些操作 - 通常是寫入文件。這些文件存儲位置取決於容器運行時和配置。

    Windows pod 的一個基本區別是它們不會生成 STDOUT。您可以運行 LogMonitor 來檢索 ETW (Windows 事件跟蹤)、Windows 事件日誌和其他應用程式特定日誌從正在運行的 Windows 容器,並將格式化的日誌輸出管道到 STDOUT。然後可以使用 fluent-bit 或 fluentd 將這些日誌流式傳輸到您的目標位置,例如 Amazon CloudWatch。

    日誌收集機制從 Kubernetes pod 檢索 STDOUT/STDERR 日誌。DaemonSet 是收集容器日誌的常見方式。它使您能夠獨立於應用程式管理日誌路由/過濾/豐富。可以使用 fluentd DaemonSet 將這些日誌和任何其他應用程式生成的日誌流式傳輸到所需的日誌聚合器。

    有關從 Windows 工作負載到 CloudWatch 的日誌流的更多詳細信息,請參閱此處。

    日誌記錄建議¶

    在 Kubernetes 中運行 Windows 工作負載時,一般日誌記錄最佳實踐並沒有什麼不同。

    • 始終記錄結構化日誌條目(JSON/SYSLOG),這樣處理日誌條目會更容易,因為對於這種結構化格式有許多預先編寫的解析器。
    • 集中日誌 - 可以使用專門的日誌容器來收集和轉發來自所有容器的日誌消息到目標位置。
    • 保持日誌詳細程度較低,除非進行調試。詳細程度會給日誌基礎設施帶來很大壓力,重要事件可能會被淹沒在噪音中。

    • 始終記錄應用程式信息以及事務/請求 ID以實現可追溯性。Kubernetes 對象不攜帶應用程式名稱,因此例如 pod 名稱 windows-twryrqyw 在調試日誌時可能沒有任何意義。這有助於使用您匯總的日誌進行應用程式的可追溯性和故障排除。

      如何生成這些事務/相關 ID 取決於編程構造。但一個非常常見的模式是使用日誌記錄方面/攔截器,它可以使用 MDC(映射的診斷上下文)為每個傳入的請求注入一個唯一的事務/相關 ID,如下所示:

    import org.slf4j.MDC;
import java.util.UUID;
Class LoggingAspect { //interceptor

    @Before(value = "execution(* *.*(..))")
    func before(...) {
        transactionId = generateTransactionId();
        MDC.put(CORRELATION_ID, transactionId);
    }

    func generateTransactionId() {
        return UUID.randomUUID().toString();
    }
}
