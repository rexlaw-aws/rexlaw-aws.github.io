    避免 OOM 錯誤¶

    Windows 沒有像 Linux 一樣的 out-of-memory 進程終止器。Windows 始終將所有使用者模式的內存分配視為虛擬的,並且分頁文件是強制性的。其淨效果是,Windows 不會像 Linux 那樣遇到內存耗盡的情況。進程將改為分頁到磁盤,而不會受到 out of memory (OOM) 終止的影響。如果內存過度配置且所有物理內存都耗盡,則分頁可能會降低性能。

    保留系統和 kubelet 內存¶

    與 Linux 不同,--kubelet-reserve 在 Windows 上不會捕獲 Kubernetes 系統守護程序(如 kubelet、容器運行時等)的資源預留,--system-reserve 也不會捕獲 OS 系統守護程序(如 sshd、udev 等)的資源預留。

    但是,您可以結合這些標誌來管理 NodeAllocatable,以減少節點上的容量,並使用 Pod 清單中的 memory resource limit 來控制每個 Pod 的內存分配。使用這種策略,您可以更好地控制內存分配,並提供一種機制來最小化 Windows 節點上的 out-of-memory (OOM)。

    在 Windows 節點上,最佳做法是為 OS 和進程保留至少 2GB 的內存。使用 --kubelet-reserve 和/或 --system-reserve 來減少 NodeAllocatable。

    根據 Amazon EKS Self-managed Windows nodes 文檔,使用 CloudFormation 模板啟動一個新的 Windows 節點組,並自定義 kubelet 配置。CloudFormation 有一個名為 BootstrapArguments 的元素,它與 KubeletExtraArgs 相同。使用以下標誌和值:

    --kube-reserved memory=0.5Gi,ephemeral-storage=1Gi --system-reserved memory=1.5Gi,ephemeral-storage=1Gi --eviction-hard memory.available<200Mi,nodefs.available<10%"

    如果使用 eksctl 作為部署工具,請查看以下文檔以自定義 kubelet 配置 https://eksctl.io/usage/customizing-the-kubelet/

    Windows 容器內存要求¶

    根據 Microsoft 文檔,NANO 的 Windows Server 基本映像至少需要 30MB,而 Server Core 需要 45MB。隨著您添加 Windows 組件(如 .NET Framework、Web Services 作為 IIS 和應用程序),這些數字會增加。

    了解您的 Windows 容器映像(即基本映像加其應用程序層)所需的最小內存量,並將其設置為 pod 規範中的容器資源/請求非常重要。您還應該設置一個限制,以避免 pod 在出現應用程序問題時消耗所有可用的節點內存。

    在下面的示例中,當 Kubernetes 調度程序嘗試將 pod 放置在節點上時,將使用 pod 的請求來確定哪個節點有足夠的可用資源進行調度。

     spec:
  - name: iis
    image: mcr.microsoft.com/windows/servercore/iis:windowsservercore-ltsc2019
    resources:
      limits:
        cpu: 1
        memory: 800Mi
      requests:
        cpu: .1
        memory: 128Mi

    結論¶

    使用這種方法可以最大程度地減少內存耗盡的風險,但無法完全防止其發生。使用 Amazon CloudWatch Metrics,您可以在發生內存耗盡時設置警報和補救措施。
