    監控¶

    Prometheus，一個畢業的 CNCF 專案，是目前最受歡迎的監控系統，原生整合到 Kubernetes。Prometheus 收集有關容器、Pod、節點和叢集的指標。此外，Prometheus 利用 AlertsManager 讓您可以編程警報，以便在叢集中出現問題時發出警告。Prometheus 將指標數據存儲為時間序列數據，由指標名稱和鍵/值對標識。Prometheus 包括一種名為 PromQL（Prometheus Query Language）的查詢語言。

    Prometheus 指標收集的高級體系結構如下所示:

    Prometheus 使用拉取機制，並使用導出器和從 Kubernetes API 使用 kube state metrics 來刮取目標的指標。這意味著應用程序和服務必須公開包含 Prometheus 格式化指標的 HTTP(S) 端點。然後，Prometheus 將根據其配置定期從這些 HTTP(S) 端點拉取指標。

    導出器可讓您以 Prometheus 格式化的指標使用第三方指標。Prometheus 導出器通常部署在每個節點上。有關完整的導出器列表，請參考 Prometheus 導出器。雖然 node exporter 適用於導出 Linux 節點的主機硬件和操作系統指標，但它不適用於 Windows 節點。

    在使用穩定的 Prometheus helm chart 的 混合節點 EKS 叢集中有 Windows 節點 時，您將看到 Windows 節點上的 Pod 失敗，因為此導出器不適用於 Windows。您需要單獨處理 Windows 工作節點池，並在 Windows 工作節點組上安裝 Windows 導出器。

    為了設置 Windows 節點的 Prometheus 監控，您需要在 Windows 服務器本身下載並安裝 WMI 導出器，然後在 Prometheus 配置文件的刮取配置中設置目標。 發行頁面提供了所有可用的 .msi 安裝程序，具有相應的功能集和錯誤修復。安裝程序將設置 windows_exporter 作為 Windows 服務，並在 Windows 防火牆中創建例外。如果不帶任何參數運行安裝程序，導出器將使用默認設置運行啟用的收集器、端口等。

    您可以查看本指南中的調度最佳實踐部分,該部分建議使用污點/容忍度或 RuntimeClass 來選擇性地將節點導出器部署到 Linux 節點,同時在 Windows 節點上安裝 Windows 導出器,作為您引導節點或使用您選擇的配置管理工具(例如 Chef、Ansible、SSM 等)的一部分。

    請注意,與 Linux 節點上安裝為 daemonset 的節點導出器不同,在 Windows 節點上 WMI 導出器是安裝在主機本身上。導出器將導出 CPU 使用率、內存和磁盤 I/O 使用率等指標,也可用於監控 IIS 站點和應用程序、網絡接口和服務。

    windows_exporter 將默認公開所有啟用的收集器的所有指標。這是收集指標的推薦方式,以避免錯誤。但是,對於高級用途,windows_exporter 可以傳遞一個可選的收集器列表來過濾指標。Prometheus 配置中的 collect[] 參數可以實現這一點。

    Windows 的默認安裝步驟包括在引導過程中下載並以參數的形式啟動導出器作為服務,例如您想要過濾的收集器。

    > Powershell Invoke-WebRequest https://github.com/prometheus-community/windows_exporter/releases/download/v0.13.0/windows_exporter-0.13.0-amd64.msi -OutFile <DOWNLOADPATH> 

> msiexec /i <DOWNLOADPATH> ENABLED_COLLECTORS="cpu,cs,logical_disk,net,os,system,container,memory"

    默認情況下,指標可以在端口 9182 上的 /metrics 端點上刮取。 此時,Prometheus 可以通過將以下 scrape_config 添加到 Prometheus 配置來使用指標

    scrape_configs:
    - job_name: "prometheus"
      static_configs: 
        - targets: ['localhost:9090']
    ...
    - job_name: "wmi_exporter"
      scrape_interval: 10s
      static_configs: 
        - targets: ['<windows-node1-ip>:9182', '<windows-node2-ip>:9182', ...]

    使用以下方式重新加載 Prometheus 配置

    > ps aux | grep prometheus
> kill HUP <PID>

    一種更好且推薦的方法是使用一個名為 ServiceMonitor 的自定義資源定義,它是 Prometheus 運算符的一部分,提供了 ServiceMonitor 對象的定義和一個控制器,該控制器將激活我們定義的 ServiceMonitor 並自動構建所需的 Prometheus 配置。

    ServiceMonitor 聲明性地指定應如何監視 Kubernetes 服務組,用於定義您希望在 Kubernetes 中監視指標的應用程序。在 ServiceMonitor 中,我們指定了運算符可以用來識別我們希望監視的 Kubernetes 服務(進而識別 Pod)的 Kubernetes 標籤。

    為了利用 ServiceMonitor,請創建一個指向特定 Windows 目標的 Endpoint 對象、一個無頭服務和一個用於 Windows 節點的 ServiceMontor。

    apiVersion: v1
kind: Endpoints
metadata:
  labels:
    k8s-app: wmiexporter
  name: wmiexporter
  namespace: kube-system
subsets:
- addresses:
  - ip: NODE-ONE-IP
    targetRef:
      kind: Node
      name: NODE-ONE-NAME
  - ip: NODE-TWO-IP
    targetRef:
      kind: Node
      name: NODE-TWO-NAME
  - ip: NODE-THREE-IP
    targetRef:
      kind: Node
      name: NODE-THREE-NAME
  ports:
  - name: http-metrics
    port: 9182
    protocol: TCP

---
apiVersion: v1
kind: Service ##Headless Service
metadata:
  labels:
    k8s-app: wmiexporter
  name: wmiexporter
  namespace: kube-system
spec:
  clusterIP: None
  ports:
  - name: http-metrics
    port: 9182
    protocol: TCP
    targetPort: 9182
  sessionAffinity: None
  type: ClusterIP

---
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor ##Custom ServiceMonitor Object
metadata:
  labels:
    k8s-app: wmiexporter
  name: wmiexporter
  namespace: monitoring
spec:
  endpoints:
  - interval: 30s
    port: http-metrics
  jobLabel: k8s-app
  namespaceSelector:
    matchNames:
    - kube-system
  selector:
    matchLabels:
      k8s-app: wmiexporter

    有關運算符和 ServiceMonitor 使用的更多詳細信息,請查看官方的 operator 文檔。請注意,Prometheus 確實支持使用許多服務發現選項進行動態目標發現。
